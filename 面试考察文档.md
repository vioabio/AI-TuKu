## 2.2 DDD 架构设计深度

### Q3：你的项目从三层架构重构到 DDD 四层架构，请画出 DDD 四层架构的依赖关系图，并解释为什么 Domain 层是核心。
> **考察点**：对 DDD 核心思想的理解、依赖倒置的实践能力
> **期望答案要点**：
> - 依赖方向：interfaces → application → domain ← infrastructure
> - Domain 层不依赖任何框架和基础设施，是纯 POJO + 接口
> - 通过 Repository 接口实现依赖倒置，Infrastructure 层实现接口
> - 核心价值：业务逻辑与技术实现解耦，领域模型可独立测试

### Q4：在你的项目中，ApplicationService 和 DomainService 的职责边界是如何划分的？什么逻辑放 Application 层，什么逻辑放 Domain 层？
> **考察点**：DDD 分层的实际落地能力、对"薄应用层"的理解
> **期望答案要点**：
> - ApplicationService：业务流程编排、事务管理、跨域调用协调
> - DomainService：单一领域内的纯业务逻辑，不涉及跨域调用
> - 判断标准：不涉及跨服务调用 → 下沉到 DomainService；需要编排多个 DomainService → 在 Application 层
> - 要能举出项目中的具体例子（如图片上传涉及 COS + 压缩 + 配额更新，编排在 Application 层）

### Q5：你提到了充血模型和贫血模型。在你的项目中，Entity 中包含了哪些业务方法？这样做的好处和风险是什么？
> **考察点**：对领域模型设计模式的理解深度
> **期望答案要点**：
> - 好处：业务逻辑内聚在 Entity 中，避免"贫血模型"的 Service 膨胀
> - 风险：Entity 可能变得臃肿；需要区分"实体自身行为"和"跨实体的领域服务"
> - 要能举例：如 Picture 实体的审核状态变更方法、Space 实体的配额校验方法

### Q6：DDD 在什么场景下反而会带来负担？你的项目是否适合 DDD？
> **考察点**：对 DDD 适用性的辩证思考，不盲从架构模式
> **期望答案要点**：
> - DDD 适用于业务复杂度高的项目（多个聚合、复杂的业务规则）
> - 简单 CRUD 项目用 DDD 是过度设计
> - 本项目 4 个有界上下文、RBAC 权限、协同编辑等，适合 DDD
> - 代价：文件数量增加、学习成本高、简单查询路径变长

# 三、后端技术深度（建议 30 分钟）

## 3.1 Java 基础与 Spring Boot

### Q9：你项目中使用了 Spring Boot 2.7，Spring 的 IOC 和 AOP 分别解决了什么问题？请结合你项目中的实际应用说明。
> **考察点**：Spring 核心概念的理解深度
> **期望答案要点**：
> - IOC：通过依赖注入管理 Bean 生命周期，项目中的 Repository 注入、@Lazy 解决循环依赖
> - AOP：横切关注点（鉴权、日志、事务），项目中的 @AuthCheck 注解 + AuthInterceptor
> - 要能讲清楚 Spring Bean 的默认作用域（Singleton）和初始化过程

### Q10：你项目中使用了 @Lazy 注解解决 Disruptor 的循环依赖问题。Spring 是如何处理循环依赖的？@Lazy 的原理是什么？三级缓存分别存什么？
> **考察点**：对 Spring 底层原理的理解
> **期望答案要点**：
> - Spring 三级缓存：singletonObjects（成品）、earlySingletonObjects（半成品）、singletonFactories（工厂）
> - 循环依赖解决流程：A 创建 → 发现依赖 B → 提前暴露 A 的 ObjectFactory → 创建 B → B 依赖 A → 从工厂获取 A 的早期引用 → B 完成 → A 完成
> - @Lazy 原理：注入的是代理对象，首次调用时才真正获取目标 Bean
> - 注意：构造器注入无法解决循环依赖

### Q11：你在项目中同时使用了 Session-Cookie 和 Sa-Token 两套鉴权体系。这两者有什么区别？什么场景下你会推荐使用 JWT？
> **考察点**：鉴权方案的选型能力
> **期望答案要点**：
> - Session-Cookie：服务端存储状态，适合传统 Web 应用
> - Sa-Token：更灵活的 Token 鉴权，支持多种存储后端
> - JWT：无状态，适合分布式/微服务/移动端场景
> - 各自优缺点：Session（无法水平扩展）、JWT（无法主动失效）、Sa-Token（折中方案）
> - 本项目选型理由：Session 用于用户登录态，Sa-Token Kit 模式独立管理空间权限

## 3.2 并发与高吞吐

### Q12：你选择 Disruptor 而不是线程池来处理 WebSocket 消息，Disruptor 的"快"体现在哪些方面？核心数据结构是什么？
> **考察点**：对高性能并发框架的理解深度
> **期望答案要点**：
> - 核心：RingBuffer（环形缓冲区），预分配内存，无 GC 压力
> - 无锁设计：CAS 操作 + Sequence 屏障，避免锁竞争
> - 伪共享（False Sharing）消除：通过缓存行填充（padding）确保 Sequence 独占缓存行
> - 单生产者 vs 多生产者模式的区别
> - 与线程池对比：线程池有上下文切换 + 队列锁竞争开销
> - 与 MQ 对比：MQ 有网络开销 + 序列化/反序列化开销

### Q13：你项目中 Disruptor 配置了 BlockingWaitStrategy，还有哪些等待策略？各自的适用场景是什么？
> **考察点**：对 Disruptor 调优的理解
> **期望答案要点**：
> - BlockingWaitStrategy：使用锁和条件变量，CPU 占用最低但延迟最高，适合对延迟要求不高的场景
> - SleepingWaitStrategy：自旋 + 让步，CPU 占用和延迟的折中
> - YieldingWaitStrategy：纯自旋，低延迟但 CPU 占用高，适合低延迟场景
> - BusySpinWaitStrategy：持续自旋，延迟最低但 CPU 100%，适合对延迟极度敏感的核心交易系统

### Q14（场景题）：假设你的 WebSocket 服务需要部署到 3 个实例上，如何保证消息能广播到所有实例上的客户端？
> **考察点**：分布式系统下的消息同步方案
> **期望答案要点**：
> - 方案一：Redis Pub/Sub 作为消息中间件，每个实例订阅相同 Channel
> - 方案二：RabbitMQ / Kafka 广播模式
> - 方案三：一致性 Hash 路由，将同一空间的用户路由到同一实例
> - 各方案 trade-off 分析

## 3.3 缓存架构

### Q15：你项目中使用 Caffeine + Redis 两级缓存，数据更新时如何保证缓存一致性？
> **考察点**：缓存一致性方案的理解
> **期望答案要点**：
> - 承认当前方案存在不一致窗口（Caffeine 和 Redis 过期时间不同）
> - Cache Aside 模式：更新数据库 → 删除缓存 → 下次查询回填
> - Redis Pub/Sub 失效广播：数据更新 → 发布 Redis 消息 → 所有实例监听 → 清除本地 Caffeine
> - 为什么先删缓存再更新数据库可能导致不一致（并发场景）
> - 延迟双删策略

### Q16：你用了 MD5 哈希作为缓存 Key，为什么不用原始查询条件 JSON？MD5 碰撞在你的场景下是问题吗？
> **考察点**：对缓存 Key 设计的工程考量
> **期望答案要点**：
> - MD5 哈希的好处：固定长度、减少 Redis 内存占用
> - MD5 碰撞概率：128 位输出空间，在业务场景下碰撞概率极低（可忽略）
> - 如果要完全避免碰撞，可以用 SHA-256 或拼接业务唯一标识
> - 要能算清楚：Redis Key 本身也是存储开销，长 JSON 作为 Key 不经济

### Q17：你如何防止缓存穿透和缓存雪崩？项目中具体做了哪些措施？
> **考察点**：缓存风险防护的实际落地
> **期望答案要点**：
> - 缓存穿透：空结果不缓存或缓存短时间；布隆过滤器前置拦截不存在的 Key
> - 缓存雪崩：随机过期时间（项目中 Redis 5-10 分钟随机）；多级缓存（Caffeine 先扛）
> - 缓存击穿：热点 Key 永不过期 + 互斥锁加载（项目中未实现但应了解）
> - 要区分这三个概念，不能混淆

## 3.4 文件存储与第三方集成

### Q18：你项目中腾讯云 COS + 万象 CI 的图片处理链路是怎样的？CI 降级容错具体是怎么做的？
> **考察点**：第三方服务集成的健壮性设计
> **期望答案要点**：
> - 正常流程：上传 → 万象 CI 压缩(webp) + 生成缩略图 → 返回 ProcessResults → 获取 Key → 拼接 URL
> - 降级流程：捕获 CosServiceException → 判断是否为 CI 服务异常 → 回退到基础 putObject → 手动生成缩略图 URL
> - 关键设计：降级是无损的（用户仍可上传成功），只是少了一些优化（无 webp 压缩、无自动缩略图）

### Q19：如果图片被恶意上传（违规内容、超大文件、脚本注入），你的系统有哪些防护措施？
> **考察点**：安全意识

> - 后端：文件大小二次校验（Manager 层）、文件类型魔数校验、内容安全审核（可接入 AI 审核）

## 4.2 WebSocket 与实时通信

### Q23：你前端是如何管理 WebSocket 连接的生命周期的？组件卸载时如果没有断开连接会有什么问题？
> **考察点**：前端资源管理意识
> **期望答案要点**：
> - 组件 mount → 建立 WebSocket 连接（onMounted）
> - 组件 unmount → 断开连接（onUnmounted）
> - 弹窗关闭 → 断开连接
> - 内存泄漏风险：未断开的 WebSocket 连接持有组件引用 → 组件无法被 GC → 事件回调仍触发但组件已销毁（报错/异常）
> - 心跳机制：检测连接是否存活

### Q24：你的协同编辑中"本地先执行 + 排除式广播"的设计很巧妙。如果两个用户同时编辑同一张图片，你的编辑锁机制如何保证数据一致性？
> **考察点**：并发冲突处理能力
> **期望答案要点**：
> - 编辑锁：ConcurrentHashMap 原子操作，先到先得
> - 获取锁成功 → 开始编辑 → 其他人收到"编辑中"提示
> - 获取锁失败 → 无法进入编辑模式
> - 释放锁：退出编辑模式 / WebSocket 断开（基于 Session 监听）
> - 锁超时：需要考虑如果用户刷新页面（锁未释放），需要超时自动释放机制

> **考察点**：前端性能优化意识

### Q26：你项目中的 Snowflake ID 精度丢失问题的本质是什么？除了后端序列化为 String，还有哪些替代方案？
> **考察点**：对 JavaScript 数值精度问题的理解深度
> **期望答案要点**：
> - 本质：JS Number 是 IEEE 754 双精度浮点数，安全整数范围 ±2^53-1（约 9 千万亿），Snowflake 19 位 Long 超出此范围
> - 方案对比：
>   1. 后端序列化为 String（本项目方案）：最简单可靠，前后端一致性强
>   2. 前端使用 BigInt：需要浏览器支持 + JSON.parse 需要 reviver
>   3. 换用短 ID 生成器（如 NanoID、Snowflake 精简版）：避免问题但功能受限
>   4. 使用 json-bigint 库：第三方依赖 + 全局影响

# 五、数据库设计能力（建议 15 分钟）

## 5.1 表设计与索引

### Q27：请画出你项目的核心 ER 图（user、picture、space、space_user 四张表的关系）。你使用了 Snowflake 作为主键，为什么不使用自增 ID？
> **考察点**：数据库设计思维
> **期望答案要点**：
> - ER 关系：user 1:N picture、user 1:N space_user、space 1:N picture、space 1:N space_user
> - Snowflake 优势：全局唯一（分布式友好）、趋势递增（对 B+Tree 索引友好）、高性能（本地生成，无 DB 交互）
> - 自增 ID 劣势：分库分表时冲突、暴露业务量（竞品可通过 ID 推算）、需要 DB 序列
> - `picture` 表使用 JSON 字段存储 tags：灵活但无法索引（MySQL 5.7+ 支持虚拟列索引）

### Q28：你使用了逻辑删除（is_delete 字段），这带来了什么便利和什么陷阱？
> **考察点**：对数据设计方案的辩证思考
> **期望答案要点**：
> - 便利：数据可恢复、保留历史数据用于分析、删除操作变为轻量 UPDATE
> - 陷阱：所有查询都要加 `is_delete = 0` 条件（MyBatis-Plus 自动处理）、唯一索引设计复杂（逻辑删除 + 唯一字段的联合索引）、数据膨胀（需要定期物理清理）

### Q29（手写 SQL）：假设 space 表中存储了每个空间的图片条数和使用量（total_size），当前空间的 picture 表有 10 万条记录，请写出统计每个用户上传图片总大小的 SQL，并设计合适的索引。
> **考察点**：SQL 能力 + 索引设计
> **期望答案**：
> ```sql
> SELECT user_id, SUM(pic_size) as total_size, COUNT(*) as pic_count
> FROM picture
> WHERE is_delete = 0
> GROUP BY user_id
> ORDER BY total_size DESC;
> ```
> **索引设计**：`(user_id, is_delete)` 联合索引或 `(is_delete, user_id, pic_size)` 覆盖索引
> **追问**：如果 picture 表有 1000 万条记录，如何优化这个查询？（物化视图/定时统计表）

## 6.2 Agent 开发思维（Agent 岗位重点）

### Q32（Agent 方向核心题）：你如何理解"Agent"和"传统 AI API 调用"的区别？你项目中的 AI 扩图功能，如果升级为一个真正的 Agent，需要增加哪些能力？
> **考察点**：对 Agent 核心概念的认知水平
> **期望答案要点**：
> - 传统 API 调用：单次输入 → 单次输出，无自主决策
> - Agent：感知 → 推理 → 决策 → 执行 → 反馈循环（ReAct 模式）
> - 升级方向：
>   1. 自然语言理解用户意图（"帮我把这张图向右扩展一些，保持风格一致"）
>   2. 自动判断需要哪些工具（扩图？裁剪？滤镜？）
>   3. 多步推理执行（先分析图片 → 选择最佳扩展参数 → 执行 → 评估结果 → 重试优化）
>   4. 记忆管理（记住用户偏好：喜欢扩多大比例、什么风格）
>   5. Function Calling：将 40+ 个 API 封装为 Agent 可调用的 Tool

### Q33（Agent 方向核心题）：如果你要把项目中的 40+ 个 RESTful API 封装为 Agent 的 Function/Tool 定义，请选择 5 个最具代表性的 API，分别描述它们的 Tool 定义（名称、描述、参数 Schema、权限要求）。
> **考察点**：Function Calling 设计能力
> **评分标准**：
> - 优秀：Tool 定义有清晰的 name/description/parameters，描述能让 LLM 正确判断何时调用，参数类型明确
> - 合格：能写出基本的 Tool 定义但描述不够精确
> - 不合格：不知道 Function Calling 是什么

### Q34（Agent 方向核心题）：Agent 在执行过程中可能产生"幻觉"（Hallucination），在图片管理场景下，Agent 的幻觉可能导致什么风险？你如何设计安全边界？
> **考察点**：对 Agent 安全性的思考
> **期望答案要点**：
> - 风险场景：
>   - Agent 调用删除图片 Tool，但传入错误的 pictureId → 误删
>   - Agent 认为用户想要公开私有图片 → 越权操作
>   - Agent 虚构不存在的图片信息 → 误导用户
> - 安全边界设计：
>   1. 关键操作需要二次确认（删除、权限变更）
>   2. Tool 执行层做权限校验（复用现有 RBAC + Sa-Token）
>   3. Tool 参数严格校验（类型、范围、存在性）
>   4. 操作审计日志（记录 Agent 的每次 Tool 调用）
>   5. 配置"只读"和"读写"两类 Agent 模式

### Q35（Agent 方向架构题）：如果产品要求"用户可以用自然语言对话来管理整个图库"（如"帮我把上周上传的猫的图片都移到团队空间里"），请设计这个系统的技术架构。
> **考察点**：Agent 系统架构设计能力
> **期望答案要点**：
> - 整体架构：前端对话界面 → 后端 Agent 编排引擎 → LLM（推理+规划） → Tool 执行层 → 现有业务 API
> - Agent 引擎核心组件：
>   1. 对话管理：维护对话上下文和历史消息
>   2. Tool Registry：注册所有可用的 Function/Tool 定义
>   3. Planner：LLM 将自然语言拆解为 Tool 调用序列
>   4. Executor：按序执行 Tool 调用，收集结果
>   5. Memory：短期（对话上下文）+ 长期（用户偏好、向量检索相似操作历史）
> - 关键技术决策：
>   - 选择 LangChain / LangGraph 还是自研编排引擎
>   - LLM 选型（成本 vs 能力）
>   - Tool 执行的超时和重试机制

# 八、代码质量与工程化（建议 15 分钟）

## 8.1 设计模式与代码组织

### Q39：你的项目中使用了模板方法模式重构图片上传。请画出模板方法模式的类图，并解释"骨架方法"和"钩子方法"的区别。
> **考察点**：设计模式的真正理解（而非死记硬背）
> **期望答案要点**：
> - 骨架方法（Template Method）：定义算法步骤的执行顺序（final 方法，不允许子类覆盖）
> - 钩子方法（Hook Method）：允许子类选择性覆盖的步骤（Protected 抽象方法或默认实现的普通方法）
> - 项目中的应用：PictureUploadTemplate 定义 6 步骨架，FilePictureUpload 和 UrlPictureUpload 各自实现钩子
> - 收益：新增上传方式只需新增子类，无需修改父类和已有子类（开闭原则）

### Q40：你使用了 OpenAPI Generator 生成前端代码。这些生成的代码你会直接修改吗？如果后端接口变更了，你如何同步？
> **考察点**：工程化协作思维
> **期望答案要点**：
> - 不直接修改生成的代码（会在下次生成时被覆盖）
> - 生成代码在独立目录（如 `api/`），业务代码通过 import 使用
> - 后端接口变更 → 更新 Swagger 文档 → 重新运行生成命令 → 前端类型报错 → 修改业务代码适配
> - 版本管理：OpenAPI 文档纳入 Git，生成代码可以纳入 Git（或 CI 自动生成）

### Q41：你的项目没有写单元测试。如果现在让你给最核心的模块写测试，你会选择哪个模块？为什么？你会怎么写？
> **考察点**：测试意识和方法论
> **期望答案要点**：
> - 优先测试：PictureDomainService（纯业务逻辑，无外部依赖）或 RBAC 权限判断逻辑
> - 测试策略：单元测试（Domain 层）→ 集成测试（Repository 层）→ E2E 测试（关键流程）
> - 要能描述 Mock 的使用场景（Mock Repository 来测试 DomainService）
