## 阶段 1：项目总览与架构设计

### 学习目标
理解项目的业务全景、技术选型理由、架构设计思想。

### 详细知识点

#### 1.1 业务模型设计
- **三种空间模式**：公共图库（所有人可见）→ 私有空间（个人）→ 团队空间（多人协作）
- **用户角色体系**：普通用户（user）/ 管理员（admin）；空间角色：浏览者（viewer）/ 编辑者（editor）/ 管理员（admin）
- **核心业务流程**：注册 → 登录 → 创建空间 → 上传图片 → 浏览/搜索/编辑 → 团队协作

#### 1.2 技术选型决策
- **为什么选择 Spring Boot 而非 Spring MVC？** 自动配置、起步依赖、内嵌服务器，减少配置工作量
- **为什么选择 MyBatis-Plus 而非 JPA/Hibernate？** 更灵活的 SQL 控制、强大的条件构造器、分页插件、代码生成器
- **为什么选择 Vue 3 而非 React？** 更平缓的学习曲线、Composition API 更好的逻辑复用、更小的包体积
- **为什么选择 Ant Design Vue？** 企业级 UI 组件库、完善的中文文档、丰富的表单/表格/布局组件

#### 1.3 物理架构设计
- **开发环境**：前端 localhost:5173（Vite 代理到后端 8123）、后端 localhost:8123
- **生产环境**：前端 Cloudflare Pages（全球 CDN）、后端 Cloudflare Tunnel（内网穿透）

#### 1.4 三层架构 vs DDD 架构
- **传统三层**：Controller → Service → Mapper（业务复杂时 Service 膨胀）
- **DDD 四层**：interfaces → application → domain ← infrastructure（依赖倒置、领域内聚）

### 面试题 1-2：如果重新从零开始，你会在哪些地方做出不同的技术决策？

**参考答案：**

**1. Spring Boot 3.x + Java 21 虚拟线程**
当前使用 Spring Boot 2.7.6 + Java 17。Spring Boot 2.7 已于 2023 年 11 月停止维护，3.x 版本支持：
- 虚拟线程（Virtual Threads）：WebSocket 和 HTTP 请求处理几乎零代码改动即可获得大幅并发提升
- GraalVM Native Image：编译为原生二进制，启动时间从秒级降至毫秒级
- @AiService 注解：LangChain4j 的原生注解模式（当前 Spring Boot 2.7 只能用 Builder 模式兼容）

**2. 密码加密方案升级**
当前仍使用 MD5("vio" + password) 加密（固定盐值），应迁移到 BCrypt（自适应 cost factor + 随机盐值），配合 30 天过渡期（老用户 MD5 验证通过后自动升级为 BCrypt）。

**3. 数据库版本管理**
当前依赖 MyBatis-Plus 自动 DDL + 手动 SQL 变更，应引入 Flyway 实现 Schema 版本化、可追溯、CI 自动校验。

**4. Docker + CI/CD 已落地**（与上一版回答的最大变化）
已实现 Docker Compose 一键编排（MySQL + Redis + BE + FE）+ GitHub Actions CI/CD 双流水线（test + deploy）。如果重来，应该从项目第一天就使用 Docker，消除"在我机器上能跑"问题。

### 面试题 1-3：微服务架构和单体架构的取舍？你的项目适合拆成微服务吗？

**参考答案：**

**当前项目适合保持单体架构**，理由：
1. 业务规模：4 张表、5 个领域、日活预估低，单体架构完全够用
2. 团队规模：个人开发，微服务引入分布式复杂度得不偿失
3. 性能：单体架构无网络调用开销，DDD 分层已提供足够的内聚性

**如果要拆分微服务，演进路径**：
- 第一步：按业务域拆分（用户服务 / 图片服务 / 空间服务 / 分析服务）
- 第二步：引入 API Gateway（Spring Cloud Gateway）统一入口
- 第三步：引入 Nacos 做服务注册与配置中心
- 第四步：引入 Sentinel 做熔断限流
- 关键挑战：分布式事务（Seata）、跨服务鉴权（JWT/OAuth2）、数据一致性

**核心观点**：不要为了微服务而微服务，DDD 分层已经为未来拆分奠定了基础（领域边界清晰）。

### 面试题 2-1：Spring Boot 自动配置原理是什么？

**参考答案：**

Spring Boot 自动配置通过 `@EnableAutoConfiguration` 注解触发，核心流程：

1. **启动入口**：`@SpringBootApplication` 包含 `@EnableAutoConfiguration`
2. **加载配置类**：通过 `SpringFactoriesLoader` 读取 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`（2.7 版本用 `spring.factories`）
3. **条件装配**：每个自动配置类上有条件注解（如 `@ConditionalOnClass`、`@ConditionalOnMissingBean`、`@ConditionalOnProperty`），只在条件满足时加载
4. **示例**：`DataSourceAutoConfiguration` 在 classpath 中存在 `DataSource` 类且未手动定义 DataSource Bean 时自动配置数据源

**实际应用**：
- MyBatis-Plus 的 `MybatisPlusAutoConfiguration`：检测到 `SqlSessionFactory` 后自动配置
- 自定义配置：通过 `@Configuration` + `@ConditionalOnProperty` 实现条件化配置

**如何排除自动配置**：`@SpringBootApplication(exclude = {DataSourceAutoConfiguration.class})`

### 面试题 2-2：Vite 和 Webpack 有什么区别？为什么 Vue 3 项目推荐 Vite？

**参考答案：**

| 维度 | Vite | Webpack |
|------|------|---------|
| 开发服务器启动 | 冷启动 < 1s（ESM 按需编译） | 需打包整个应用，几十秒到分钟级 |
| 热更新（HMR） | 极快（只更新修改的模块） | 随项目增大变慢 |
| 构建 | 基于 Rollup 生产构建 | 自身完成打包 |
| 原理 | 利用浏览器原生 ES Module，请求时按需编译 | 先打包所有模块为 bundle |
| 插件生态 | 兼容 Rollup 插件，生态快速增长 | 插件生态成熟丰富 |

**为什么快**：

### 面试题 2-3：MyBatis-Plus 的分页插件是如何工作的？

**参考答案：**

MyBatis-Plus 分页插件基于 MyBatis 的拦截器（Interceptor）机制实现：

1. **注册分页插件**：
```java
@Bean
public MybatisPlusInterceptor mybatisPlusInterceptor() {
    MybatisPlusInterceptor interceptor = new MybatisPlusInterceptor();
    interceptor.addInnerInterceptor(new PaginationInnerInterceptor(DbType.MYSQL));
    return interceptor;
}
```

2. **原理**：
   - MyBatis 在执行 SQL 前会调用已注册的拦截器
   - `PaginationInnerInterceptor` 拦截 `Executor.query()` 方法
   - 检测到 `IPage` 参数时，自动拼接 `LIMIT offset, size`（MySQL 方言）
   - 同时执行 `COUNT` 查询获取总记录数

3. **方言适配**：不同的 `DbType` 生成不同方言的分页 SQL（MySQL → LIMIT、Oracle → ROWNUM、PostgreSQL → OFFSET FETCH）

4. **本项目遇到的坑**：未配置分页插件导致 `IPage` 参数不生效，返回全部数据而非分页数据。根本原因是缺少分页拦截器拼接 LIMIT 子句。

### 面试题 3-1：Session-Cookie 和 JWT 的区别？什么场景下选哪个？

**参考答案：**

| 维度 | Session-Cookie | JWT |
|------|---------------|-----|
| 状态存储 | 服务端存储（有状态） | 客户端存储（无状态） |
| 扩展性 | 难以水平扩展（需 Session 共享） | 天然支持分布式 |
| 安全性 | Cookie 可设置 HttpOnly/Secure | Token 存储在客户端，存在 XSS 风险 |
| 主动失效 | 服务端删除 Session 即可 | 无法主动失效（需配合黑名单） |
| 携带方式 | Cookie 自动携带 | Header Authorization 手动携带 |
| 适合场景 | 传统 Web 应用、单体架构 | 分布式/微服务、移动端、跨域 API |

**本项目选型**：使用 Session-Cookie 是因为项目为传统 Web 前后端分离架构，Session 简单可靠，且单体部署无水平扩展需求。同时使用 Sa-Token 的 Kit 模式，为空间权限体系增加独立的 Token 维度的鉴权。

**JWT 的问题**：
1. Token 一旦签发，在过期前无法主动撤销（用户修改密码后旧 Token 仍有效）
2. Payload 是 Base64 编码而非加密，不能存放敏感信息
3. Token 体积较大，每次请求都携带

### 面试题 3-2：Spring AOP 的原理是什么？JDK 动态代理和 CGLIB 有什么区别？

**参考答案：**

**AOP 核心概念**：
- Aspect（切面）：横切关注点的模块化（如鉴权、日志、事务）
- Join Point（连接点）：方法执行点
- Pointcut（切入点）：匹配连接点的表达式
- Advice（通知）：在切入点执行的动作（@Before / @After / @Around）

**Spring AOP 实现原理**：
1. Spring AOP 基于动态代理实现（非编译期织入，区别于 AspectJ）
2. 当 Bean 有接口时 → JDK 动态代理（`Proxy.newProxyInstance`）
3. 当 Bean 无接口时 → CGLIB 代理（字节码生成子类）
4. Spring Boot 2.x 默认使用 CGLIB（`spring.aop.proxy-target-class=true`）

**JDK 动态代理 vs CGLIB**：

| 维度 | JDK 动态代理 | CGLIB 代理 |
|------|------------|-----------|
| 原理 | 反射，基于接口生成代理 | ASM 字节码，基于继承生成子类 |
| 限制 | 目标类必须实现接口 | 不能代理 final 类/方法 |
| 性能 | 创建快，调用略慢（反射） | 创建慢（字节码生成），调用快 |
| 本项目 | AuthInterceptor 使用 @Around 切面鉴权 |

**本项目中的应用**：`AuthInterceptor` 使用 `@Around("@annotation(com.vio.aitukuviobe.infrastructure.annotation.AuthCheck)")` 拦截带有 `@AuthCheck` 注解的方法，从 Session 获取用户信息并校验角色。

### 面试题 3-3：BCrypt 加密为什么安全？彩虹表攻击是什么？

**参考答案：**

**BCrypt 安全特性**：
1. **内置盐值（Salt）**：每次加密自动生成随机盐值，同一密码两次加密结果不同
2. **可配置的计算成本**：`BCryptPasswordEncoder(strength=10)`，strength 越高哈希计算越慢（指数关系，2^strength 次迭代）
3. **抗暴力破解**：单次验证耗时 ~100ms，攻击者无法高速尝试

**彩虹表攻击**：
- 彩虹表是预先计算好的"明文 → 哈希"映射表
- 如果密码仅做一次简单哈希（如 MD5、SHA-256），攻击者可通过查表反查明文
- **BCrypt 为什么能防**：每个密码的盐值不同，攻击者无法预先为所有盐值生成彩虹表

**本项目实战**：
```java
// 注册时加密
String encryptedPassword = passwordEncoder.encode(userPassword);
// 登录时验证
passwordEncoder.matches(inputPassword, storedEncryptedPassword);
```

### 面试题 4-4：JavaScript 中为什么 `0.1 + 0.2 !== 0.3`？Snowflake ID 精度丢失是同一个原因吗？如何全链路修复？

**参考答案：**

**0.1 + 0.2 的问题**：
- IEEE 754 浮点数用二进制存储小数，0.1 和 0.2 在二进制中是无限循环小数
- 存储时被截断，产生舍入误差，`0.1 + 0.2` 实际是 `0.30000000000000004`

**Snowflake ID 精度丢失**：
- **根因**：JavaScript Number 是 IEEE 754 双精度浮点数（64 位 = 1 符号位 + 11 指数位 + 52 尾数位），安全整数范围 `Number.MAX_SAFE_INTEGER` = `2^53 - 1` ≈ 9 千万亿（约 16 位）
- Snowflake 算法生成的 19 位 Long 型 ID（如 `1801234567890123456`）超出 JS 安全整数范围
- JSON 解析时超出部分被舍入，末尾 2-3 位变成 0 → `1801234567890123400`

**两者相同点**：根因都是 IEEE 754 的精度限制

**两者不同点**：
- `0.1 + 0.2` 是小数表示的精度问题（计算中的误差）
- Snowflake ID 是**大整数无法精确表示**的问题（超出 52 位尾数）

**全链路修复方案**：
1. **后端 → 前端**：`JsonConfig` 中所有 Long 类型使用 `ToStringSerializer` 序列化为 String
2. **前端 → 后端**：所有 ID 参数类型 `number` → `string | number`，保持 String 传递
3. **Spring MVC 自动解析**：接收 String 参数 → 自动转换为 Long（`StringToLongConverter`）
4. **关键路径修复**：排查并修正 `Number(props.id)` / `Number(route.params.id)` 调用

**Snowflake 算法结构**：
- 1 位（保留）+ 41 位（毫秒时间戳）+ 10 位（机器 ID）+ 12 位（序列号）= 64 位
- 优势：全局唯一、趋势递增（B+Tree 友好）、本地生成（无网络调用）
- 本项目配置：MyBatis-Plus `IdType.ASSIGN_ID` 自动使用 Snowflake

**其他解决方案对比**：
- 前端使用 BigInt → 需要浏览器支持 + JSON.parse reviver
- 使用 json-bigint 库 → 第三方依赖 + 全局影响
- 换用短 ID 生成器（NanoID）→ 避免问题但功能受限
- **String 方案**（本项目采用）：最简单可靠，前后端一致性强

## 阶段 6：图片优化模块

### 学习目标

### 详细知识点

#### 6.1 Caffeine 本地缓存
- **核心特性**：W-TinyLFU 淘汰算法（Window TinyLFU）、近乎最优命中率
- **配置项**：
  - `initialCapacity(1024)`：初始容量
  - `maximumSize(10000)`：最大条目数
  - `expireAfterWrite(5, TimeUnit.MINUTES)`：写入后过期
- **使用方式**：`cache.get(key, k -> loadFromDB())`（Read-Through）

#### 6.2 Redis 分布式缓存
- **Spring Data Redis**：`RedisTemplate<String, Object>`
- **序列化**：JSON 序列化（Jackson）存储对象
- **过期时间随机化**：`TTL ± 随机偏移` 防雪崩

#### 6.3 缓存 Key 设计
- **方案**：查询条件 JSON → MD5 → `yupicture:listPictureVOByPage:{hash}`
- **优点**：固定长度、减少 Redis 内存占用

#### 6.4 缓存三大问题
- **缓存穿透**：查询不存在的数据 → 布隆过滤器 或 空值缓存短时间
- **缓存击穿**：热点 Key 过期 → 互斥锁加载 或 永不过期
- **缓存雪崩**：大量 Key 同时过期 → 随机过期时间 + 多级缓存

#### 6.5 Read-Through 缓存模式
- **流程**：请求命中 Caffeine → 未命中查 Redis → 未命中查 DB → 回填 Caffeine + Redis

### 面试题 6-1：请详细解释缓存穿透、缓存击穿、缓存雪崩的区别和各自的解决方案。

**参考答案：**

**缓存穿透**（查询不存在的数据）：
- 场景：恶意查询不存在的 ID（如 id=-1），每次都穿透缓存查 DB
- 解决：
  1. **布隆过滤器**：前置过滤不存在的 Key（有误判率，但不会漏判）
  2. **空值缓存**：将 null 结果也缓存（短过期时间，如 60 秒）
  3. **本项目做法**：空结果不写入缓存，依赖前端参数校验拦截非法 ID

**缓存击穿**（热点 Key 过期）：
- 场景：秒杀商品的缓存刚好过期，大量请求直接打到 DB
- 解决：
  1. **互斥锁**：只有获得锁的线程去加载 DB，其他等待
  2. **逻辑过期**：热点 Key 设置永不过期，后台异步刷新
  3. **本项目**：Caffeine 作为本地缓存先扛一波，Redis 过期后 Caffeine 可能还有数据

**缓存雪崩**（大量 Key 同时过期）：
- 场景：一批缓存设置了相同的过期时间，同时过期导致 DB 压力激增
- 解决：
  1. **随机过期时间**：TTL 增加随机偏移（本项目：5-10 分钟随机）
  2. **多级缓存**：Caffeine（本地）→ Redis（分布式）→ DB
  3. **限流降级**：保护 DB，部分请求快速失败

**关键区分口诀**：
- 穿透：查不存在的数据 → 布隆过滤器
- 击穿：一个热点挂了 → 互斥锁
- 雪崩：大片缓存同时挂 → 随机过期

### 面试题 6-2：Caffeine 的 W-TinyLFU 算法相比 LRU 有什么优势？

**参考答案：**

**LRU（Least Recently Used）的缺陷**：
- 只考虑"最近访问"，不考虑"访问频率"
- 容易被"扫描性"访问污染：如果大量一次性数据被访问，会把高频热点数据挤出去
- 例如：批量导出报表操作扫描大量数据 → 热点缓存被冲掉

**W-TinyLFU 的设计**：
1. **Window Cache**：新进入的数据先放入 Window 区（LRU），防止新数据被低频数据立即淘汰
2. **Main Cache**（SLRU）：分 Probation（考察区）和 Protected（保护区）
3. **TinyLFU**：使用 Count-Min Sketch 维护访问频率计数器，判断是否淘汰
4. **核心思想**：淘汰时综合考虑"最近性"和"频率"，保护高频数据

**优势**：在大多数场景下命中率接近最优（Optimal），且内存开销可控。

**本项目使用**：图片列表查询的缓存，高频率首页查询不会被一次性扫描操作冲掉。

### 面试题 6-3：数据更新时如何保证缓存一致性？

**参考答案：**

**Cache Aside 模式（旁路缓存）**：
```
写操作：更新 DB → 删除缓存（或更新缓存）
读操作：查缓存 → 命中返回 / 未命中查 DB → 回填缓存
```

**为什么是"删缓存"而非"更新缓存"**：
- 如果更新缓存，并发写可能造成缓存和 DB 数据不一致
- 删除是幂等操作，更安全

**为什么先更新 DB 再删缓存**：
- 如果先删缓存 → 更新 DB 前另一个线程读 → 读到旧数据 → 写入缓存 → 缓存是脏数据
- 如果先更新 DB → 删缓存 → 其他线程读到新数据或缓存为空 → 回填时读新数据

**本项目的不足与改进**：
- 当前只在数据更新后删 Redis 缓存，但 Caffeine 本地缓存未同步失效
- 改进方案：Redis Pub/Sub 广播缓存失效消息 → 所有实例监听 → 清除本地 Caffeine
- 进一步：考虑延迟双删（删缓存 → 更新 DB → 等一会 → 再删缓存）

**最终一致性认知**：对于图片列表这种场景，允许短时间的不一致（Caffeine 5 分钟过期），不需要强一致性。

5. Controller 层通过 `@SaSpaceCheckPermission` 注解声明式鉴权

**相比硬编码的优势**：

### 面试题 7-3：你是如何解决 Servlet Body 只能读取一次的问题的？

**参考答案：**

**问题背景**：
- Servlet 的 `HttpServletRequest.getInputStream()` 返回的流只能消费一次
- Sa-Token 在执行权限校验时需要读取请求体（获取 pictureId 等参数）
- Controller 方法再次读取请求体时，流已为空 → 参数绑定失败

**解决方案：自定义 RequestWrapper**

```java
public class RequestWrapper extends HttpServletRequestWrapper {
    private final byte[] body;  // 缓存请求体
    
    public RequestWrapper(HttpServletRequest request) {
        super(request);
        // 读取原始流并缓存
        this.body = StreamUtils.copyToByteArray(request.getInputStream());
    }
    
    @Override
    public ServletInputStream getInputStream() {
        // 每次调用返回新的流（基于缓存的 byte[]）
        return new ByteArrayServletInputStream(body);
    }
}
```

**配合过滤器使用**：
```java
@WebFilter(urlPatterns = "/*")
public class HttpRequestWrapperFilter implements Filter {
    @Override
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain) {
        if (request instanceof HttpServletRequest) {
            // 包装原始请求，后续所有读取都经过 RequestWrapper
            chain.doFilter(new RequestWrapper((HttpServletRequest) request), response);
        }
    }
}
```

**核心思想**：在过滤器中一次性读完 Body 并缓存在 byte 数组中，后续每次 `getInputStream()` 都基于缓存的 byte[] 重新创建流。

### 面试题 8-2：请画出模板方法模式的类图，并解释"骨架方法"和"钩子方法"的区别。

**参考答案：**

```
abstract PictureUploadTemplate {
    + uploadPicture(inputSource)  ← 骨架方法（final，定义步骤，不可覆盖）
    # validateFile(inputSource)   ← 钩子方法（abstract / protected，子类覆盖）
    # createTempFile(inputSource)
    # processPicture(file)
    # uploadToCOS(file)
    # buildResult(uploadResult)
    # cleanup(file)
}

FilePictureUpload extends PictureUploadTemplate {
    # validateFile(MultipartFile file)     → 校验文件大小/类型
    # createTempFile(MultipartFile file)   → 从 MultipartFile 创建临时文件
    ...
}

UrlPictureUpload extends PictureUploadTemplate {
    # validateFile(String url)             → 校验 URL 格式
    # createTempFile(String url)           → 从 URL 下载创建临时文件
    ...
}
```

**骨架方法**（Template Method）：
- `uploadPicture()` 定义了上传的**固定算法步骤**：校验 → 创建临时文件 → 处理 → 上传 → 构建结果 → 清理
- 用 `final` 修饰，**不允许子类覆盖**，保证了上传流程的一致性

**钩子方法**（Hook Method）：
- `validateFile()`、`createTempFile()` 等是可覆盖的步骤
- 用 `protected abstract` 或 `protected`（提供默认实现）
- 子类根据自己的需求实现具体逻辑

**收益**：
1. **复用**：6 步流程代码只需写一次（父类 ~150 行），新增上传方式只需新增子类
2. **扩展**：符合开闭原则——对扩展开放（新增子类），对修改关闭（无需改父类）
3. **一致性**：所有上传方式遵循相同的流程规范，降低维护成本

**本项目实战效果**：上传代码从 300 行降至父类 150 行 + 各子类 40 行，可维护性显著提升。

### 面试题 9-1：异步任务模式 vs 同步模式在 AI API 调用中各有什么优劣？

**参考答案：**

**同步模式**：
- 流程：客户端请求 → 服务端调用 AI API → 等待处理完成（10-60s） → 返回结果
- 优点：逻辑简单、一次请求完成
- 缺点：
  - HTTP 连接长时间占用（可能超时）
  - 用户无进度反馈，体验差
  - 服务端线程阻塞等待

**异步模式**：
- 流程：客户端请求 → 服务端提交任务，返回 taskId → 服务端立即响应 → 客户端轮询查询结果
- 优点：
  - 服务端线程立即释放，不阻塞
  - 用户有进度反馈（轮询间隔显示状态）
  - 可支持更大并发
- 缺点：
  - 实现复杂（需要状态管理）
  - 轮询有延迟
  - 需要持久化 taskId

**本项目的选型**：AI 扩图处理需要 10-12 秒，同步模式会导致 HTTP 超时，因此选择异步模式。

**其他方案对比**：
- **WebSocket 推送**：任务完成时服务端主动推送（延迟最低，但增加连接管理复杂度）
- **SSE（Server-Sent Events）**：基于 HTTP 的单向推送（比 WebSocket 简单，但不能双向通信）
- **Webhook**：任务完成时回调指定 URL（解耦但需要公网可达的回调地址）

- **解决方案**：`@Lazy` 注解 → 注入代理对象 → 首次调用时才真正获取目标 Bean
- **Spring 三级缓存**：
  - `singletonObjects`（一级）：成品 Bean
  - `earlySingletonObjects`（二级）：半成品 Bean（提前暴露引用）
  - `singletonFactories`（三级）：Bean 工厂（ObjectFactory）

### 面试题 11-1：Spring 是如何解决循环依赖的？@Lazy 的原理是什么？

**参考答案：**

**Spring 解决循环依赖的核心流程**（以 A ↔ B 为例）：

```
步骤 1: 创建 A
  → 实例化 A（调用构造器）
  → 将 A 的 ObjectFactory 放入三级缓存（singletonFactories）
  → 属性填充 → 发现需要 B

步骤 2: 创建 B
  → 实例化 B（调用构造器）
  → 将 B 的 ObjectFactory 放入三级缓存
  → 属性填充 → 发现需要 A
  → 从三级缓存获取 A 的 ObjectFactory → 创建 A 的早期引用
  → 将 A 的早期引用放入二级缓存（earlySingletonObjects）
  → B 完成属性填充 → 放入一级缓存（singletonObjects）

步骤 3: 回到 A
  → A 获得 B 的引用（从一级缓存）
  → A 完成属性填充 → 放入一级缓存
```

**为什么构造器注入无法解决循环依赖**：
- 构造器注入在"实例化"阶段就需要依赖 → 此时依赖对象还未创建 → 死循环

**@Lazy 原理**：
- 当字段标注 `@Lazy` 时，Spring 不直接注入目标 Bean，而是注入一个代理对象（CGLIB 代理）
- 只有首次调用该代理对象的方法时，代理才会从 Spring 容器中获取真正的 Bean
- 此时目标 Bean 已经完成初始化，所以不会出现循环依赖

**本项目中的应用**：
```java
@Service
public class SpaceServiceImpl {
    @Lazy  // 打破 SpaceServiceImpl → SpaceUserServiceImpl → SpaceServiceImpl 循环
    @Autowired
    private SpaceUserService spaceUserService;
}
```

### 学习目标
实现 WebSocket 双向通信、Disruptor 异步消息处理、编辑锁互斥、排除式广播。

### 详细知识点

#### 12.1 WebSocket 协议
- **与 HTTP 区别**：全双工、长连接、帧协议（非请求-响应）
- **握手**：HTTP Upgrade 请求 → `101 Switching Protocols` 响应
- **Spring WebSocket**：`WebSocketHandler` + `WebSocketConfigurer`
- **拦截器**：`HandshakeInterceptor` 在握手前校验权限

#### 12.2 Disruptor 环形缓冲区
- **核心数据结构**：RingBuffer（预分配数组，无 GC 压力）
- **核心组件**：
  - `RingBuffer`：存储事件
  - `Sequence`：序列号（跟踪生产者和消费者的进度）
  - `SequenceBarrier`：协调生产者和消费者
  - `EventProcessor`：消费事件
- **无锁设计**：CAS 操作替代锁
- **伪共享消除**：`@Contended` / 缓存行填充（Padding），确保 Sequence 独占缓存行

#### 12.3 Disruptor vs 线程池 vs MQ
| 维度 | Disruptor | 线程池 | MQ（RabbitMQ/Kafka） |
|------|----------|-------|---------------------|
| 延迟 | 微秒级 | 毫秒级 | 毫秒~秒级 |
| 吞吐量 | 极高（无锁） | 受锁竞争限制 | 高（但有网络开销） |
| 持久化 | 无（内存） | 无 | 有 |
| 适用场景 | 同进程超高吞吐 | 通用异步 | 分布式/跨服务 |

#### 12.4 协同编辑设计
- **编辑锁**：`ConcurrentHashMap<pictureId, editingUserId>`，`putIfAbsent` 原子操作
- **排除式广播**：操作者本身不接收广播（本地已即时执行），避免重复渲染
- **前端即时反馈**：编辑操作先更新本地 UI → 再通过 WebSocket 发送 → 其他客户端同步

#### 12.5 等待策略对比
| 策略 | CPU占用 | 延迟 | 适用场景 |
|------|--------|------|---------|
| BlockingWaitStrategy | 最低 | 最高 | 对延迟不敏感 |
| SleepingWaitStrategy | 中等 | 中等 | 折中选择 |
| YieldingWaitStrategy | 较高 | 低 | 低延迟场景 |
| BusySpinWaitStrategy | 100% | 最低 | 极致延迟 |

### 面试题 12-2：Disruptor 为什么比线程池快？核心设计思想是什么？

**参考答案：**

**Disruptor 快的核心原因**：

**1. 无锁设计（CAS）**
- 生产者和消费者通过 `Sequence`（序列号）+ CAS 原子操作协调
- 避免了传统阻塞队列的 `synchronized` / `ReentrantLock` 锁竞争

**2. RingBuffer 预分配内存**
- 数组在创建时一次性分配所有槽位
- 事件对象复用（创建后不清除，覆盖写入）
- **零 GC 压力**：运行期间不会产生新对象

**3. 伪共享（False Sharing）消除**
- CPU 缓存以缓存行（Cache Line，通常 64 字节）为单位加载
- 如果两个变量在同一缓存行，修改一个会导致另一个变量的缓存失效
- Disruptor 的 `Sequence` 通过前后填充 7 个 long（56 字节 + 8 = 64 字节）独占缓存行

**4. 单生产者优化**
- 多生产者需要 CAS 竞争写位置
- 单生产者模式（`SingleProducerSequencer`）无需 CAS，直接递增 Sequence

**与线程池对比**：
- 线程池 + 阻塞队列 → `ArrayBlockingQueue.put()` 使用 `ReentrantLock` → 锁竞争 + 线程上下文切换
- Disruptor → CAS + 内存屏障 → 用户态完成，无系统调用

### 面试题 12-3：如果 WebSocket 服务部署到多个实例，如何保证消息广播到所有实例的客户端？

**参考答案：**

**问题**：用户 A 在实例 1 上，用户 B 在实例 2 上，A 的编辑操作如何广播给 B？

**方案一：Redis Pub/Sub**
- 每个实例启动时订阅同一个 Redis Channel
- 实例 1 收到消息 → 本地处理 + 发布到 Redis
- 实例 2 的订阅者收到消息 → 广播给本地连接的客户端
- 优点：实现简单、无额外组件
- 缺点：Redis Pub/Sub 不持久化（消息丢失风险）

**方案二：消息队列（RabbitMQ / Kafka）**
- 使用 Fanout Exchange（RabbitMQ）或广播模式（Kafka）
- 每个实例订阅队列 → 收到消息后广播给本地客户端
- 优点：消息持久化、可靠投递
- 缺点：引入额外组件，增加运维复杂度

**方案三：一致性 Hash 路由**
- Nginx / Gateway 根据 spaceId 做一致性 Hash
- 同一空间的用户始终路由到同一实例
- 优点：无需消息中间件
- 缺点：实例上下线会导致重分配、负载不均

**本项目**：当前单实例部署，如果扩展到多实例，推荐 Redis Pub/Sub 方案（改动最小、依赖已有 Redis）。

## 阶段 13：DDD 架构重构

### 学习目标
理解 DDD 核心概念、掌握四层架构设计、完成从三层到四层的重构。

### 详细知识点

#### 13.1 DDD 核心概念
- **有界上下文（Bounded Context）**：领域的边界划分，本项目 4 个：User / Picture / Space / SpaceUser
- **实体（Entity）**：有唯一标识的对象（User、Picture、Space）
- **值对象（Value Object）**：无唯一标识，通过属性值定义相等性（SpaceTypeEnum、UserRoleEnum）
- **聚合（Aggregate）**：实体和值对象的集合，通过聚合根访问（Picture 聚合、Space 聚合）
- **领域服务（DomainService）**：不适合放在 Entity 中的领域逻辑
- **仓储（Repository）**：领域层的持久化接口，由 Infrastructure 层实现

#### 13.2 DDD 四层架构
```
interfaces（接口层）
│  Controller / DTO / VO / Assembler
│  ↓ 依赖
application（应用层）
│  ApplicationService（流程编排 / 事务管理）
│  ↓ 依赖
domain（领域层）← 核心，不依赖任何框架
│  Entity / ValueObject / DomainService / Repository接口
│  ↑ 实现
infrastructure（基础设施层）
   RepositoryImpl / Mapper / COS / Redis / AI API
   
shared（横切层）
   Auth / WebSocket / 跨域共享代码
```

#### 13.3 依赖倒置
- **Domain 层定义接口**：`PictureRepository extends IService<Picture>`
- **Infrastructure 层实现接口**：`PictureRepositoryImpl extends ServiceImpl<PictureMapper, Picture> implements PictureRepository`
- **核心思想**：高层模块（Domain）不依赖低层模块（Infrastructure），二者都依赖抽象

#### 13.4 ApplicationService vs DomainService 职责边界
- **ApplicationService**：业务流程编排、事务管理、跨域调用协调（如上传涉及 COS + 压缩 + 配额更新）
- **DomainService**：单一领域内的纯业务逻辑（如 Picture 的审核状态变更规则）
- **判断标准**：不涉及跨服务调用 → 下沉到 DomainService；需要编排多个 DomainService → Application 层

#### 13.5 充血模型 vs 贫血模型
- **贫血模型**：Entity 只有 getter/setter，业务逻辑全在 Service
- **充血模型**：Entity 包含自身行为方法（如 `Picture.approve()`）
- **平衡**：简单逻辑放 Entity，复杂跨实体逻辑放 DomainService

### 面试题 13-1：DDD 和传统三层架构的本质区别是什么？什么样的项目适合 DDD？

**参考答案：**

**本质区别**：
- **三层架构**：以数据库为中心（数据驱动），Controller → Service → Mapper，Service 层容易膨胀成"大泥球"
- **DDD**：以领域模型为中心（领域驱动），代码组织围绕业务概念而非技术分层

**DDD 的核心价值**：
1. **业务逻辑内聚**：图片相关的所有代码在 Picture 领域内，修改不会波及 User 领域
2. **依赖方向可控**：所有依赖指向 Domain 层（依赖倒置），Domain 层可独立测试
3. **代码即文档**：目录结构直接反映业务概念（domain/picture/entity/Picture）

**DDD 适合的场景**：
- 业务规则复杂（多个状态流转、复杂的权限逻辑）
- 多个有界上下文需要清晰边界
- 长期维护的项目（业务会持续演进的）

**DDD 不适合的场景**：
- 简单 CRUD 系统（无复杂业务规则）
- 快速原型验证
- 团队对 DDD 不熟悉（学习成本高）

**本项目的适合性**：4 个有界上下文、RBAC 权限体系、多种空间模式、协同编辑，业务复杂度适中，适合 DDD。但文件数量确实增加了，简单查询路径变长了。

### 面试题 13-2：Repository 模式和 DAO 模式有什么区别？

**参考答案：**

| 维度 | DAO（Data Access Object） | Repository |
|------|--------------------------|-----------|
| 关注点 | 数据访问技术细节 | 领域对象的集合管理 |
| 层次 | 持久化层 | 领域层（接口）+ 基础设施层（实现） |
| 粒度 | 单表操作 | 聚合根操作 |
| 返回值 | 原始数据（Entity / List） | 领域对象（可包含业务含义） |
| 依赖方向 | Service → DAO | Domain ← Infrastructure |

**核心区别**：
- **DAO** 是从数据库角度设计的（一张表一个 DAO），接口反映数据库结构
- **Repository** 是从领域角度设计的（一个聚合一个 Repository），接口反映业务语义

**示例对比**：
```java
// DAO 风格：暴露数据库操作细节
User findById(Long id);
void updateSelective(User user);
List<User> findByStatus(Integer status);

// Repository 风格：以领域语言表达
Optional<User> findById(UserId id);
void save(User user);
List<User> findActiveUsers();
```

**本项目中的实现**：
```java
// Domain 层定义接口（领域语义）
public interface PictureRepository extends IService<Picture> {
    // IService 来自 MyBatis-Plus，提供 CRUD 基础方法
}

// Infrastructure 层实现
@Repository
public class PictureRepositoryImpl 
    extends ServiceImpl<PictureMapper, Picture> 
    implements PictureRepository {
}
```

### 面试题 13-3：你在 DDD 重构过程中，最纠结的一个技术决策是什么？

**参考答案：**

最纠结的是 **"是否所有查询都需要走 DomainService"**。

**场景**：首页图片列表查询（分页 + 多条件筛选 + 排序），这是一个纯查询操作，没有任何业务规则。

**两种方案**：
- 方案 A（严格 DDD）：查询也走 DomainService → ApplicationService → Controller。好处是架构一致性，代价是简单的查询也要经过 3 层。
- 方案 B（务实折中）：简单查询由 ApplicationService 直接调用 Repository/IService 方法完成，DomainService 只处理有业务规则的写操作。

**最终决策**：选择了方案 B。理由是 DDD 的目的是让复杂业务更清晰，而不是为了架构纯净性把简单事情变复杂。简单查询直接通过 MyBatis-Plus 的 `IService` 方法完成即可。

**判断标准**：
- 如果查询包含业务规则（如"用户只能看到自己空间 或 公共图库中审核通过的图片"） → 走 DomainService
- 如果查询是纯数据检索（如"所有审核通过的图片按时间倒序"） → ApplicationService 直接操作

**反思**：过度设计 DDD 同样有害。好的架构是在"纯粹性"和"实用性"之间找到平衡。

### 面试题 15-1：从手写 HTTP 到 LangChain4j，代码量减少了多少？核心优势是什么？

## 阶段 16：Docker 容器化 ⭐ 新增

### 学习目标
实现 Docker 多服务编排、多阶段构建优化镜像体积、环境变量注入安全管理密钥。

### 详细知识点

#### 16.1 Docker 核心概念
- **镜像 vs 容器**：镜像是静态模板，容器是运行实例
- **多阶段构建**：构建阶段（Maven/Node 编译）→ 运行阶段（JRE/Nginx 托管），最终镜像不含源码和构建工具
- **Docker Compose**：多服务编排工具，`docker-compose.yml` 声明服务拓扑
- **Volume**：数据持久化（MySQL 数据目录、Redis AOF 文件）
- **Network**：容器间通过服务名互相访问（bridge 网络模式）

#### 16.2 项目 Docker 架构
```
docker compose up
    ├── mysql:8.0          ← Volume 持久化 + 自动初始化 sql/ 目录脚本
    ├── redis:7-alpine     ← AOF 持久化 + 256MB 内存限制
    ├── aituku-backend     ← 多阶段构建 (Maven → JRE 瘦镜像)
    └── aituku-frontend    ← 多阶段构建 (Node 22 → Nginx 托管)
```
依赖链：`depends_on` + `healthcheck` 保证启动顺序 MySQL → Redis → Backend → Frontend

#### 16.3 后端 Dockerfile 关键设计
```dockerfile
# 阶段 1: Maven 编译
FROM maven:3.8.6-openjdk-11-slim AS builder
RUN mvn dependency:go-offline -B -q  # 利用 Docker 缓存层
RUN mvn package -DskipTests -B -q

# 阶段 2: JRE 运行（非 root 用户）
FROM openjdk:11-jre-slim
RUN groupadd -r aituku && useradd -r -g aituku aituku
USER aituku
```

#### 16.4 安全最佳实践
- 敏感配置全部通过 `.env` 环境变量注入，不写入镜像
- 非 root 用户运行
- HEALTHCHECK 健康检查
- `docker-compose.yml` 中 Redis 配置内存上限 + AOF 持久化

### 面试题 16-1：Docker 多阶段构建的好处是什么？

1. **镜像体积最小化**：最终镜像只包含 JRE + jar，不包含 Maven/Node/源码（体积可减少 80%+）
2. **安全攻击面缩减**：不包含编译工具链，减少潜在漏洞
3. **构建缓存优化**：pom.xml / package.json 单独复制并下载依赖，源码变更时依赖层可复用缓存
4. **分离构建与运行环境**：构建可用完整 JDK，运行仅需 JRE

### 面试题 16-2：docker-compose.yml 中用 depends_on 和 healthcheck 的区别？

- `depends_on`：仅保证容器启动顺序（A 在 B 之前启动），不等待服务就绪
- `healthcheck`：定义服务就绪标准（如 `mysqladmin ping` / `curl /api/health`）
- 正确做法：`depends_on` + `condition: service_healthy`，确保前置服务不仅启动了而且真正可用

### 面试题 18-1：ES 相比 MySQL LIKE 搜索的核心优势？

| 维度 | MySQL LIKE | Elasticsearch |
|------|-----------|---------------|
| 原理 | 全表扫描 B+Tree（LIKE '%x%' 不走索引） | 倒排索引 |

### 面试题 18-2：ES 不可用时如何保证搜索服务可用？

4. 保证搜索服务 100% 可用，只是性能有所下降（毫秒级→秒级）

## 阶段 19：安全纵深防御体系 ⭐ 新增

### 学习目标
构建多层安全防线，实现 OAuth2 第三方登录、令牌桶限流、XSS/CSRF 防护、安全响应头注入。

### 详细知识点

#### 19.1 6 层安全防线
```
HTTP 请求进入
    → 第1层: CORS 白名单校验（仅允许受信域名跨域）
    → 第2层: XSS 清洗 (Jsoup.clean(value, Safelist.none()))
    → 第3层: Bucket4j 令牌桶限流
              - 注册: 3次/分钟, 登录: 3次/分钟
              - 上传: 20次/分钟, 搜索: 10次/秒
    → 第4层: Sa-Token RBAC 鉴权 + @AuthCheck 注解
    → 第5层: Controller 参数校验 (ThrowUtils.throwIf)
    → 第6层: MyBatis-Plus #{} 参数化查询（防 SQL 注入）
```

#### 19.2 Bucket4j 令牌桶算法
- 相比固定窗口/滑动窗口更平滑，允许短期突发流量（桶容量 = rate）
- ConcurrentHashMap 缓存令牌桶，按需创建，内存友好
- 支持按 IP / userId 限流，分布式扩展可用 bucket4j-redis

#### 19.3 OAuth2 第三方登录 (JustAuth)
- **OAuth2 完整流程**：
  1. 用户点击"GitHub 登录" → 前端跳转 `GET /api/oauth/github/authorize`
  2. 后端生成随机 state（防 CSRF）+ 拼接 GitHub 授权 URL → 302 重定向
  3. 用户在 GitHub 确认授权 → GitHub 回调 `/api/oauth/github/callback?code=xxx&state=xxx`
  4. 后端用 code 换 access_token → 用 access_token 获取用户信息
  5. JustAuth 封装为 `AuthUser`（统一用户模型：uuid/username/avatar）
  6. 自动注册/登录流程：查数据库 → 存在则更新信息 → 不存在则自动注册 → Sa-Token 签发会话
- **零密码登录**：OAuth2 用户无需设置密码，首次登录即自动注册
- **账号绑定**：同一邮箱不同平台登录 → 自动识别为同一用户（可选策略）
- **支持平台**：GitHub / Google / WeChat / Gitee（JustAuth 支持 30+ 平台，扩展新平台只需添加配置）
- **OAuth2Config 配置**：平台级 ClientId/ClientSecret/RedirectUri + Scope 管理

#### 19.4 Bucket4j 令牌桶限流
- **@RateLimit 自定义注解**：
  ```java
  @RateLimit(key = "register", permitsPerSecond = 0.05)  // 3次/分钟 = 0.05次/秒
  @RateLimit(key = "login", permitsPerSecond = 0.05)      // 3次/分钟
  @RateLimit(key = "upload", permitsPerSecond = 0.33)     // 20次/分钟
  @RateLimit(key = "search", permitsPerSecond = 10)       // 10次/秒
  ```
- **RateLimitInterceptor**：AOP 切面拦截 @RateLimit 注解 → ConcurrentHashMap 缓存令牌桶 → tryConsume() 尝试消费
- **多维度限流**：支持按 IP、按 userId、按 IP+userId 组合限流
- **分布式扩展**：当前内存级令牌桶（单机有效），分布式场景可替换为 bucket4j-redis（基于 Redis 的分布式令牌桶）
- **限流响应**：429 Too Many Requests + `Retry-After` 响应头（告诉客户端何时可重试）

#### 19.5 安全响应头
| 响应头 | 防护目标 |
|--------|----------|
| CSP (Content-Security-Policy) | XSS / 数据注入 |
| X-Frame-Options: DENY | 点击劫持 |
| X-Content-Type-Options: nosniff | MIME 类型嗅探 |
| Strict-Transport-Security | 强制 HTTPS |
| Referrer-Policy | Referer 泄露 |
| Cross-Origin-Opener-Policy | 跨域窗口隔离 |

---

### 面试题 19-1：Bucket4j 令牌桶和固定窗口限流的区别？

- **固定窗口**：每个时间窗口（如 1 分钟）独立计数，窗口切换瞬间可能出现 2 倍流量突发
- **滑动窗口**：统计过去 N 秒的请求数，更精确但实现复杂
- **令牌桶**：以固定速率生成令牌，请求消耗令牌。允许短期突发（桶容量 = 积攒的令牌数），同时长期速率受限。更平滑、更适合生产环境

### 面试题 19-2：OAuth2 第三方登录如何防止 CSRF 攻击？

JustAuth 内置 state 参数校验：
1. 发起授权时生成随机 state，存入 Session
2. GitHub 回调时携带 state
3. 回调接口校验 state 是否匹配 → 不匹配则拒绝（CSRF 攻击者无法预测 state 值）

### 面试题 20-1：为什么 Repository 测试要用 Testcontainers 而不是 H2？

H2 是 Java 内存数据库，虽然启动快但：

### 面试题 20-2：Prometheus 的 Pull 模式和 Push 模式有什么区别？

- **Push（Pushgateway）**：短生命周期任务（如 Cron Job）主动推送指标。适合无法被拉取的场景
- Micrometer 作为指标门面，切换 Pull/Push 后端只需改配置（无需改代码）

### 面试题 20-3：K6 压测和你项目中 CI 的集成方式是什么？如何设定合理的性能基线？

**参考答案：**

**CI 集成方式**：
- PR → test.yml 中的 `perf-test` Job → 30 秒冒烟测试（10 VU）
- push main → 同样触发（确保主干性能不退化）
- 全量压测（11 分钟，500 VU）手动触发，用于上线前验证
- `continue-on-error: true` → 性能测试失败不阻断构建（避免网络波动影响 CI 通过率）

**性能基线设定**：
1. **基准采集**：在正常负载下运行 3 次，取 P95 平均值作为基线
2. **阈值公式**：基线 × 1.5（允许 50% 性能退化触发告警）
3. 本项目阈值：图片列表查询 P95 < 500ms、单图片查询 P95 < 200ms

## 阶段 21：Kafka Streams 事件驱动 ⭐ 新增

### 学习目标
实现基于 Kafka Streams 的事件驱动架构，完成图片操作事件溯源、审计日志、缓存失效通知。

### 详细知识点

#### 21.1 Kafka Streams 拓扑
```
picture-events (Topic, 4分区)
    ├── [分支1] 审计日志 → audit-events → AuditEventConsumer → audit_log 表
    ├── [分支2] 操作统计 → 5分钟滑动窗口聚合 → RocksDB 状态存储
    └── [分支3] 缓存/ES失效 → cache-invalidation → 清除 Redis/Caffeine + ES 重索引
    └── [错误] 死信队列 → picture-events-dlq (反序列化失败的消息)
```

#### 21.2 事件类型矩阵
| 事件 | 来源 | 审计级别 | 触发下游 |
|------|------|----------|----------|
| UPLOADED | 图片上传 | INFO | 审计 + 缓存失效 + ES 索引 |
| EDITED | 图片编辑 | INFO | 审计 + 缓存失效 + ES 重索引 |
| DELETED | 图片删除 | WARN | 审计 + 缓存失效 + ES 删除 |
| REVIEW_APPROVED | 审核通过 | INFO | 审计 + 通知用户 |
| REVIEW_REJECTED | 审核拒绝 | WARN | 审计 + 通知用户 |

#### 21.3 Exactly-Once 语义
- Kafka Streams 2.8+ 支持 `EXACTLY_ONCE_V2`
- 通过 Kafka 事务保证事件处理不重不丢
- pictureId 作为消息键 → 同一图片的事件路由到同一分区 → 保证有序性

---

### 面试题 21-1：Kafka Streams 和 Spring @Async 的区别？

- **@Async**：简单的异步方法调用，无状态、无持久化、无故障恢复。适合轻量异步任务
- **Kafka Streams**：声明式事件处理拓扑，Exactly-Once 语义 + 状态存储 + 故障恢复 + 水平扩展
- 本项目：图片操作事件需要多方消费（审计/统计/缓存失效），用 Kafka Streams 解耦各消费者

### 面试题 21-2：为什么用 pictureId 作为消息键？

1. 同一图片的 UPLOADED → EDITED → DELETED 事件有序（必须先创建后删除）
2. 按 pictureId 哈希分区 → 同一图片的事件路由到同一分区 → 分区内严格有序
3. 多消费者时可并行处理不同图片的事件

## 阶段 24：MCP 协议与 JSON-RPC 2.0 自建协议栈 ⭐ 新增

### 学习目标
理解 MCP（Model Context Protocol）协议的设计哲学，掌握 JSON-RPC 2.0 协议实现和双传输模式。

### 详细知识点

#### 24.1 MCP 协议架构
```
┌─────────────────────────────────────────────┐
│           MCP Host (IDE / Claude Desktop)     │
│                   ↕ JSON-RPC 2.0              │
│           MCP Client (Spring AI)              │
│         ↕ Stdio Transport / SSE Transport     │
│           MCP Server (自定义实现)              │
│  ┌─────────┬──────────┬──────────┬────────┐  │
│  │ Tool 1  │ Tool 2   │ Resource │ Prompt │  │
│  │ 查询    │ 写入     │ 数据源   │ 模板   │  │
│  └─────────┴──────────┴──────────┴────────┘  │
└─────────────────────────────────────────────┘
```

#### 24.2 JSON-RPC 2.0 协议对象模型
```java
// 请求
{ "jsonrpc": "2.0", "id": 1, "method": "tools/call",
  "params": { "name": "searchPokemon", "arguments": { "type": "fire" } } }

// 响应
{ "jsonrpc": "2.0", "id": 1, "result": { "content": [...] } }

// 错误（13 个标准错误码）
{ "jsonrpc": "2.0", "id": 1, "error": { "code": -32601, "message": "Method not found" } }
```

#### 24.3 四层协议栈架构
```
Layer 1: Codec（编码解码层）
  → JSON-RPC 2.0 消息序列化/反序列化 + 请求/响应匹配

Layer 2: Protocol Handler（协议处理层）
  → 方法路由、参数校验、错误码映射

Layer 3: Session Manager（会话管理层）
  → 连接生命周期、双向消息队列、心跳检测

Layer 4: Gateway（网关层）
  → 与 Spring AI ToolCallback 体系的 Bridge 层协作
  → 协议映射器支持多版本共存
```

#### 24.4 双传输模式
| 维度 | Stdio Transport | SSE Transport |
|------|----------------|---------------|
| 通信方式 | 子进程 stdin/stdout | HTTP SSE 长连接 |
| 适用场景 | 本地 CLI 工具、IDE 插件 | 远程 Agent、Web 服务 |
| 协议格式 | 换行分隔的 JSON | HTTP POST（请求）+ SSE（推送） |
| 优势 | 简单、无网络依赖 | 支持远程部署、更灵活 |

#### 24.5 与 Spring AI MCP 的 Bridge 层
- Spring AI 1.0 的 `ToolCallback` 体系通过 Bridge 层与自建 MCP 协议栈无缝协作
- 自建协议栈的 Tool 可以注册为 Spring AI 的 `ToolCallback` — ID 级别的互操作性
- 关键设计：**协议无关性** — Bridge 层抽象掉 JSON-RPC 细节，上层只看到 Tool 定义

### 面试题 24-2：Stdio 和 SSE 传输模式各适合什么场景？如何做协议映射？

**参考答案：**

**Stdio Transport 适用场景**：
- Claude Desktop 插件、VS Code 扩展、本地 CLI Agent
- 无网络依赖，通过子进程通信
- 消息格式：换行分隔的 JSON（`Content-Length: N\r\n\r\n{json}`）

**协议映射器设计**：

// 协议映射器接口
interface ProtocolMapper {

ProtocolMapper V1 = new JsonRpcV1Mapper();  // 仅支持基础 Tool 调用
ProtocolMapper V2 = new JsonRpcV2Mapper();  // 支持 Resource + Prompt

关键设计：映射器模式实现协议版本共存，同一个 Tool 可同时响应 V1 和 V2 协议请求。

## 阶段 25：Plan-and-Execute 多智能体协作 ⭐ 新增

### 学习目标
理解 Plan-and-Execute 多智能体架构的设计思想，掌握 Kahn 拓扑排序在任务调度中的应用。

### 详细知识点

#### 25.1 架构总览
```
用户输入: "帮我找到火属性、攻击大于100的宝可梦，并推荐最佳配招"
    │
    ▼
Planner Agent (LLM 驱动)
    │ 任务拆解为 DAG 依赖图:
    │  [搜索火属性] ──┐
    │  [搜索攻击>100] ─┤→ [筛选交集] → [查询技能库] → [生成推荐]
    │                  │
    ▼                  │
Orchestrator (Kahn 拓扑排序)
    │ 第1层（并行）: [搜索火属性] + [搜索攻击>100]
    │ 第2层（串行）: [筛选交集]  ← 依赖第1层结果
    │ 第3层（串行）: [查询技能库] ← 依赖第2层结果
    │ 第4层（串行）: [生成推荐]   ← 依赖第3层结果
    │
    ▼
Worker 池 (共享工具集)
  ├── Worker-1: 执行 [搜索火属性]
  ├── Worker-2: 执行 [搜索攻击>100]
  └── Worker-N: 后续任务...

    │
    ▼
Reviewer Agent (质量评分 0-100)
  ├── 评分 ≥ 80 → 通过，输出结果
  └── 评分 < 80 → 不通过，重试（最多 2 次）
```

#### 25.2 Kahn 算法拓扑排序
- **算法原理**：计算每个节点的入度（依赖数），入度为 0 的节点可立即执行
- **并行策略**：同一层所有入度为 0 的任务并行执行；一个任务完成后，其所有后继节点的入度减 1
- **效果**：同层任务最大并行化，无依赖阻塞 → 效率相比单 Agent 顺序执行提升 55%

#### 25.3 各组件职责
| 组件 | 职责 | 输入 | 输出 | 测试数 |
|------|------|------|------|:---:|
| Planner | LLM 驱动任务拆解 | 自然语言用户输入 | DAG 依赖图（JSON） | — |
| Orchestrator | Kahn 排序 + 调度 | DAG 依赖图 | 执行计划 | 21 |
| Worker 池 | 执行单个子任务 | 子任务描述 + 工具集 | 子任务结果 | — |
| Reviewer | 质量评分 | 最终结果 + 原始需求 | 0-100 评分 + 通过/不通过 | — |

#### 25.4 错误处理与重试
- Worker 执行失败 → Orchestrator 标记失败 → 影响下游任务 → Planner 重新规划
- Reviewer 不通过 → 最多重试 2 次 → 2 次后仍不通过则降级输出（标注"未经验证"）
- 超时控制：每个子任务 30 秒超时 → 超时后跳过（标注"已跳过"）

---

### 面试题 25-1：为什么选择 Kahn 算法而不是简单的顺序执行？

**参考答案：**

**Kahn 算法的核心优势**：
1. **自动识别并行机会**：不需要硬编码哪些任务可并行，算法根据依赖图自动发现入度为 0 的任务
2. **最大并行度**：每一层所有无依赖的任务同时执行 → 总耗时 = 关键路径长度（而非任务总数）
3. **依赖安全**：绝不会出现任务 A 在依赖项 B 完成前开始执行的情况
4. **循环检测**：Kahn 算法天然能检测到循环依赖（退出时仍有入度 > 0 的节点）

**效率提升 55% 的计算**：
- 假设每个子任务耗时 ~2s
- 顺序执行：7 个任务 × 2s = 14s
- Kahn 调度：关键路径 4 层（第1层并行 2 任务 + 3 层串行）= 4 × 2s = 8s
- 提升：(14 − 8) / 14 ≈ 43%，实际测试中由于部分任务耗时不均，提升约 55%

### 面试题 25-2：Plan-and-Execute 和 ReAct 模式有什么区别？各适合什么场景？

**参考答案：**

| 维度 | ReAct | Plan-and-Execute |
|------|-------|-----------------|
| 决策方式 | 每步观察→思考→行动（逐步） | 先规划整体 → 再执行 |
| 灵活性 | 高（每步可调整方向） | 中（计划可重规划） |
| Token 消耗 | 高（每步推理都消耗） | 中（规划一次，执行复用） |
| 适合场景 | 探索性任务、信息检索 | 结构清晰的多步任务 |
| 不适合场景 | 长链任务（Token 成本高） | 高度不确定的任务 |

**本项目为什么选 Plan-and-Execute**：
- 宝可梦游戏场景任务结构化清晰（搜索 → 筛选 → 查询 → 推荐）
- Plan-and-Execute 的 Token 效率更高（规划一次，执行 N 步）
- 21 个测试覆盖各种依赖拓扑（线性、分叉、汇合、菱形）

## 阶段 26：三层记忆系统与 RAG 全链路 ⭐ 新增

### 学习目标
掌握 Agent 记忆系统的三层设计、RAG 全链路（ETL → 向量化 → 检索 → 生成）、查询重写优化。

### 详细知识点

#### 26.1 三层记忆架构
```
┌────────────────────────────────────────────────────┐
│  三层记忆系统 (28 测试)                              │
├──────────────┬──────────────┬──────────────────────┤
│ 工作记忆      │ 短期记忆      │ 长期记忆              │
│ (对话上下文)  │ (会话级缓存)  │ (跨会话持久化)        │
├──────────────┼──────────────┼──────────────────────┤
│ HashMap 内存  │ Redis 缓存    │ PostgreSQL/PgVector   │
│ 当前对话轮次  │ 最近 N 次对话 │ 历史所有对话          │
│ 生命周期: 请求│ 生命周期: 会话│ 生命周期: 永久         │
│ 容量: 4K token│ 容量: 100轮   │ 容量: 无限（向量压缩）│
└──────────────┴──────────────┴──────────────────────┘
```

#### 26.2 Map-Reduce 上下文压缩
- **问题**：对话历史越来越长，超过 LLM 上下文窗口限制
- **Map 阶段**：将长对话按轮次分段（如每 10 轮为一段），每段由 LLM 生成摘要
- **Reduce 阶段**：将多段摘要再次压缩为最终摘要
- **效果**：原始对话 10,000 tokens → 压缩后 ~4,000 tokens → **压缩率 60%+**
- **保留重点**：用户意图、关键实体、重要决策不丢失 — 纯粹闲聊部分被丢弃

#### 26.3 加权检索排序
```
最终相关性分数 = 关键词匹配 × 50% + 时间衰减 × 30% + 来源权重 × 20%

- 关键词匹配 (50%): 用户查询与记忆条目的 TF-IDF 余弦相似度
- 时间衰减 (30%): e^(-λ × 天数) — 越近的记忆权重越高
- 来源权重 (20%): 用户输入 > Agent 输出 > 系统消息
```

#### 26.4 RAG 全链路
```
数据准备 (ETL)
  ├── 宝可梦图鉴: 929 只宝可梦 (名称/属性/种族值/进化链)
  └── 技能库: 612 个技能 (威力/命中/PP/效果描述)

向量化 (Embedding)
  ├── text-embedding-3-small → 1536 维向量
  └── 批量处理: 每批 100 条，异步并发

向量存储 (PgVector)
  ├── HNSW 索引 (m=16, ef_construction=200) — 检索速度比 IVF Flat 快 5-10 倍
  ├── 混合搜索: 向量相似度 + 元数据过滤（属性=火、攻击>100）
  └── 连接池: HikariCP 20 连接

检索 (Retrieval)
  ├── 粗排: HNSW ANN Top-50
  └── 精排: 元数据过滤 + 相似度阈值 > 0.7

查询重写 (Query Rewriting)
  ├── 原始查询: "火系最强"
  ├── 重写后: "火属性 宝可梦 种族值 总和 排序 降序"
  └── 技术: LLM 驱动的查询扩展 + 同义词映射

生成 (Generation)
  ├── Prompt: System Message (角色设定) + Context (Top-10 检索结果) + User Message (用户查询)
  └── 结构化输出: { "pokemon": [...], "recommended_skills": [...], "reasoning": "..." }
```

### 面试题 26-1：Map-Reduce 压缩和固定窗口截断有什么区别？压缩率 60%+ 如何验证？

**参考答案：**

**固定窗口截断**（如只保留最近 10 轮）：
- 优点：简单、无额外 Token 消耗
- 缺点：可能丢掉早期重要信息（如用户在对话开始时提到的核心需求）

**Map-Reduce 压缩**：
- 优点：保留语义信息而非机械截断，压缩率可控
- 缺点：额外 LLM 调用成本（每次压缩约消耗 500-1000 tokens）

**压缩率验证方法**：
1. Token 计数对比：压缩前 `beforeTokens` vs 压缩后 `afterTokens`
2. 信息保留率：准备 10 个测试对话，人工标注关键信息点，检查压缩后保留了多少
3. 下游任务影响：压缩后的记忆用于下一次 Agent 调用，对比使用完整记忆的效果差异

### 面试题 26-2：PgVector 的 HNSW 索引和传统的 IVF Flat 索引有什么区别？

**参考答案：**

| 维度 | HNSW | IVF Flat |
|------|------|----------|
| 原理 | 分层可导航小世界图 | 聚类中心 + 桶内暴力搜索 |
| 构建速度 | 慢（需构建图结构） | 快（K-means 聚类） |
| 查询速度 | 极快（图遍历，O(log N)） | 中等（先定位聚类，再暴力搜索） |
| 召回率 | 高（95%+，适合精准搜索） | 中（80-90%，依赖聚类质量） |
| 内存占用 | 高（存储图边） | 低（只存储聚类中心） |
| 适合数据量 | 百万 ~ 亿级 | 千万 ~ 亿级 |

**本项目选 HNSW 的原因**：
- 宝可梦数据量不大（< 2000 条），内存占用不是瓶颈
- 要求高召回率（用户搜索"火属性"不能漏掉宝可梦）
- HNSW 的查询速度优势让 RAG 检索延迟 < 50ms

### 面试题 27-1：HITL（Human-in-the-Loop）在 Agent 系统中为什么重要？如何平衡安全和用户体验？

**参考答案：**

**HITL 的重要性**：
1. **Agent 的不可预测性**：LLM 可能产生幻觉，导致 Agent 做出错误决策
2. **责任边界**：关键操作（删除数据、修改权限）必须有人的确认——否则出了问题无法追责
3. **合规要求**：某些行业（金融、医疗）监管要求关键决策必须有人参与

**平衡策略**：
- **分级审批**：读操作自动执行（体验好）、写操作二次确认（安全 + 体验平衡）、危险操作管理审批（安全优先）
- **智能判断**：分析用户历史行为，信任用户高频操作 → 降低审批级别
- **批量确认**：多个同类操作合并为一次确认（如"已为你找到 10 张猫的图片，确认全部添加标签？"）
- **可配置规则**：不同组织/用户可自定义 HITL 级别
