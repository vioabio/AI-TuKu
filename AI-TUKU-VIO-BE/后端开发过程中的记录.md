⑤ 用户登录失败："系统错误" —— 注册成功但使用相同账号密码登录报系统错误

问题现象：使用注册页面成功注册账号（如vio001/12345678），但使用相同账号密码在登录页面登录时提示"登录失败，系统错误"。

根因分析：
- User实体（com.vio.aitukuviobe.model.entity.User）中定义了vipExpireTime、vipCode、vipNumber三个字段
- 但数据库user表中并没有这些列（SQL建表时尚未包含会员相关字段）
- 注册时使用selectCount（SELECT COUNT(*)）和save（INSERT仅包含非null字段），不会触发缺失列的问题
- 登录时使用selectOne，MyBatis-Plus生成SQL会SELECT实体类的全部字段，包括不存在的vipExpireTime、vipCode、vipNumber
- MySQL报"Unknown column 'vipExpireTime'" → BadSqlGrammarException（RuntimeException）→ GlobalExceptionHandler捕获返回code=50000, message="系统错误"

解决方案：在User实体的vipExpireTime、vipCode、vipNumber三个字段上添加@TableField(exist = false)注解，告诉MyBatis-Plus这些字段不对应数据库列，SELECT时不再包含。

4. 开发带缓存的分页查询接口

新增 POST /picture/list/page/vo/cache
实现 Caffeine 本地缓存 + Redis 分布式缓存 两级缓存架构：

查询流程：
  ① 构建缓存 Key：查询条件 JSON → MD5 哈希 → "yupicture:listPictureVOByPage:" + hashKey
  ② 查 Caffeine 本地缓存 → 命中直接返回（~1-5ms）
  ③ 查 Redis 分布式缓存 → 命中回填 Caffeine 后返回（~5-10ms）
  ④ 未命中 → 查询数据库（~50ms）
  ⑤ 结果写入 Redis（5-10分钟随机过期防缓存雪崩）+ Caffeine（5分钟固定过期）

关键设计：
  - 缓存 Key 使用查询条件 JSON 的 MD5，确保相同查询命中缓存
  - Redis 过期时间 = 300s + random(0, 300)s，防止大量缓存同时过期引发雪崩
  - Caffeine 本地缓存 5 分钟过期，零网络开销
  - Caffeine 配置：initialCapacity=1024, maximumSize=10000

性能对比（预期）：
  无缓存：~50ms（直查数据库）
  Redis：~25ms（网络开销）
  Caffeine：~12ms（纯内存，比 Redis 再快 1 倍）

1. 编译错误：找不到符号 @Alias 注解

问题：CreateOutPaintingTaskRequest.java 编译报错"找不到符号: 类 Alias"
  位置：import org.springframework.web.bind.annotation.Alias（第7行）
       @Alias("x_scale")（第45行）
       @Alias("y_scale")（第52行）
       @Alias("add_watermark")（第59行）

根因：`org.springframework.web.bind.annotation.Alias` 不是 Spring 的标准注解，该类在 Spring Framework 中根本不存在。
  开发步骤文档中提到的 @Alias 属于误写，实际只有 Jackson 的 @JsonProperty 用于属性名映射。

解决：
  ① 删除 `import org.springframework.web.bind.annotation.Alias;`
  ② 删除所有 @Alias 注解
  ③ 修正 @JsonProperty 值——原先错误地映射为 camelCase（如 @JsonProperty("xScale")），
     但阿里云百炼 API 期望的是 snake_case 格式：
     - xScale → @JsonProperty("x_scale")
     - yScale → @JsonProperty("y_scale")
     - addWatermark → @JsonProperty("add_watermark")

涉及文件：model/dto/picture/CreateOutPaintingTaskRequest.java

2. 功能缺陷：普通用户无法查看公共图库图片

问题：普通用户（非管理员、非图片所有者）访问图片详情页时提示"无权限"
  影响范围：GET /api/picture/get/vo 接口

根因：PictureController.getPictureVOById 对所有图片不分公共/私有，一律调用 checkPictureAuth 校验权限。
  checkPictureAuth 的逻辑：公共图库(spaceId=null)→仅本人或管理员可操作；私有空间(spaceId≠null)→仅本人可操作。
  这导致普通用户无法查看其他用户上传的公共图库图片，违背了公共图库"所有人可见"的设计意图。

checkPictureAuth 的定位：这是一个"操作权限校验"方法，适用于编辑、删除、扩图等变更操作，
  但不适用于"查看"操作。查看权限应有不同的粒度：
  - 公共图库（spaceId=null）：任何人（登录用户）可查看
  - 私有空间（spaceId≠null）：仅空间创建人可查看

解决：修改 PictureController.getPictureVOById，仅在私有空间图片时调用 checkPictureAuth：
  修改前：
    User loginUser = userService.getLoginUser(request);
    pictureService.checkPictureAuth(loginUser, picture);
  修改后：
    User loginUser = userService.getLoginUser(request);
    Long spaceId = picture.getSpaceId();
    if (spaceId != null) {
        pictureService.checkPictureAuth(loginUser, picture);
    }

涉及文件：controller/PictureController.java（第185行）

3. 诊断增强：AI 扩图任务失败但无法定位原因

     - HTTP 错误时：追加打印 HTTP 状态码

响应：List<SpaceCategoryAnalyzeResponse>（category, count, totalSize）

  空分类显示为"未分类"

3. 空间图片标签分析（POST /space/analyze/tag）

请求：SpaceTagAnalyzeRequest extends SpaceAnalyzeRequest
响应：List<SpaceTagAnalyzeResponse>（tag, count）

逻辑：只查 tags 列 → selectObjs 获取 JSON 字符串列表 → Java Stream 解析 JSON 数组
  → flatMap 合并 → groupingBy + counting 统计 → 按 count 降序排序

4. 空间图片大小分析（POST /space/analyze/size）

请求：SpaceSizeAnalyzeRequest extends SpaceAnalyzeRequest
响应：List<SpaceSizeAnalyzeResponse>（sizeRange, count）

逻辑：只查 picSize 列 → 按 4 个分段统计（<100KB, 100KB-500KB, 500KB-1MB, >1MB）
  使用 LinkedHashMap 保持分段顺序

5. 用户上传行为分析（POST /space/analyze/user）

请求：SpaceUserAnalyzeRequest extends SpaceAnalyzeRequest（+ userId, timeDimension）
响应：List<SpaceUserAnalyzeResponse>（period, count）

逻辑：按 timeDimension 选择 MySQL 日期函数：
  - day → DATE_FORMAT(createTime, '%Y-%m-%d')
  - week → YEARWEEK(createTime)
  - month → DATE_FORMAT(createTime, '%Y-%m')
  groupBy("period").orderByAsc("period")

6. 空间使用排行分析（POST /space/analyze/rank）

请求：SpaceRankAnalyzeRequest（topN=10）
响应：List<Space>

逻辑：仅管理员可访问。select 4 列 → orderByDesc("totalSize") → LIMIT topN

5. 登录切换账号报"系统错误"：Sa-Token StpKit 未初始化导致 login 异常

问题：普通用户退出后，使用管理员账号登录时报"系统错误"。
  后端日志：StpKit.SPACE.login() 调用失败（Sa-Token 缺少配置导致 StpLogic 未正确初始化）。

根因：StpKit 中使用 new StpLogic("space") 手动创建的 StpLogic 实例，
  在没有 Sa-Token 配置（application.yml）的情况下，其内部存储层未正确初始化，
  调用 login() 时抛出异常。该异常未被捕获，穿透到 GlobalExceptionHandler 的
  RuntimeException 处理器，返回"系统错误"。

解决：
  ① userLogin：将 Sa-Token 登录/会话保存包裹在 try-catch 中，失败只记 warn 日志
     修改前：
       StpKit.SPACE.login(user.getId());
       StpKit.SPACE.getSession().set(UserConstant.USER_LOGIN_STATE, user);
     修改后：
       try {
           try { StpKit.SPACE.logout(); } catch (Exception ignored) {}
           StpKit.SPACE.login(user.getId());
           StpKit.SPACE.getSession().set(UserConstant.USER_LOGIN_STATE, user);
       } catch (Exception e) {
           log.warn("Sa-Token 空间登录态保存失败（不影响正常登录）: {}", e.getMessage());
       }
  
  ② userLogout：同样用 try-catch 保护 Sa-Token 登出操作
     修改前：StpKit.SPACE.logout();
     修改后：try { StpKit.SPACE.logout(); } catch (Exception ignored) {}

设计思路：Sa-Token 在本项目中为辅助权限体系（原有 Session AOP 鉴权仍正常工作），
  即使 Sa-Token 登录/登出失败也不应阻塞正常用户流程。

涉及文件：service/impl/UserServiceImpl.java（userLogin + userLogout 方法）

根因：LMAX Disruptor 库中 `Disruptor` 类的正确包路径是 `com.lmax.disruptor.dsl.Disruptor`，
  而非 `com.lmax.disruptor.Disruptor`。在 disruptor-3.4.2.jar 中：
  - `com/lmax/disruptor/dsl/Disruptor.class` ← 正确位置（dsl 子包）
  - `com/lmax/disruptor/` 包下不存在 Disruptor.class → 编译失败

验证方法：`jar tf disruptor-3.4.2.jar | grep Disruptor` 显示 Disruptor.class 位于 dsl/ 子包下。

对比参考项目（yu-picture-backend）：
  - PictureEditEventProducer.java：`import com.lmax.disruptor.dsl.Disruptor;` ✓
  - PictureEditEventDisruptorConfig.java：`import com.lmax.disruptor.dsl.Disruptor;` ✓

解决：
  ① PictureEditEventProducer.java 第3行：
     修改前：import com.lmax.disruptor.Disruptor;
     修改后：import com.lmax.disruptor.dsl.Disruptor;
  
  ② PictureEditEventDisruptorConfig.java 第4行：
     修改前：import com.lmax.disruptor.Disruptor;
     修改后：import com.lmax.disruptor.dsl.Disruptor;
  
  其余 import（RingBuffer, BlockingWaitStrategy, ProducerType, DaemonThreadFactory）路径正确，无需修改。

涉及文件：
  manager/websocket/disruptor/PictureEditEventProducer.java
  manager/websocket/disruptor/PictureEditEventDisruptorConfig.java

2. 编译错误：无法取消引用int — SpaceTypeEnum.getValue().equals() 调用在基本类型上

问题：WsHandshakeInterceptor.java 第71行编译报错"java: 无法取消引用int"

根因：`SpaceTypeEnum.TEAM.getValue()` 返回 `int`（基本类型），Java 无法对基本类型调用 `.equals()` 方法。
  修改前：`!SpaceTypeEnum.TEAM.getValue().equals(space.getSpaceType())`
  这里 `getValue()` 返回的是 int 基本类型，.equals() 是对象方法，需要显式装箱为 Integer 才能调用。

对比参考项目（yu-picture-backend）：
  WsHandshakeInterceptor.java 第87行：`space.getSpaceType() != SpaceTypeEnum.TEAM.getValue()`
  使用 `!=` 进行基本类型比较，Java 自动拆箱 `space.getSpaceType()` (Integer→int)，
  然后与 `SpaceTypeEnum.TEAM.getValue()` (int) 做基本类型比较。推荐此方式。

解决：`SpaceTypeEnum.TEAM.getValue().equals(...)` → `space.getSpaceType() != SpaceTypeEnum.TEAM.getValue()`
  先用 `== null` 短路保护，再用 `!=` 基本类型比较（Integer 自动拆箱为 int）。
  
  修改前：
    if (space.getSpaceType() == null
            || !SpaceTypeEnum.TEAM.getValue().equals(space.getSpaceType()))
  修改后：
    if (space.getSpaceType() == null
            || space.getSpaceType() != SpaceTypeEnum.TEAM.getValue())

涉及文件：manager/websocket/WsHandshakeInterceptor.java

3. 启动失败：循环依赖 PictureEditHandler → PictureEditEventProducer → DisruptorConfig → PictureEditEventWorkHandler → PictureEditHandler

问题：应用启动报错 BeanCurrentlyInCreationException，循环依赖链：
  webSocketConfig → pictureEditHandler → pictureEditEventProducer
  → pictureEditEventDisruptorConfig → pictureEditEventWorkHandler → pictureEditHandler

根因：四个 Bean 形成了完整的循环引用链：
  - PictureEditHandler 注入 PictureEditEventProducer（@Resource）
  - PictureEditEventProducer 注入 Disruptor<PictureEditEvent>（即 pictureEditEventDisruptorConfig 创建的 Bean）
  - PictureEditEventDisruptorConfig 注入 PictureEditEventWorkHandler（@Resource）
  - PictureEditEventWorkHandler 注入 PictureEditHandler（@Resource）
  Spring Boot 2.6+ 默认禁止循环引用，启动直接失败。

对比参考项目（yu-picture-backend）：
  PictureEditHandler.java 第40-41行：
    @Resource
    @Lazy   ← 关键的 @Lazy 注解
    private PictureEditEventProducer pictureEditEventProducer;
  
  @Lazy 延迟初始化打破循环：PictureEditEventProducer 的代理对象在首次调用时
  才真正创建，此时 PictureEditHandler 已经完成初始化。

解决：在 PictureEditHandler 中对 PictureEditEventProducer 添加 @Lazy 注解 + 新增 import
  修改前：
    @Resource
    private PictureEditEventProducer pictureEditEventProducer;
  修改后：
    @Resource
    @Lazy
    private PictureEditEventProducer pictureEditEventProducer;
  新增：import org.springframework.context.annotation.Lazy;

涉及文件：manager/websocket/PictureEditHandler.java

2026/6/28：DDD 架构重构

本次重构将项目从传统分层架构（controller → service → mapper）重构为 DDD（Domain-Driven Design）
四层架构 + 共享模块，参考项目 `yu-picture-backend-ddd`。

═══════════════════════════════════════════════════════════
架构总览
═══════════════════════════════════════════════════════════

新架构分为 5 个顶层模块：

1. interfaces/（接口层 / 用户界面层）
   - controller/：REST 控制器
   - dto/：请求数据传输对象（按有界上下文分包：user/picture/space/spaceuser）
   - vo/：视图对象（返回给前端）
   - assembler/：DTO↔Entity 装配器（静态工厂方法）

2. application/（应用层）
   - service/：应用服务接口（继承 MyBatis-Plus IService）
   - service/impl/：应用服务实现（继承 ServiceImpl，轻量编排 + VO 组装）

3. domain/（领域层）— 按有界上下文分包
   - user/：entity(User)、valueobject(UserRoleEnum)、repository(UserRepository)、
            service(UserDomainService+Impl)、constant(UserConstant)
   - picture/：entity(Picture)、valueobject(PictureReviewStatusEnum)、
               repository(PictureRepository)、service(PictureDomainService+Impl)
   - space/：entity(Space,SpaceUser)、valueobject(SpaceLevelEnum,SpaceRoleEnum,SpaceTypeEnum)、
             repository(SpaceRepository,SpaceUserRepository)、
             service(SpaceDomainService+Impl,SpaceUserDomainService+Impl)

4. infrastructure/（基础设施层）
   - annotation/：@AuthCheck 自定义注解
   - aop/：AuthInterceptor AOP 鉴权切面
   - api/：CosManager（腾讯云 COS）、aliyunai/（阿里云百炼 AI API + model DTOs）
   - common/：BaseResponse、ResultUtils、PageRequest、DeleteRequest
   - config/：CorsConfig、CosClientConfig、JsonConfig、MybatisPlusConfig、WebSocketConfig、
             HttpRequestWrapperFilter、RequestWrapper
   - exception/：BusinessException、ErrorCode、GlobalExceptionHandler、ThrowUtils
   - manager/upload/：PictureUploadTemplate(抽象)、FilePictureUpload、UrlPictureUpload、
                     FileManager(@Deprecated)、UploadPictureResult
   - mapper/：UserMapper、PictureMapper、SpaceMapper、SpaceUserMapper（MyBatis-Plus BaseMapper）
   - repository/：UserRepositoryImpl、PictureRepositoryImpl、SpaceRepositoryImpl、
                  SpaceUserRepositoryImpl（仓储实现）

5. shared/（共享层 / 跨域横切）
   - auth/annotation/：@SaSpaceCheckPermission（Sa-Token 空间权限注解）
   - auth/model/：SpaceUserPermissionConstant、SpaceUserAuthConfig、SpaceUserAuthContext
   - auth/：StpKit（多账号体系）、StpInterfaceImpl（权限加载）、SpaceUserAuthManager（权限管理器）
   - websocket/：PictureEditHandler、WsHandshakeInterceptor、disruptor/（4个Disruptor文件）、
                 model/（4个消息模型）

关键技术点
═══════════════════════════════════════════════════════════

• DDD 四层架构：interfaces → application → domain → infrastructure，依赖方向向内
• 有界上下文划分：user / picture / space 三个领域上下文，各自独立分包
• 仓储模式：领域层定义接口（纯抽象），基础设施层提供 MyBatis-Plus 实现
• 领域服务 vs 应用服务：领域服务包含核心业务逻辑；应用服务负责编排+VO组装
• 装配器模式：interfaces/assembler/ 集中管理 DTO↔Entity 转换
• 共享模块：Sa-Token 鉴权 + WebSocket 协同编辑等横切关注点独立为 shared/
• 向后兼容：旧包目录保留，可渐进式迁移或删除
