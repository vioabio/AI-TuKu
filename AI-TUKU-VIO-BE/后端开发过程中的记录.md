6.数据精度类型
前端js的数据精度小于后端lang类型
在项目中使用全局精度进行解决：com.vio.aitukuviobe.config.JsonConfig
原理：Jackson序列化时，将Long类型统一转为String，避免前端Number精度丢失（JS的Number最大安全整数为2^53-1，而Java Long可达2^63-1）

8.用户注册接口 POST /api/user/register
流程：校验参数（非空、长度、两次密码一致性）→ 检查账号重复 → 密码MD5加盐加密 → 插入数据库 → 返回新用户ID
默认昵称："无名"
默认角色：user
请求体：UserRegisterRequest（userAccount, userPassword, checkPassword）

9.用户登录接口 POST /api/user/login
流程：参数校验 → 密码加密 → 数据库匹配账号密码 → 存入Session → 返回脱敏用户信息
登录态通过HttpSession维护，key为UserConstant.USER_LOGIN_STATE
返回LoginUserVO（不含密码等敏感字段）
请求体：UserLoginRequest（userAccount, userPassword）

10.用户注销 POST /api/user/logout
流程：检查登录态 → 移除Session中的用户信息

11.获取当前登录用户 GET /api/user/get/login
从Session中获取当前登录用户，返回脱敏后的LoginUserVO

12.用户增删改查（管理员权限）

13.用户角色与权限控制
角色枚举：UserRoleEnum（USER/VIP/ADMIN），位于com.vio.aitukuviobe.model.enums
权限常量：UserConstant中定义DEFAULT_ROLE="user"、VIP_ROLE="vip"、ADMIN_ROLE="admin"
使用自定义注解@AuthCheck(mustRole)标记需要权限的接口
通过AOP切面AuthInterceptor拦截@AuthCheck注解，从Session获取当前用户，比对角色是否满足要求
不需要权限时mustRole为空字符串，直接放行

⑤ 用户登录失败："系统错误" —— 注册成功但使用相同账号密码登录报系统错误

问题现象：使用注册页面成功注册账号（如vio001/12345678），但使用相同账号密码在登录页面登录时提示"登录失败，系统错误"。

根因分析：
- User实体（com.vio.aitukuviobe.model.entity.User）中定义了vipExpireTime、vipCode、vipNumber三个字段
- 但数据库user表中并没有这些列（SQL建表时尚未包含会员相关字段）
- 注册时使用selectCount（SELECT COUNT(*)）和save（INSERT仅包含非null字段），不会触发缺失列的问题
- 登录时使用selectOne，MyBatis-Plus生成SQL会SELECT实体类的全部字段，包括不存在的vipExpireTime、vipCode、vipNumber
- MySQL报"Unknown column 'vipExpireTime'" → BadSqlGrammarException（RuntimeException）→ GlobalExceptionHandler捕获返回code=50000, message="系统错误"

解决方案：在User实体的vipExpireTime、vipCode、vipNumber三个字段上添加@TableField(exist = false)注解，告诉MyBatis-Plus这些字段不对应数据库列，SELECT时不再包含。

涉及文件：com.vio.aitukuviobe.model.entity.User（第55/61/67行）
