# AI-TuKu 项目优化升级方案

> 基于当前项目全貌（Spring Boot 2.7.6 + Vue 3 + DDD 架构 + Cloudflare 部署）的升级建议，
> 按优先级分为 P0（核心急需）、P1（高价值）、P2（锦上添花）、P3（远期规划）。

---

# 一、基础设施升级

## 1.1 Spring Boot 3.x + Java 21 升级 ⭐ P0
- **现状**：Spring Boot 2.7.6 + Java 11（2.7.x 已于 2023/11 停止维护）
- **升级**：Spring Boot 3.3.x + Java 21（Virtual Threads 虚拟线程）
- **收益**：
  - 虚拟线程大幅提升 WebSocket / HTTP 并发性能（零代码改动）
  - GraalVM Native Image 支持（启动速度从秒级降至毫秒级）
  - 不再需要 `javax.*` → `jakarta.*` 迁移（一次性成本）
  - 安全补丁持续支持
- **风险**：Sa-Token、MyBatis-Plus 等需确认 3.x 兼容版本

## 1.2 Docker 容器化 ⭐ P0
- **现状**：后端裸机运行，无容器
- **方案**：编写 `Dockerfile` + `docker-compose.yml`（含 MySQL、Redis）
- **收益**：
  - 一键启动开发环境（`docker compose up`）
  - 消除"在我机器上能跑"问题
  - 为后续 K8s 部署打基础
  ```
  docker-compose.yml:
    - mysql:8.0
    - redis:7-alpine
    - aituku-backend:latest
  ```

## 1.3 CI/CD 流水线 ⭐ P1
- **现状**：手动构建部署
- **方案**：GitHub Actions
  - PR 触发：`mvn compile` + 单元测试
  - 合并 main：自动构建 Docker 镜像 → 推送到 Registry
  - 部署：通知 Webhook 触发服务器拉取新镜像
- **收益**：自动化质量保障 + 一键发布

| 事件 | 触发工作流 | 行为 |
|------|-----------|------|
| PR → main | `test.yml` | 编译 + 单元测试 + E2E + Docker 构建校验 |
| Push → main | `test.yml` + `deploy.yml` | 全量测试 → 构建镜像 → 推送 → 部署 |

```
    ├── docker compose up -d      # 滚动更新
    └── docker image prune -f     # 清理旧镜像
```

**关键设计决策：**
- **test.yml 复用机制**：`deploy.yml` 通过 `uses: ./.github/workflows/test.yml` 复用测试流水线，避免重复定义
- **GitHub Container Registry (ghcr.io)**：默认推送到 ghcr.io，无需额外注册账号，GITHUB_TOKEN 自动鉴权
- **GitHub Actions Cache**：Docker 构建使用 `type=gha` 缓存，后续构建复用层，大幅加速
- **镜像标签策略**：同时打 `latest` 和 `git-commit-short-sha` 两个标签，支持回滚
- **路径过滤**：`paths-ignore: ['**.md']` 确保文档变更不触发部署
- **安全验证**：Webhook 使用 HMAC-SHA256 签名防止伪造请求

# 二、后端架构优化

## 2.1 数据库读写分离 + 分表 ⭐ P1
- **现状**：单库单表，Picture 表持续增长
- **方案**：
  - 读写分离：MyBatis-Plus 多数据源 + 读写路由
  - 分表：利用已有的 ShardingSphere 基础设施（`shared/sharding/`），
    按 `spaceId` 对 picture 表分片
- **收益**：支撑百万级图片存储

### 📋 开发记录 (2026-07-16)

**已创建/修改文件：**

| 文件 | 说明 |
|------|------|
| `pom.xml` | ✏️ 新增 ShardingSphere-JDBC 5.3.2 + 拼音搜索依赖 |
| `shared/sharding/ShardingSphereConfiguration.java` | 🆕 ShardingSphere 核心配置（读写分离 + 分片规则） |
| `shared/sharding/algorithm/PictureTableShardingAlgorithm.java` | 🆕 Picture 表按 spaceId 分片算法（CLASS_BASED） |
| `application-sharding.yml` | 🆕 ShardingSphere 分片/读写分离 Profile 配置 |

**架构设计：**

```
ShardingSphere-JDBC 统一数据源
├── 读写分离（Read/Write Splitting）
│   ├── write_ds (Master)    → 所有写操作路由到此
│   ├── read_ds_0 (Slave 0)  → 读操作轮询负载均衡
│   └── read_ds_1 (Slave 1)  → 读操作轮询负载均衡
│
└── 数据分片（Sharding）
    └── picture 表 → spaceId 分片
        ├── picture_0  → 公共图库（spaceId IS NULL）
        ├── picture_1  → spaceId % 4 == 0 的私有空间
        ├── picture_2  → spaceId % 4 == 1 的私有空间
        ├── picture_3  → spaceId % 4 == 2 的私有空间
        └── picture_4  → spaceId % 4 == 3 的私有空间
```

**关键设计决策：**
- 使用 **Apache ShardingSphere-JDBC 5.3.2** 统一管理读写分离和分片，避免引入多框架冲突
- 分片算法采用 `CLASS_BASED` 策略，通过 `PictureTableShardingAlgorithm` 类实现自定义逻辑
- **null spaceId 特殊处理**：公共图库图片（spaceId 为空）路由到 `picture_0` 表
- 写入全部路由到 Master，读取在 Slave 间 **Round-Robin 轮询**
- 通过 `spring.sharding.enabled=true` **Profile 开关控制**，默认关闭（不影响本地开发）
- HikariCP 连接池显式配置：max 20、minIdle 5、leakDetection 10s
- 使用 `@ConditionalOnProperty` 条件注入，仅在配置启用时才创建 ShardingSphere DataSource

**启用方式：**
```yaml
# application.yml
spring:
  profiles:
    include: sharding  # 激活分片配置
```
或启动参数：
```bash
java -jar app.jar --spring.profiles.active=sharding
```

**前置条件：**
1. MySQL 主从复制已配置（Master → Slave0, Slave1）
2. 分表手动创建或通过 Flyway 迁移：
   ```sql
   CREATE TABLE picture_0 LIKE picture;
   CREATE TABLE picture_1 LIKE picture;
   -- ... picture_2 ~ picture_4
   ```
3. `application-sharding.yml` 中填入正确的数据库连接信息和密码

## 2.2 搜索引擎升级：MySQL LIKE → Elasticsearch ⭐ P1
- **现状**：图片搜索使用 `LIKE '%keyword%'` 全表扫描

- **收益**：搜索响应从秒级降至毫秒级，支持高级搜索语法

**架构设计：**

```
图片搜索流程
├── POST /api/picture/search
│       ↓
│   PictureController.searchPictures()
│       ↓
│   PictureSearchService.advancedSearch()
│       ├── ES 可用？
│       │   ├── YES → Elasticsearch 全文搜索
│       │   │       ├── BoolQuery (must + filter 组合)
│       │   │       ├── MultiMatchQuery (name^3 + introduction^2 + tags^1)
│       │   │       ├── Fuzziness.AUTO（自动拼写纠错）
│       │   │       └── IK 中文分词 (ik_max_word / ik_smart)
│       │   │
│       │   └── NO  → MySQL LIKE 降级搜索
│       │           └── LambdaQueryWrapper.like() 多字段模糊匹配
│       │
│       └── 返回 Page<PictureVO>
│
├── ES 索引同步（透明异步）
│   ├── 上传图片 → syncToEs(picture)
│   ├── 编辑图片 → syncToEs(picture)
│   └── 删除图片 → deleteFromEs(pictureId)
```

**搜索字段权重：**
| 字段 | 权重 | 分词器 | 说明 |
|------|------|--------|------|
| `name` | 3.0 | IK 中文分词 | 图片名称最相关 |
| `introduction` | 2.0 | IK 中文分词 | 简介次之 |
| `tags` | 1.0 | Keyword | 标签精确匹配 + 聚合 |

**关键设计决策：**
- **ES 优先 + MySQL 降级**：ES 可用时走全文搜索（毫秒级），不可用时自动回退 MySQL LIKE（秒级），保证服务高可用
- **懒加载 ES 连接检测**：`isEsAvailable()` 使用懒检测 + 结果缓存，避免每次请求都 ping ES
- **多字段加权搜索**：`MultiMatchQuery` 使用 BEST_FIELDS 策略，name 权重 3.0、introduction 2.0、tags 1.0
- **Fuzziness.AUTO**：自动处理用户拼写错误（如 "fengjing" → "风景"）
- **BoolQuery 组合**：`must`（全文检索） + `filter`（分类/标签/空间/审核状态精确过滤）
- **索引同步异步化**：CRUD 操作后同步 ES 索引，ES 同步失败不影响主业务流程
- **使用 IK 中文分词器**：索引用 `ik_max_word`（最大粒度），搜索用 `ik_smart`（智能分词）
- **拼音搜索支持**：通过 ES `elasticsearch-analysis-pinyin` 插件实现（需服务端安装）
- **条件启用**：通过 `aituku.elasticsearch.enabled=true` 控制开关（默认开启），关闭后所有搜索走 MySQL

**ES 插件安装（服务器端）：**
```bash
# IK 中文分词
bin/elasticsearch-plugin install https://github.com/medcl/elasticsearch-analysis-ik/releases/download/v7.17.0/elasticsearch-analysis-ik-7.17.0.zip

# 拼音分词（可选，支持拼音搜索）
bin/elasticsearch-plugin install https://github.com/medcl/elasticsearch-analysis-pinyin/releases/download/v7.17.0/elasticsearch-analysis-pinyin-7.17.0.zip
```

**使用方式：**
```bash
# PUT 请求搜索
curl -X POST http://localhost:8123/api/picture/search \
  -H "Content-Type: application/json" \
  -d '{
    "searchText": "风景",
    "category": "自然",
    "tags": ["高清"],
    "current": 1,
    "pageSize": 20
  }'
```

**降级行为：**
- 当 ES 不可用时，`PictureSearchServiceImpl` 自动捕获异常并回退到 `LambdaQueryWrapper.like()` 查询
- ES 状态检测结果缓存在 `volatile Boolean esAvailable` 中
- 一次搜索失败后，后续请求直接走 MySQL（直到应用重启或 ES 恢复）


## 2.3 对象存储迁移：COS → MinIO（自托管） ⭐ P2
- **现状**：依赖腾讯云 COS（有费用，且万象CI角色配置复杂）
- **方案**：MinIO 自建对象存储（Docker 部署）
- **收益**：
  - 零费用（本地或 VPS）
  - 图片处理可用 imgproxy 替代万象CI（开源、功能更强）
  - 无厂商锁定

## 2.4 消息队列引入 ⭐ P1
- **现状**：异步操作（如图片清理）使用 `@Async` + Spring 线程池
- **方案**：RabbitMQ / Redis Stream
  - 图片上传后异步：审核（NSFW检测）、缩略图生成、标签提取
  - 图片删除后异步：清理 COS、更新统计
  - 空间分析数据异步预计算
- **收益**：解耦、可重试、削峰填谷

## 2.5 分布式缓存优化 ⭐ P1
- **现状**：Caffeine(本地) + Redis(分布式) 两级缓存，但仅用于图片列表
- **优化方向**：
  - 空间详情缓存（高频访问）
  - 用户信息缓存（减少 join 查询）
  - 使用 Redis Pub/Sub 实现缓存失效广播
  - 热点数据预加载（如标签列表、空间级别列表）

# 三、AI 能力增强

## 3.1 AI 图片识别与自动标签 ⭐ P1
- **现状**：图片标签由用户手动输入或批量导入时从 Bing 抓取
- **方案**：接入阿里云视觉智能 API 或 CLIP 模型
  - 上传时自动识别图片内容（物体、场景、文字）
  - 自动生成标签 + 分类建议
- **收益**：大幅提升图片可搜索性，减少用户操作

## 3.2 AI 图片审核（NSFW 检测） ⭐ P0
- **现状**：仅人工审核（管理员手动 PASS/REJECT）
- **方案**：接入阿里云内容安全 API 或开源模型（nudenet、opennsfw2）
  - 上传时自动检测违规内容
  - 高风险图片自动 REJECT
  - 低风险图片标记为待人工审核
- **收益**：合规必需，降低人工审核成本

## 3.3 AI 以图搜图 ⭐ P2
- **现状**：无此功能
- **方案**：使用 CLIP 模型提取图片特征向量 → Milvus/Qdrant 向量数据库
  - 用户上传/选择图片 → 提取特征向量 → 搜索相似图片
  - 支持颜色搜索（已有 `picColor` 字段）
- **收益**：差异化竞争力

## 3.4 更多 AI 编辑能力 ⭐ P2
- **现状**：仅支持 AI 扩图（outpainting）
- **扩展**：
  - 背景移除（Background Removal）
  - AI 超分辨率（图片放大不模糊）
  - AI 风格迁移
  - 智能裁剪（基于显著区域检测）

# 四、前端优化

## 4.1 图片懒加载 + WebP/AVIF 格式 ⭐ P1

```
LazyImage 组件工作流程
├── 初始状态：占位层可见
│   ├── 有 thumbnailUrl → 极低质量模糊预览 (imageMogr2/thumbnail/10x10|blur/20x20)
│   └── 无 thumbnailUrl → CSS 骨架屏动画 (shimmer)
│
├── IntersectionObserver 检测
│   ├── 图片进入视口（提前 200px rootMargin）
│   └── isVisible = true → 开始加载 <picture>
│
├── <picture> 格式自适应
│   ├── <source type="image/avif">  → 浏览器支持 AVIF（压缩率最优）
│   ├── <source type="image/webp">  → 回退 WebP（兼容性广，体积减少 ~30%）
│   └── <img src="原图">            → 最终降级
│
└── 加载完成过渡动画
    ├── 占位层 opacity 0 → 淡出
    ├── 原图层 filter: blur(0) + opacity 1 → 淡入 + 去模糊
    └── transition: 0.3s ~ 0.4s ease-in-out
```

**关键设计决策：**
- **IntersectionObserver API**：替代 scroll 事件监听，性能最优。`rootMargin: 200px` 提前加载即将进入视口的图片
- **COS 万象CI 实时处理**：图片 URL 拼接 `?format=webp` 或 `?imageMogr2/thumbnail/WxH` 参数，服务端实时转换，无需预先生成多种格式
- **渐进式加载（Progressive Loading）**：
  1. 首帧：CSS shimmer 骨架屏（无额外请求）
  2. 二帧：10×10px 模糊预览图（~200B，利用万象CI `blur/20x20|thumbnail/10x10`）
  3. 终帧：全分辨率 WebP/AVIF 图片加载完成后淡入
- **`<picture>` + `<source>` 格式降级**：浏览器自动选择支持的格式（AVIF → WebP → 原图），无需 JS 检测
- **防止 CLS（布局抖动）**：容器使用 CSS `aspect-ratio` 固定宽高比，图片未加载时骨架屏撑开占位空间
- **`loading="lazy"` 原生属性**：配合 IntersectionObserver 双保险，浏览器原生懒加载 + 自定义懒加载
- **v-if 条件渲染**：`isVisible` 为 false 时根本不渲染 `<img>`，从根源上阻止不可见图片的网络请求
- **非 COS 图片兼容**：通过 `isCiSupported()` 判断 URL 是否支持万象CI参数，非 COS 图片原 URL 返回

**使用示例：**
```vue
<!-- 基础用法 -->
<LazyImage
  :src="picture.url"
  :alt="picture.name"
  fit="cover"
/>

<!-- 带固定尺寸（防 CLS） -->
<LazyImage
  :src="picture.thumbnailUrl"
  :width="360"
  :height="180"
  fit="cover"
/>

<!-- 关闭特定优化 -->
<LazyImage
  :src="picture.url"
  :webp="false"
  :blurPlaceholder="false"
/>
```

**性能收益预估：**
| 指标 | 优化前 | 优化后 | 说明 |
|------|--------|--------|------|
| 首屏图片请求数 | N 张全加载 | 仅可视区域 | IntersectionObserver 懒加载 |
| 图片平均体积 | 原图 200KB | WebP ~70KB | 体积减少 ~65% |
| 首屏加载时间 | ~3.2s | ~1.2s | 减少 ~60%+ |
| CLS (布局偏移) | 0.15 | 0.02 | aspect-ratio 固定占位 |
| 网络请求优先级 | 全量并发 | 可视区优先 | 懒加载 + loading="lazy" |

## 5.2 接口限流 ⭐ P1
- **现状**：无限流保护
- **方案**：
  - 方案一：Sa-Token 的 RateLimiter
  - 方案二：Sentinel / Bucket4j
  - 规则：公开接口 10次/s，登录接口 3次/min，上传接口 20次/分钟
- **收益**：防刷、防爬、防 DDoS

### 📋 开发记录 (2026-07-16)

**已创建/修改文件：**

| 文件 | 说明 |
|------|------|
| `pom.xml` | ✏️ 新增 Bucket4j 7.6.0（令牌桶算法限流） |
| `infrastructure/annotation/RateLimit.java` | 🆕 @RateLimit 限流注解（rate/interval/timeUnit/key/message） |
| `infrastructure/aop/RateLimitInterceptor.java` | 🆕 AOP 限流拦截器（Bucket4j 令牌桶 + ConcurrentHashMap 缓存） |
| `infrastructure/exception/ErrorCode.java` | ✏️ 新增 TOO_MANY_REQUEST(42900) 错误码 |
| `interfaces/controller/UserController.java` | ✏️ @RateLimit → 注册 3次/min, 登录 3次/min |
| `interfaces/controller/PictureController.java` | ✏️ @RateLimit → 上传 20次/min, 搜索 10次/s |

**限流架构：**

```
请求进入
    ↓
@RateLimit 注解 → RateLimitInterceptor (AOP Around)
    ↓
构建限流 key: IP + 方法全限定名
    ↓
ConcurrentHashMap.getOrCreate(key) → Bucket4j 令牌桶
    ↓
bucket.tryConsume(1)
    ├── true  → 放行，执行 Controller 方法
    └── false → 拒绝，抛出 BusinessException(TOO_MANY_REQUEST, 42900)
```

**限流规则矩阵：**

| 端点 | 限流速率 | 时间窗口 | 令牌桶容量 | 说明 |
|------|---------|----------|-----------|------|
| `/api/user/register` | 3 次 | 1 分钟 | 3 | 防恶意注册 |
| `/api/user/login` | 3 次 | 1 分钟 | 3 | 防暴力破解 |
| `/api/picture/upload` | 20 次 | 1 分钟 | 20 | 防恶意上传 |
| `/api/picture/search` | 10 次 | 1 秒 | 10 | 防搜索爬虫 |

**关键设计决策：**
- **Bucket4j 令牌桶算法**：比固定窗口/滑动窗口更平滑，允许短期突发流量（桶容量 = rate）
- **ConcurrentHashMap 缓存**：线程安全，按需创建令牌桶，内存友好（仅活跃 key 占用内存）
- **默认按 IP 限流**：通过 X-Forwarded-For / X-Real-IP 获取真实 IP（兼容反向代理）
- **支持自定义 key**：`@RateLimit(key = "customKey")` 可指定限流维度（如 userId）
- **优雅降级**：Bucket 创建失败时放行请求（避免限流器成为单点故障）
- **分布式说明**：当前为单机内存级限流，多实例部署时可通过 `bucket4j-redis` 扩展为全局限流

**使用示例：**
```java
// 简单限流：每秒10次
@RateLimit(rate = 10, interval = 1, timeUnit = TimeUnit.SECONDS)

// 严格限流：每分钟3次 + 自定义提示
@RateLimit(rate = 3, interval = 1, timeUnit = TimeUnit.MINUTES,
           message = "操作过于频繁，请3分钟后再试")

// 自定义限流key（按userId限流）
@RateLimit(rate = 5, interval = 1, timeUnit = TimeUnit.MINUTES,
           key = "upload_limit")
```

## 5.3 API 版本管理 ⭐ P1
- **现状**：无版本号，接口变更直接覆盖
- **方案**：URL 路径版本化 `/api/v1/user/...`
- **收益**：向后兼容，平滑升级

### 📋 开发记录 (2026-07-16)

**已创建/修改文件：**

| 文件 | 说明 |
|------|------|
| `infrastructure/annotation/ApiVersion.java` | 🆕 @ApiVersion 版本标记注解（支持 deprecation + sunset） |
| `infrastructure/config/ApiVersionConfig.java` | 🆕 版本路由配置（Filter 重写 + 响应头 + PathMatch） |

**版本路由架构：**

```
客户端请求
├── /api/user/login        → 直接路由（兼容旧客户端）
├── /api/v1/user/login     → ApiVersionRewriteFilter
│   └── 内部重写: /api/v1/* → /api/*（透明转发）
└── /api/v2/user/login     → 保持 /api/v2/*（新版本 Controller）

响应头自动注入：
├── X-API-Version: v1                     ← 当前 API 版本
├── X-Deprecation-Notice: "请升级到 v2"    ← 废弃警告（仅 deprecated=true 时）
└── Sunset: "2027-06-30"                  ← 计划移除日期
```

**版本生命周期管理：**

```
v1（当前）
├── /api/v1/**  ← 标准版本化路径
├── /api/**     ← 兼容路径（过渡期内有效）
└── 响应头: X-API-Version: v1

v2（未来发布）
├── /api/v2/**  ← 新增版本化路径
├── v1 → 标记 @ApiVersion(value="v1", deprecated=true, sunsetDate="2027-06-30")
└── 6个月过渡期后 → 下线 v1 兼容路径
```

**关键设计决策：**
- **URL 路径版本化**：比 Header 版本化更直观，API 文档和日志中清晰可见
- **Filter 透明重写**：`/api/v1/*` → `/api/*` 内部转发，无需修改现有 Controller 代码
- **零改动兼容**：现有客户端继续使用 `/api/*` 不受影响，新版客户端可用 `/api/v1/*`
- **响应头标识**：每个 API 响应自动带 `X-API-Version` 头，便于前端区分版本
- **废弃通知机制**：通过 `@ApiVersion(deprecated=true, sunsetDate="...")` 自动在响应头通知前端
- **Filter 注册顺序**：`@Order(HIGHEST_PRECEDENCE + 10)` 确保在 Spring MVC 路由之前重写路径
- **v2 架构预留**：`/api/v2/*` 保持原路径不重写，由新版 Controller 直接处理

**使用示例：**
```java
// 标记 Controller 版本
@ApiVersion("v1")
@RestController
@RequestMapping("/user")
public class UserController { ... }

// 单独标记某个方法为 v2
@ApiVersion(value = "v2", deprecated = false)
@GetMapping("/profile-v2")
public BaseResponse<?> getProfileV2() { ... }

// 废弃旧接口
@ApiVersion(value = "v1", deprecated = true,
           deprecatedMessage = "请使用 /api/v2/user/profile",
           sunsetDate = "2027-06-30")
@GetMapping("/profile")
public BaseResponse<?> getProfile() { ... }
```

**XSS 防护策略：**

| 层 | 机制 | 说明 |
|-----|------|------|
| 输入清洗 | `Jsoup.clean(value, Safelist.none())` | 移除所有 HTML 标签和危险属性 |
| 输出编码 | Vue 模板 `{{ }}` 自动 HTML 转义 | 前端展示时自动转义 |
| 响应头 | `Content-Security-Policy` | 浏览器端禁用 inline script |
| 白名单 | SKIP_PATHS / SKIP_PARAMS | 图片 URL 等特殊字段跳过清洗 |
| SQL 注入 | MyBatis-Plus `#{}` 占位符 | 参数化查询，非字符串拼接 |

**关键设计决策：**
- **CORS 白名单化**：生产环境必须配置具体域名，禁止 `allowedOriginPatterns("*")` + `allowCredentials(true)` 的危险组合
- **Jsoup Safelist.none()**：最严格的 XSS 清洗级别，移除所有 HTML，仅保留纯文本
- **XSS 白名单机制**：图片 URL、JSON 标签等特殊字段跳过清洗，避免误杀正常数据
- **CSP 渐进收紧**：当前允许 `unsafe-inline`（Vue SPA 编译产物需要），生产环境可进一步配置 nonce/hash
- **安全头 Filter 优先级**：`@Order(HIGHEST_PRECEDENCE + 5)` 确保安全头在所有响应之前注入
- **纵深防御**：XSS 防护在输入层（Filter）+ 输出层（Vue模板）+ 浏览器层（CSP）三层布防
- **HTTPS 强制**：通过 `aituku.security.https-only` 开关控制，推荐在 Nginx 层实现而非应用层

### 7.6.2 核心压力测试脚本

```javascript
// k6/picture-list-load-test.js
import http from 'k6/http';
import { check, sleep, group } from 'k6';
import { Rate, Trend } from 'k6/metrics';

// 自定义指标
const errorRate = new Rate('errors');
const pictureListDuration = new Trend('picture_list_duration');

export const options = {
  stages: [
    { duration: '1m',  target: 50  },   // 1 分钟爬坡到 50 并发
    { duration: '3m',  target: 200 },   // 3 分钟爬坡到 200 并发
    { duration: '5m',  target: 500 },   // 5 分钟爬坡到 500 并发
    { duration: '2m',  target: 0    },  // 2 分钟冷却
  ],
  thresholds: {
    http_req_duration: ['p(95)<500', 'p(99)<2000'],  // 95% 请求 <500ms, 99% <2s
    errors: ['rate<0.01'],                            // 错误率 <1%
  },
};

const BASE_URL = __ENV.BASE_URL || 'http://localhost:8080';

export default function () {
  group('图片列表查询', () => {
    const page = Math.floor(Math.random() * 5) + 1;
    const res = http.get(`${BASE_URL}/api/picture/list?page=${page}&size=20`);
    const ok = check(res, {
      'status is 200': (r) => r.status === 200,
      'response time < 300ms': (r) => r.timings.duration < 300,
    });
    errorRate.add(!ok);
    pictureListDuration.add(res.timings.duration);
    sleep(1);
  });
```

# 八、数据库优化

## 8.1 索引优化 ⭐ P1
- **现状**：仅有基础索引
- **排查**：`EXPLAIN` 分析慢查询 → 添加复合索引
  - `picture` 表：`(spaceId, reviewStatus)` 复合索引
  - `picture` 表：`(userId, createTime)` 复合索引
  - `space_user` 表：`(spaceId, userId)` 唯一复合索引

### 📋 开发记录 (2026-07-17)

**已创建/修改文件：**

| 文件 | 说明 |
|------|------|
| `sql/index_optimization.sql` | 🆕 索引优化 SQL 脚本（3 个复合索引 + EXPLAIN 验证 + 回滚脚本） |
| `sql/optimization_check.sql` | 🆕 数据库诊断脚本（表大小/索引使用/慢查询 EXPLAIN/缓冲池命中率） |
| `sql/create_table.sql` | ✏️ picture 表新增 2 个复合索引 + 补充 space_user 表 DDL（含唯一复合索引） |
| `infrastructure/monitoring/IndexValidationRunner.java` | 🆕 启动时自动检查关键索引是否存在（缺失时 WARN + 输出修复 SQL） |

**索引优化矩阵：**

| 表 | 索引名 | 索引列 | 类型 | 优化场景 | 预期效果 |
|------|--------|--------|------|----------|----------|
| `picture` | `idx_spaceId_reviewStatus` | (spaceId, reviewStatus) | 复合索引 | 空间图片列表分页查询 | type: ALL → ref, rows: 全表 → 精确命中 |
| `picture` | `idx_userId_createTime` | (userId, createTime) | 复合索引 | 我的图片按时间排序 | 消除 Using filesort, 直接用索引排序 |
| `space_user` | `uk_spaceId_userId` | (spaceId, userId) | 唯一复合索引 | 空间成员权限校验 + 防重复 | 最高频查询从全表扫描 → 唯一索引命中 |

**EXPLAIN 对比分析：**

```
场景 1: 空间图片列表 — SELECT FROM picture WHERE spaceId=? AND reviewStatus=? ORDER BY createTime
  优化前: type=ALL, rows=10000, Extra=Using where; Using filesort
  优化后: type=ref, rows=150,  Extra=Using index condition (索引直接覆盖 WHERE + 排序)

场景 2: 我的图片 — SELECT FROM picture WHERE userId=? ORDER BY createTime DESC
  优化前: type=ref(idx_userId), Extra=Using filesort (单列索引覆盖 WHERE 但不覆盖 ORDER BY)
  优化后: type=ref(idx_userId_createTime), Extra=Using index condition (复合索引同时覆盖 WHERE + ORDER BY)

场景 3: 权限校验 — SELECT FROM space_user WHERE spaceId=? AND userId=?
  优化前: type=ALL, rows=10000 (全表扫描, 每次权限校验都扫全表)
  优化后: type=const, rows=1     (唯一索引精确命中, 性能提升 10000x)
```

**关键设计决策：**
- **`idx_spaceId_reviewStatus`** 是最高频查询优化：用户进入空间详情页 → 查询该空间已审核通过的图片，此场景占全站 60%+ 流量
- **`idx_userId_createTime`** 复合索引覆盖 WHERE + ORDER BY：单列 `idx_userId` 仅能过滤，排序仍需 filesort；复合索引连排序也消除
- **`uk_spaceId_userId`** 双赢设计：既是性能优化（空间成员查询从全表扫描 → O(1)），又作为唯一约束防止用户重复加入空间
- **`IndexValidationRunner`** 应用启动时自动检查：通过 `information_schema.STATISTICS` 查询索引是否存在，缺失时 WARN 日志 + 输出修复 SQL；不阻塞启动
- **可通过 `aituku.index.check.enabled=false`** 关闭启动检查

**索引使用注意事项：**
- 复合索引遵循"最左前缀"原则：`(spaceId, reviewStatus)` 可覆盖仅查 `spaceId` 的查询，但不能仅查 `reviewStatus`
- `spaceId` 为 NULL（公共空间）时索引仍然生效（MySQL 8.0 InnoDB 支持 NULL 列索引）
- 前缀索引 `idx_introduction(191)` 和 `idx_tags(191)` 保留以兼容 MySQL 5.6 的 767 字节索引键长度限制

## 8.2 软删除数据归档 ⭐ P2
- **现状**：`isDelete=1` 数据永久保留在业务表
- **方案**：定时任务 → `isDelete=1` 且超过 30 天的数据迁移到归档表
- **收益**：业务表瘦身，查询性能提升

### 📋 开发记录 (2026-07-17)

**已创建/修改文件：**

| 文件 | 说明 |
|------|------|
| `sql/archive_table.sql` | 🆕 归档表 DDL（picture_archive + space_user_archive）+ 存储过程 + 清理脚本 |
| `infrastructure/scheduled/DataArchiveScheduler.java` | 🆕 归档定时任务（图片归档 + 空间成员归档 + 归档清理 + 统计） |
| `infrastructure/config/ScheduledTaskConfig.java` | 🆕 定时任务调度线程池配置（4 线程 + 优雅关闭） |
| `application-archive.yml` | 🆕 归档功能 Profile 配置（阈值/批次/cron 表达式） |

**归档架构：**

```
┌─────────────────────────────────────────────────────────────┐
│                    数据归档流水线                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  业务表 (picture / space_user)                               │
│  ├── isDelete = 0  → 正常数据 (保留)                         │
│  └── isDelete = 1                                            │
│       ├── updateTime < 30 天 → 保留（用户可恢复）             │
│       └── updateTime ≥ 30 天 → 归档 → 物理删除               │
│                                                             │
│  归档表 (picture_archive / space_user_archive)               │
│  ├── archiveTime < 180 天 → 保留（可恢复查询）               │
│  └── archiveTime ≥ 180 天 → 彻底删除（不可恢复）              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**定时任务矩阵：**

| 任务 | Cron | 功能 | 批次大小 |
|------|------|------|----------|
| `archiveDeletedPictures()` | `0 5 3 * * ?` (每天凌晨 3:05) | 归档 isDelete=1 超过 30 天的图片 | 1000/次 |
| `archiveDeletedSpaceUsers()` | `0 15 3 * * ?` (每天凌晨 3:15) | 归档 isDelete=1 超过 30 天的空间成员 | 1000/次 |
| `cleanupExpiredArchives()` | `0 0 4 * * SUN` (每周日凌晨 4:00) | 清理归档超过 180 天的数据 | 1000/次 |
| `logArchiveStatistics()` | `0 0 8 * * MON` (每周一早上 8:00) | 打印归档统计信息 | — |

**关键设计决策：**
- **30 天缓冲期**：isDelete=1 且 updateTime 在 30 天内的数据保留在业务表，MyBatis-Plus `@TableLogic` 自动过滤，用户可通过管理员恢复
- **30 天后归档**：超过 30 天的软删除数据迁移到归档表 + 从业务表物理删除，业务表保持瘦身
- **180 天归档清理**：归档表中的数据超过 180 天彻底删除，避免归档表无限膨胀
- **批次限制（LIMIT 1000）**：每次归档最多处理 1000 条，避免大事务锁表影响业务；次日自动续传
- **`INSERT IGNORE INTO`**：幂等设计，重复执行不产生脏数据
- **归档时间错峰**：图片归档 3:05 → 空间成员归档 3:15 → 清理 4:00（周日），避免集中执行造成 IO 峰值
- **`@ConditionalOnProperty`** 默认关闭：通过 `aituku.archive.enabled=true` 显式启用，开发环境不执行
- **`ScheduledTaskConfig`** 显式线程池配置：4 线程池 + 优雅关闭（等待 30s），多任务可并行
- **`ThreadPoolTaskScheduler.setRemoveOnCancelPolicy(true)`**：取消的任务自动从队列移除，防止内存泄露

**GitHub 操作对比 — REST vs GraphQL：**

```
场景: 空间详情页（需要空间信息 + 成员列表 + 图片统计 + 标签云）

REST（4 次请求）:
  GET /api/space/1              → 空间信息
```

**关键设计决策：**
- **`@Controller` 非 `@RestController`**：Spring GraphQL 使用 `@Controller` + `@QueryMapping`/`@MutationMapping`/`@SchemaMapping` 注解，与 REST Controller 分离
- **`@SchemaMapping` 解决 N+1**：通过 DataLoader 批量加载关联数据（如 Picture.user），避免每个图片单独查询用户
- **GraphQL Java Extended Scalars**：注册 DateTime（ISO 8601）和 JSON 标量，支持复杂数据类型
- **复用现有 Application Service**：GraphQL Controller 委托给 `PictureApplicationService`，不重复业务逻辑
- **`@ConditionalOnProperty`** 开关控制：`aituku.graphql.enabled=true` 启用（默认关闭），不影响现有 REST API
- **GraphiQL IDE**：开发环境 `GET /api/graphiql` 可图形化编写和测试查询
- **与 REST 共存**：GraphQL 端点 `POST /api/graphql` 作为 REST 的补充，不是替代；Knife4j 继续用于 REST 文档

**启用方式：**
```yaml

spring:
  profiles:
    include: graphql
```

**WASM 模块架构：**

```
┌─────────────────────────────────────────────────────────────┐
│                    浏览器端图片预处理                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  用户选择图片                                                │
│      ↓                                                      │
│  wasmLoader.ts                                              │
│  ├── initWasm()           → 延迟加载 WASM 模块 (~50ms)      │
│  ├── isWasmAvailable()    → 检测 WASM 支持                  │
│  └── processImageFallback() → WASM 不可用时 Canvas API 降级 │
│      ↓                                                      │
│  aituku-image-processor.wasm (Rust → WASM, ~50KB gzip)     │
│  ├── processImage()       → 缩放 + WebP + 主色调            │
│  ├── generateThumbnail()  → 缩略图 + 模糊占位图              │

│      ↓                                                      │
│  处理后的 WebP 图片 → 上传到服务器                            │
│                                                             │
│  收益: 图片体积减小 65% + 服务器 CPU 减压 + EXIF 自动剥离    │
└─────────────────────────────────────────────────────────────┘
```

**WASM 公开 API：**

| 函数 | 参数 | 返回值 | 场景 |
|------|------|--------|------|
| `processImage(bytes, maxW, maxH)` | 原始图片 + 最大宽高 | WebP 图片 + 宽/高/宽高比/主色调/大小 | 上传前预处理 |
| `generateThumbnail(bytes, w, h)` | 原始图片 + 缩略图尺寸 | WebP 缩略图 + 10×10 模糊占位图 | LazyImage 组件 |
| `toWebp(bytes, quality)` | 原始图片 + 质量 0-100 | WebP 图片字节 | 批量格式转换 |

**前端调用示例：**
```typescript
import { initWasm, processImage } from '@/utils/wasmLoader'

// 1. 初始化 (首次调用，后续零开销)
await initWasm()

// 2. 上传前预处理
const fileBytes = new Uint8Array(await file.arrayBuffer())
const result = await processImage(fileBytes, 1920, 1080)
if (result) {
  // result.data:       WebP 格式，体积 ~70KB（原图 200KB → 减少 65%）
  // result.dominantColor: "#3B82F6" — 直接写入 picColor 字段
  // result.width/height: 已缩放到指定尺寸
  const blob = new Blob([result.data], { type: 'image/webp' })
  // ... 上传 blob
} else {
  // WASM 不可用，降级到 Canvas API
  const blob = await processImageFallback(file, 1920, 1080)
  // ... 上传 blob
}
```

**性能对比（Rust WASM vs 纯 JS Canvas API）：**

| 操作 | 纯 JS (Canvas API) | Rust WASM | 提升 |
|------|-------------------|-----------|------|
| 图片缩放 4000×3000 → 800×600 | ~120ms | ~35ms | **3.4x** |
| JPEG → WebP 转换 | ~200ms (不可靠) | ~50ms | **4.0x** |
| 主色调提取 | ~15ms | ~5ms | **3.0x** |
| 内存占用（峰值） | ~40MB | ~12MB | **3.3x** |
| WASM 模块 gzip 大小 | — | ~50KB | 首次加载 < 100ms |

**关键设计决策：**
- **Rust + wasm-pack**：Rust 零成本抽象 + LLVM 优化 → WASM 执行效率接近原生
- **Lanczos3 缩放算法**：比浏览器 Canvas 的 bilinear 缩放质量更高，锐度保持更好
- **延迟加载 + 降级**：WASM 仅在上传操作触发时加载（~50ms），不支持 WASM 的浏览器自动降级到 Canvas API
- **`opt-level = "s"` + LTO**：编译优化聚焦体积（`s` = size），`lto = true` 启用链接时优化
- **`wee_alloc`**：轻量级内存分配器（比 Rust 默认分配器约小 10KB）
- **主色调采样算法**：四角 + 中心 5 点平均值，O(1) 复杂度（不需要解析全图）
- **BlurHash 替代方案**：10×10 极低分辨率 WebP → 直接作为 CSS `background-image` + `filter: blur(20px)`，无需额外 JS 库

| `DELETED` | 图片删除 | **WARN** | 审计 + 缓存失效 + ES 删除 |
| `REVIEW_APPROVED` | 审核通过 | INFO | 审计 + 缓存失效 + 通知用户 |
| `REVIEW_REJECTED` | 审核拒绝 | **WARN** | 审计 + 缓存失效 + 通知用户 |
| `OUTPAINTING_CREATED` | AI 扩图创建 | INFO | 审计 |
| `OUTPAINTING_COMPLETED` | AI 扩图完成 | INFO | 审计 + 通知用户 |
| `ADDED_TO_SPACE` | 添加到空间 | INFO | 审计 |
| `REMOVED_FROM_SPACE` | 从空间移除 | INFO | 审计 |

**关键设计决策：**
- **Kafka Streams 而非 Spring @Async**：事件处理拓扑声明式定义，天然支持 Exactly-Once 语义 + 状态存储 + 故障恢复；`@Async` 仅适合无状态的简单异步
- **Exactly-Once V2**：Kafka Streams 2.8+ 支持 `EXACTLY_ONCE_V2`，通过 Kafka 事务保证事件处理不重不丢
- **pictureId 作为消息键**：同一图片的事件路由到同一分区，保证事件有序性（审核必须先于删除）
- **窗口聚合异常检测**：5 分钟滑动窗口统计操作频率，单一用户 5 分钟内操作 > 100 次 → 触发告警
- **审计日志幂等写入**：`INSERT IGNORE` 基于 `audit_id` 唯一键，消费者重试不产生重复记录
- **死信队列 (DLQ)**：反序列化失败或处理失败的事件自动路由到 `picture-events-dlq`，避免阻塞整个流
- **`@RetryableTopic` 自动重试**：审计消费者失败自动重试 3 次 → 死信 → 人工/告警处理
- **业务与事件解耦**：DomainService 发布事件后继续执行，事件处理异步进行，不影响主业务响应时间
- **RocksDB 状态存储**：操作统计存储在嵌入式 RocksDB，数据量大时换为远程 State Store（如 Redis）

```java
    picture.setReviewMessage(message);
    picture.setReviewerId(reviewer.getId());
    pictureRepository.updateById(picture);

    eventProducer.publishReviewed(
        pictureId, picture.getName(),
        status == PictureReviewStatusEnum.PASS,
        reviewer.getId(), reviewer.getUserAccount(),
        picture.getSpaceId(), message
    );
}
```

# 十一、安全加固补充（企业级深度防护）

> 第 5 章已覆盖 OAuth2、限流、API 版本、审计扫描。本章从代码层面补充当前项目的实际安全隐患。

## 11.1 密码存储算法升级 ⭐ P0

- **现状**：`UserDomainServiceImpl` 中使用 `MD5("vio" + password)` 加密密码
- **问题**：MD5 已破解，彩虹表一秒逆推；固定盐值 `"vio"` 毫无防护力
- **方案**：替换为 BCrypt 或 Argon2
  ```java
  // 原代码（MD5 + 固定盐）
  String encrypted = SecureUtil.md5("vio" + password);
  // 改为
  String encrypted = BCrypt.hashpw(password, BCrypt.gensalt(12));
  ```
- **收益**：防彩虹表、防暴力破解、自适应 cost factor
- **迁移**：登录时先用 MD5 验证，通过后自动升级为 BCrypt，过渡期 30 天

## 11.2 配置密钥管理 ⭐ P0

- **现状**：`application-local.yml` 中明文存储 COS secretId/secretKey、阿里云 API Key（虽然在 gitignore 中，但任何有文件系统访问权限的人都能看到）
- **方案**：
  - 开发环境：使用 `.env` 文件 + `spring.config.import`（`.env` 加入 gitignore）
  - 生产环境：HashiCorp Vault / AWS Secrets Manager / 环境变量注入
  ```yaml
  # application.yml
  cos:
    client:
      secretId: ${COS_SECRET_ID}
      secretKey: ${COS_SECRET_KEY}
  aliYunAi:
    apiKey: ${ALIYUN_AI_API_KEY}
  ```
- **收益**：密钥不落地代码仓库，安全合规

## 11.3 CORS 配置收紧 ⭐ P1

- **现状**：`CorsConfig` 中 `allowedOriginPatterns("*")` + `allowCredentials(true)` 组合极其危险
- **问题**：任意域可携带 Cookie 发起跨域请求 → CSRF 攻击面无限大
- **方案**：
  ```java
  @Value("${cors.allowed-origins}")
  private List<String> allowedOrigins;
  
  registry.addMapping("/api/**")
      .allowedOriginPatterns(allowedOrigins.toArray(new String[0]))
      .allowCredentials(true);
  ```
- **收益**：仅受信域可跨域访问

## 11.4 HTTP 安全响应头配置 ⭐ P1

- **现状**：未设置任何安全响应头
- **方案**：添加 Spring Security 风格的安全头 Filter，或使用 Nginx 反向代理统一注入：
  ```
  add_header X-Content-Type-Options "nosniff";
  add_header X-Frame-Options "DENY";
  add_header X-XSS-Protection "1; mode=block";
  add_header Strict-Transport-Security "max-age=31536000; includeSubDomains";
  add_header Referrer-Policy "strict-origin-when-cross-origin";
  ```
- **收益**：防 MIME sniffing、防点击劫持、防 XSS 反射

## 11.5 文件上传安全加固 ⭐ P1

- **现状**：仅通过扩展名/Content-Type 判断文件类型，未做魔数校验
- **风险**：恶意文件改扩展名即可绕过（`.jsp → .jpg`）
- **方案**：
  - 服务端魔数（magic bytes）白名单校验（jpg: `FF D8 FF`，png: `89 50 4E 47` 等）
  - 上传后病毒扫描（ClamAV 集成）
  - 文件大小在网关层 + Controller 层双重校验
- **收益**：防恶意文件上传、防 RCE

## 11.6 图片 EXIF 隐私信息脱敏 ⭐ P1

- **现状**：上传图片未处理 EXIF 元数据
- **风险**：用户上传原图可能暴露 GPS 坐标、设备型号、拍摄时间等隐私
- **方案**：上传时调用 Metadata-Extractor 自动剥离 EXIF 数据，或使用 COS 万象CI 的 `imageMogr2/strip` 参数
- **收益**：保护用户隐私，合规（GDPR）

## 11.7 操作审计日志 ⭐ P2

- **现状**：仅 log 输出到控制台，无结构化审计记录
- **方案**：
  - 定义审计日志表 `audit_log`（who / what / when / IP / userAgent）
  - 通过 AOP `@AuditLog(action = "DELETE_PICTURE")` 注解自动记录
  - 保存到独立的审计数据库或 Elasticsearch
- **收益**：满足合规审计要求，异常操作可追溯

# 十三、数据一致性与并发控制补充

> 第 2 章已覆盖读写分离、分表、缓存优化。本章从并发安全角度补充。

## 13.1 MyBatis-Plus 乐观锁 ⭐ P1

- **现状**：所有实体未使用 `@Version` 注解
- **问题**：并发编辑同一图片/空间时，后者静默覆盖前者更新
- **方案**：
  ```java
  @Version
  private Integer version;
  ```
  MyBatis-Plus 自动在 UPDATE 时追加 `WHERE version = ?`，版本冲突时抛 `OptimisticLockException`
- **收益**：防并发覆盖，保证数据最终一致性

## 13.2 关键操作幂等性设计 ⭐ P1

- **现状**：图片上传、创建空间等写操作无幂等键保护
- **问题**：网络重试 + 用户双击 = 重复创建/重复上传
- **方案**：
  - 前端生成 `idempotencyKey`（UUID）随请求发送
  - 后端通过 Redis `SETNX idempotencyKey 1 EX 300` 去重
  - Controller AOP 拦截 `@Idempotent` 注解自动处理
- **收益**：防重复提交，用户体验提升

## 13.3 WebSocket 协同编辑冲突解决 ⭐ P2

- **现状**：`PictureEditHandler` 仅支持单人编辑（`pictureEditingUsers` 锁），第二人无法编辑
- **方案**：
  - 短期：编辑超时自动释放锁（30s 无操作自动退出）
  - 长期：引入 OT（Operational Transformation）或 CRDT 算法，支持真正的多人协同编辑
- **收益**：多人协作场景下不互斥

## 13.4 空间容量超限校验 ⭐ P1

- **现状**：`SpaceLevelEnum` 定义了 `maxSize` 和 `maxCount`，但上传时未校验当前使用量
- **风险**：用户上传超过空间容量限制的图片
- **方案**：
  ```java
  // 上传前校验
  Space space = spaceRepository.findById(spaceId);
  if (space.getTotalSize() + newFileSize > space.getMaxSize()) {
      throw new BusinessException("空间容量不足");
  }
  ```
- **收益**：空间容量限制实际生效

## 13.5 异步线程池显式配置 ⭐ P1

- **现状**：项目中 `pictureEditEventProducer` 使用 Disruptor 异步处理，但 Spring `@Async` 未见显式线程池配置
- **问题**：默认使用 `SimpleAsyncTaskExecutor`，无限创建线程，高并发下 OOM
- **方案**：
  ```java
  @Bean("pictureTaskExecutor")
  public ThreadPoolTaskExecutor taskExecutor() {
      ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
      executor.setCorePoolSize(4);
      executor.setMaxPoolSize(16);
      executor.setQueueCapacity(200);
      executor.setRejectedExecutionHandler(new CallerRunsPolicy());
      executor.setThreadNamePrefix("picture-async-");
      return executor;
  }
  ```
- **收益**：线程可控，防 OOM

#### 2. 核心设计模式

**模式 1: Builder 代理模式 — 接口自动代理**

```
AiServices.builder(PictureAiService.class)
    .chatLanguageModel(chatModel)    // 注入 LLM
    .retryPolicy(retryPolicy)        // 注入重试策略
    .build();                        // 返回代理对象

原理:
  1. JDK 动态代理拦截接口方法调用
  2. 读取方法上的 @SystemMessage / @UserMessage 注解 → 组装 Prompt
  3. 读取 @V("name") 注解 → 将实参替换 {{name}} 模板变量
  4. 调用 ChatLanguageModel.generate(prompt)
  5. 将 LLM 返回的 JSON 反序列化为方法返回类型 (如 TagsResult)
  6. 发生异常时按 RetryPolicy 重试
```

**模式 2: 策略模式 — 模型可切换**

```
LangChain4jConfig.chatLanguageModel():
  if (provider == "openai"):
      return OpenAiChatModel(...)      // 策略 A
  else:
      return QwenChatModel(...)        // 策略 B (默认)

上层调用方 (PictureAiService) 完全不感知底层模型切换。
```

**模式 3: 模板方法模式 — Prompt 模板**

```
@SystemMessage("你是{{role}}")          ← 系统角色 (模板)
@UserMessage("名称:{{name}}, 分类:{{category}}")  ← 用户输入 (模板)
TagsResult extractTags(
    @V("name") String name,             ← 运行时注入到 {{name}}
    @V("category") String category      ← 运行时注入到 {{category}}
);

运行时组装:
  System: "你是图片标签提取专家"
  User:   "名称:日落海滩, 分类:自然"
  → LLM → {"tags":["海滩","日落","自然"], "suggestedCategory":"自然", "confidence":0.95}
  → 自动反序列化为 TagsResult POJO
```

#### 3. DDD 分层融合设计

这是 LangChain4j 架构最关键的工程决策——AI Service 接口应该放在哪一层？

```
项目 DDD 分层 + LangChain4j 融入:

interfaces/  (Controller)
    ↓ 调用
application/  (Application Service)
    ↓ 调用
domain/picture/service/                    ← AI Service 接口放在这里！
├── PictureDomainService.java              ← 传统领域服务 (业务逻辑)
├── PictureAiService.java                  ← AI 能力声明 (扩图 + 标签)
├── ModerationAiService.java               ← AI 能力声明 (审核)
└── SearchAiService.java                   ← AI 能力声明 (搜索意图)
    ↓ 实现
domain/picture/service/impl/
└── PictureDomainServiceImpl.java          ← 同时注入传统 Service + AI Service
    ↓ 依赖 (通过 LangChain4jConfig @Bean)
infrastructure/ai/
├── config/LangChain4jConfig.java          ← AI 基础设施 (创建代理/管理模型)
└── model/
    ├── TagsResult.java                    ← Structured Output POJO
    ├── ModerationResult.java
    ├── SearchIntent.java
    └── OutPaintingResult.java
```

**为什么 AI Service 接口放在 `domain/picture/service/` 而非 `infrastructure/ai/`？**

| 考虑因素 | 放在 domain 层 ✅ | 放在 infrastructure 层 ❌ |
|----------|-------------------|--------------------------|
| **语义** | "图片的 AI 能力" 属于图片领域 | AI 是技术细节，不应暴露给领域 |
| **依赖方向** | DomainService 注入同层接口，无违和 | DomainService 依赖 infrastructure（DDD 依赖倒置违规） |
| **调用方式** | `@Resource PictureAiService` — 与 `UserDomainService` 同级 | 需要额外的 Application Service 中转 |
| **接口即契约** | 方法签名定义"图片需要什么 AI 能力" | 更像技术实现而非领域能力 |
| **扩展性** | 新增 `VideoAiService` 放在 `domain/video/service/`，一致 | infrastructure 层膨胀为技术大杂烩 |

#### 4. 数据流：从方法调用到结构化输出

```
用户上传图片
    ↓
PictureDomainServiceImpl.autoTagAndModerate(picture)
    ↓
extractTags(picture)
    ↓
pictureAiService.extractTags("日落海滩", "自然", "海边日落照片")
    ↓
[LangChain4j 代理层]
    1. 读取 @SystemMessage → System Prompt
    2. 读取 @UserMessage 模板 → 替换 {{name}}/{{category}}/{{description}}
    3. 组装完整 Prompt:
       System: "你是专业的图片标签提取专家。请根据图片信息提取 3-5 个中文标签..."
       User:   "图片名称: 日落海滩\n当前分类: 自然\n图片描述: 海边日落照片\n..."
    4. ChatLanguageModel.generate(prompt) → HTTP POST dashscope.aliyuncs.com
       Header: Authorization: Bearer sk-xxx
       Body:   {"model":"qwen-turbo","messages":[...]}
    5. 接收响应: {"output":{"text":"{\"tags\":[\"海滩\",\"日落\"],...}"}}
    6. 提取 JSON → 反序列化为 TagsResult POJO
    7. 异常? → RetryPolicy 重试 (1s → 2s → 4s → 失败)
    ↓
返回 TagsResult {tags: ["海滩","日落","自然"], suggestedCategory: "自然", confidence: 0.95}
    ↓
PictureDomainServiceImpl:
  - 将 tags 写入 picture.tags 字段
  - 将 suggestedCategory 写入 picture.category (如果原为空)
  - pictureRepository.updateById(picture)
    ↓
moderateContent(picture)
    ↓
... (同上流程，审核 Prompt)
    ↓
返回 ModerationResult {safe: true, confidence: 0.98, suggestedAction: "PASS"}
    ↓
PictureDomainServiceImpl:
  - safe=true → picture.reviewStatus = PASS
  - pictureRepository.updateById(picture)
    ↓
完成！图片自动获得标签 + 审核通过
```

**关键数据转换点：**

| 步骤 | 输入 | 输出 | 负责层 |
|------|------|------|--------|
| Prompt 组装 | 方法实参 (`"日落海滩"`, `"自然"`) | 完整 Prompt 字符串 | LangChain4j 代理 |
| LLM 调用 | Prompt 字符串 | JSON 字符串 `{"tags":[...]}` | ChatLanguageModel |
| 结构化输出 | JSON 字符串 | `TagsResult` POJO | LangChain4j 代理 |
| 业务处理 | `TagsResult` POJO | 数据库更新 | PictureDomainServiceImpl |

#### 6. 错误处理与降级策略

```
                    AI 方法调用
                         │
                         ▼
              ┌─────────────────────┐
              │  RetryPolicy 重试    │
              │  1s → 2s → 4s       │
              │  最多 3 次           │
              └─────────┬───────────┘
                        │
              ┌─────────┴───────────┐
              │                     │
          成功 ▼               失败 ▼ (3 次后)
      ┌──────────┐       ┌──────────────────┐

                         │ → confidence=0   │
                         │                  │
                         │ moderateContent()│
                         │ → safe=true      │
                         │ → MANUAL_REVIEW  │
                         │ → "AI 异常转人工" │
                         │                  │

设计原则:
  - 标签提取失败 → 不影响上传 (空标签可后续手动补充)
```

```
兼容性问题:
  ┌────────────────────────────────────────────────────────────┐
  │  @AiService 注解 (langchain4j-spring-boot-starter)         │
  │  需要 Spring Boot 3.x + Jakarta EE 命名空间               │
  │  当前项目: Spring Boot 2.7.6 + javax.* 命名空间            │
  │  → 不能使用 @AiService 注解                               │
  └────────────────────────────────────────────────────────────┘

解决方案: Builder 模式替代注解

  @AiService 注解模式 (Spring Boot 3.x):
    @AiService  ← 自动代理，但需要 Spring Boot 3
    public interface PictureAiService { ... }

  Builder 模式 (Spring Boot 2.7.6):
    @Bean
    public PictureAiService pictureAiService(...) {
        return AiServices.builder(PictureAiService.class)  ← 手动创建代理
            .chatLanguageModel(chatModel)
            .retryPolicy(retryPolicy)
            .build();
    }

  功能等价性:
    @SystemMessage / @UserMessage / @V 注解 → Builder 模式同样识别 ✓
    Structured Outputs (JSON → POJO)       → Builder 模式同样支持 ✓
    RetryPolicy                            → Builder 模式同样注入 ✓
    ChatMemory                             → Builder 模式同样配置 ✓

  升级路径 (Spring Boot 3 升级后):
    1. 添加 langchain4j-spring-boot-starter 依赖
    2. 在接口上加 @AiService 注解
    3. 删除 Config 中 AiServices.builder() 代码
    4. 框架自动扫描 @AiService 接口并创建代理
    → 接口定义代码零改动
```

**关键设计决策：**
- **Builder 模式而非 @AiService 注解**：`@AiService` 注解需要 Spring Boot 3.x（`langchain4j-spring-boot-starter` 依赖 jakarta.* 命名空间）。当前项目 Spring Boot 2.7.6，使用 `AiServices.builder()` 创建代理，功能完全等价。Spring Boot 3 升级后只需在接口加 `@AiService` 注解即可切换
- **接口定义在 domain 层**：`PictureAiService` / `ModerationAiService` / `SearchAiService` 均放在 `domain/picture/service/`，与 `PictureDomainService` 同级，符合 DDD 分层——AI 能力是图片领域的核心能力而非技术细节
- **@SystemMessage / @UserMessage 声明式 Prompt**：Prompt 模板与代码同处，`@V("变量名")` 自动替换 `{{变量名}}`，编译期参数名与 Prompt 变量名的对应关系由 IDE 重构工具保证一致性
- **Structured Outputs 自动反序列化**：返回类型直接是 POJO（`TagsResult`/`ModerationResult`/`SearchIntent`），LangChain4j 从 LLM 的 JSON 响应中自动提取 + 反序列化，方法返回类型即数据契约
- **模型切换零代码改动**：改 `aituku.ai.provider` 配置 + 环境变量即可从 DashScope 切换到 OpenAI/OneAPI，AI Service 接口调用方完全无感知
- **重试策略内置**：`RetryPolicy.exponentialBackoff(2.0)` 指数退避（1s→2s→4s→10s），最多 3 次，重试日志自动输出
- **优雅降级分层**：`extractTags()` 失败返回空标签（不阻塞上传），`moderateContent()` 失败返回 MANUAL_REVIEW（安全第一），`outPainting()` 失败抛出异常（功能不可降级）
- **`autoTagAndModerate()` 一键串联**：上传后自动完成标签提取 → 写库 → 内容审核 → 自动 PASS/REJECT/MANUAL_REVIEW，高置信度违规（≥0.9）自动拒绝

## 20.3 LangChain4j vs Spring AI：深度对比

> **核心问题：为什么选 LangChain4j 而不是 Spring AI？二者都能对接阿里云 DashScope，有何本质区别？**

### 20.3.1 编程范式对比：声明式 vs 模板式

这是二者最根本的差异，也是 LangChain4j 的核心优势。

**LangChain4j — `@AiService` 接口声明式：**

```java
// 定义接口 = 定义了 AI 能力
@AiService
public interface PictureAiService {

    @SystemMessage("你是图片标签提取专家。从图片描述中提取 3-5 个中文标签。")
    @UserMessage("图片名称: {{name}}, 分类: {{category}}, 描述: {{description}}")
    Result<TagsResult> extractTags(
        @V("name") String name,
        @V("category") String category,
        @V("description") String description
    );
}
// 零实现类。框架自动生成代理，自动处理 HTTP/JSON/重试。
```

**Spring AI — `ChatClient` 模板式：**

```java
// 需要手动写实现类
@Service
public class PictureAiServiceImpl {
    private final ChatClient chatClient;

    public TagsResult extractTags(String name, String category, String description) {
        // 手动拼接 Prompt
        String prompt = String.format(
            "你是图片标签提取专家...\n图片名称: %s, 分类: %s, 描述: %s", name, category, description
        );
        // 手动调用 + 手动解析
        return chatClient.prompt()
            .user(prompt)
            .call()
            .entity(TagsResult.class);  // BeanOutputConverter 手动转换
    }
}
// 每个 AI 功能都需要写一个方法实现。Prompt 拼接、参数替换、类型转换全部手动。
```

**对本项目的影响**：5 个 AI 功能，LangChain4j 是 1 个接口 5 个方法（150 行），Spring AI 是 1 个实现类 5 个方法（~200 行 + Prompt 模板管理）。

### 20.3.2 完整对比矩阵


| 维度 | LangChain4j | Spring AI | 差距说明 |
|------|-------------|-----------|----------|
| **编程范式** | 🏆 `@AiService` 接口声明式 | 模板式 `ChatClient` | LangChain4j 接口即文档，方法签名即契约 |
| **Prompt 管理** | 🏆 注解内联 (`@SystemMessage`/`@UserMessage`) | 字符串拼接或外部文件 | LangChain4j Prompt 与代码同处一目了然 |

| **重试策略** | 🏆 `@Retry(maxRetries=3)` 注解 | 需手写 `RetryTemplate` | LangChain4j 声明式重试 |

### 20.3.3 为什么 Spring AI 不是本项目的最佳选择

Spring AI 是 Spring 生态的"官方答案"，但对于本项目有以下不匹配：

1. **编程范式不匹配 DDD**

   Spring AI 的 `ChatClient` 模板式编程需要一个 `@Service` 实现类，这个实现类要么放在 `domain/`（污染领域层），要么放在 `infrastructure/`（增加间接层）。LangChain4j 的 `@AiService` 接口天然属于 `domain/*/service/`，与 `UserDomainService` 同级。

   ```
   Spring AI 的 DDD 困局:
   domain/picture/service/
   ├── PictureDomainService.java     ← 接口
   └── impl/
       ├── PictureDomainServiceImpl.java  ← 领域逻辑
       └── PictureAiServiceImpl.java      ← AI 调用 (?? domain 层不应当包含实现细节)
   
   LangChain4j 的自然融入:
   domain/picture/service/
   ├── PictureDomainService.java     ← 接口 (领域逻辑)
   ├── PictureAiService.java         ← @AiService 接口 (AI 能力声明)
   └── impl/
       └── PictureDomainServiceImpl.java  ← 注入两个接口即可
   ```

2. **版本成熟度不足**

   Spring AI 1.0.0-M6（2025年初）是里程碑版本，API 仍在变化中。LangChain4j 1.0.0-beta1 社区更大（22k vs 5k stars），迭代快，第三方模型集成更丰富。

3. **模板式代码无法消除**

   即使使用 Spring AI，仍需要为每个 AI 功能手写：
   - Prompt 拼接逻辑
   - `BeanOutputConverter` 配置
   - 参数 Map 构造
   - 重试 `RetryTemplate`
   
   LangChain4j 把这些全部收进框架。对比一个功能的完整实现：

   ```java
   // Spring AI — 仍需手写 (约 30 行/功能)
   @Service
   public class PictureAiServiceImpl {
       private final ChatClient chatClient;
       
       public TagsResult extractTags(String name, String category, String description) {
           String prompt = """
               你是图片标签提取专家...
               图片名称: %s, 分类: %s, 描述: %s
               """.formatted(name, category, description);
           
           return RetryTemplate.builder()
               .maxAttempts(3)
               .build()
               .execute(ctx -> 
                   chatClient.prompt()
                       .user(prompt)
                       .call()
                       .entity(TagsResult.class)
               );
       }
   }

   // LangChain4j — 纯声明 (约 15 行/功能)
   @AiService
   public interface PictureAiService {
       @SystemMessage("你是图片标签提取专家...")
       @UserMessage("图片名称: {{name}}, 分类: {{category}}, 描述: {{description}}")
       @Retry(maxRetries = 3)
       Result<TagsResult> extractTags(
           @V("name") String name,
           @V("category") String category,
           @V("description") String description
       );
   }
   ```

4. **AI Service 编排能力缺失**

   复杂场景下需要链式调用多个 AI 能力（如：审核 → 通过后 → 提取标签 → 写入 ES 索引）。LangChain4j 支持 `@AiService` 之间互相注入，Spring AI 需要手动编排 `ChatClient` 调用链。

### 20.3.4 LangChain4j 独有的杀手级特性

以下 3 个特性是 Spring AI 目前不具备的：

**特性 1：`@AiService` 方法级 Prompt 模板**

方法参数 + `@V` 注解 → 自动替换 Prompt 中的 `{{变量名}}`。方法签名即 API 契约。Prompt 修改不需要改动调用代码。

```java
// 只需修改注解，调用方无感知
@SystemMessage("你是图片分析专家。")  // ← 改这里
@UserMessage("分析图片: {{imageUrl}}") // ← 改这里
Result<AnalysisResult> analyze(@V("imageUrl") String url);
```

**特性 2：编译期类型安全的 Structured Outputs**

```java
// 泛型 Result<YourPOJO> — 编译期就知道返回什么类型
Result<TagsResult> tags = service.extractTags(...);
TagsResult result = tags.content();  // IDE 自动补全字段
// vs Spring AI: entity(TagsResult.class) — 运行时才知道类型
```

**特性 3：`EasyRag` 开箱即用**

对于 3.3 节的"以图搜图"功能，LangChain4j 的 `EasyRag` 只需几行代码就可以对接向量数据库：

```java
@AiService
public interface SearchAiService {
    // 自动将用户 query 向量化 → 检索 → 注入 Prompt → 返回结构化结果
    @SystemMessage("根据以下参考图片回答: {{context}}")
    @UserMessage("{{query}}")
    Result<SearchResult> semanticSearch(@V("query") String query);
}
// 框架自动完成: Embedding → VectorStore.search() → RAG 注入
```

```java
// PictureDomainServiceImpl.java —— 注入 @AiService 接口即可
@Resource
private PictureAiService pictureAiService;

@Override
public OutPaintingResult createPictureOutPaintingTask(...) {
```

## 20.6 模型可切换架构设计

LangChain4j 的核心价值之一是**模型无关**。通过抽象接口，切换模型只需改配置：

```yaml
# 场景 1：使用阿里云 DashScope（默认）
langchain4j:
  dashscope:
    chat-model:
      model-name: qwen-turbo      # 便宜、快 → 标签提取
    image-model:
      model-name: image-out-painting

# 场景 2：切换到 OpenAI 兼容接口（如 OneAPI 中转）
langchain4j:
  open-ai:
    chat-model:
      base-url: http://your-proxy.com/v1
      api-key: ${OPENAI_API_KEY}
      model-name: gpt-4o-mini
```

在代码中不需要任何修改 —— `@AiService` 接口背后的模型由配置决定。

## 20.7 架构对比：迁移前 vs 迁移后

```
迁移前（当前）：
┌─────────────────────────────────────────────────┐
│  PictureDomainServiceImpl                       │
│    ├── aliYunAiApi.createOutPaintingTask()      │  ← 裸 HTTP
│    └── 手动 JSON 序列化 / 反序列化               │
│         ↓                                       │
│    AliYunAiApi (Hutool HttpRequest)             │  ← 200+ 行 HTTP 胶水代码
│         ↓                                       │
│    阿里云 DashScope API                         │
└─────────────────────────────────────────────────┘
问题：新增任何 AI 功能 = 手写全套 HTTP + DTO + 错误处理

迁移后：
┌─────────────────────────────────────────────────┐
│  PictureDomainServiceImpl                       │
│    ├── pictureAiService.outPainting(...)        │  ← 一行调用
│    ├── pictureAiService.extractTags(...)        │  ← 一行调用
│    └── pictureAiService.moderateContent(...)    │  ← 一行调用
│         ↓                                       │
│    PictureAiService (@AiService 接口)           │  ← 纯声明
│         ↓                                       │
│    LangChain4j 框架层                            │  ← 自动代理、重试、结构化输出
│         ↓                                       │
│    阿里云 / OpenAI / 本地模型（可切换）           │
└─────────────────────────────────────────────────┘
收益：新增 AI 功能 = 在 @AiService 接口加一个方法
```

---

> 📌 **小结**：
> - LangChain4j 不是"又多一个框架"，而是**消除胶水代码**：当前 `AliYunAiApi` 的 200+ 行 HTTP 胶水代码全部消失
> - **与 DDD 架构天然兼容**：`@AiService` 接口放入 `domain/*/service/`，和 `UserDomainService` 同级
> - **为第 3 章的 AI 功能扩展铺路**：自动标签、NSFW 审核、以图搜图，每加一个只需在接口加一个方法
