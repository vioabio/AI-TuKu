# AI-TuKu 项目优化升级方案

> 基于当前项目全貌（Spring Boot 2.7.6 + Vue 3 + DDD 架构 + Cloudflare 部署）的升级建议，
> 按优先级分为 P0（核心急需）、P1（高价值）、P2（锦上添花）、P3（远期规划）。

---

# 一、基础设施升级

## 1.1 Spring Boot 3.x + Java 21 升级 ⭐ P0
- **现状**：Spring Boot 2.7.6 + Java 11（2.7.x 已于 2023/11 停止维护）
- **升级**：Spring Boot 3.3.x + Java 21（Virtual Threads 虚拟线程）
- **收益**：
  - 虚拟线程大幅提升 WebSocket / HTTP 并发性能（零代码改动）
  - GraalVM Native Image 支持（启动速度从秒级降至毫秒级）
  - 不再需要 `javax.*` → `jakarta.*` 迁移（一次性成本）
  - 安全补丁持续支持
- **风险**：Sa-Token、MyBatis-Plus 等需确认 3.x 兼容版本

## 1.2 Docker 容器化 ⭐ P0
- **现状**：后端裸机运行，无容器
- **方案**：编写 `Dockerfile` + `docker-compose.yml`（含 MySQL、Redis）
- **收益**：
  - 一键启动开发环境（`docker compose up`）
  - 消除"在我机器上能跑"问题
  - 为后续 K8s 部署打基础
  ```
  docker-compose.yml:
    - mysql:8.0
    - redis:7-alpine
    - aituku-backend:latest
  ```

## 1.3 CI/CD 流水线 ⭐ P1
- **现状**：手动构建部署
- **方案**：GitHub Actions
  - PR 触发：`mvn compile` + 单元测试
  - 合并 main：自动构建 Docker 镜像 → 推送到 Registry
  - 部署：通知 Webhook 触发服务器拉取新镜像
- **收益**：自动化质量保障 + 一键发布

# 二、后端架构优化

## 2.1 数据库读写分离 + 分表 ⭐ P1
- **现状**：单库单表，Picture 表持续增长
- **方案**：
  - 读写分离：MyBatis-Plus 多数据源 + 读写路由
  - 分表：利用已有的 ShardingSphere 基础设施（`shared/sharding/`），
    按 `spaceId` 对 picture 表分片
- **收益**：支撑百万级图片存储

## 2.2 搜索引擎升级：MySQL LIKE → Elasticsearch ⭐ P1
- **现状**：图片搜索使用 `LIKE '%keyword%'` 全表扫描
- **方案**：引入 Elasticsearch
  - 图片 name、introduction、tags、category 写入 ES 索引
  - 支持全文搜索、拼音搜索、模糊搜索、标签聚合
- **收益**：搜索响应从秒级降至毫秒级，支持高级搜索语法

## 2.3 对象存储迁移：COS → MinIO（自托管） ⭐ P2
- **现状**：依赖腾讯云 COS（有费用，且万象CI角色配置复杂）
- **方案**：MinIO 自建对象存储（Docker 部署）
- **收益**：
  - 零费用（本地或 VPS）
  - 图片处理可用 imgproxy 替代万象CI（开源、功能更强）
  - 无厂商锁定

## 2.4 消息队列引入 ⭐ P1
- **现状**：异步操作（如图片清理）使用 `@Async` + Spring 线程池
- **方案**：RabbitMQ / Redis Stream
  - 图片上传后异步：审核（NSFW检测）、缩略图生成、标签提取
  - 图片删除后异步：清理 COS、更新统计
  - 空间分析数据异步预计算
- **收益**：解耦、可重试、削峰填谷

## 2.5 分布式缓存优化 ⭐ P1
- **现状**：Caffeine(本地) + Redis(分布式) 两级缓存，但仅用于图片列表
- **优化方向**：
  - 空间详情缓存（高频访问）
  - 用户信息缓存（减少 join 查询）
  - 使用 Redis Pub/Sub 实现缓存失效广播
  - 热点数据预加载（如标签列表、空间级别列表）

# 三、AI 能力增强

## 3.1 AI 图片识别与自动标签 ⭐ P1
- **现状**：图片标签由用户手动输入或批量导入时从 Bing 抓取
- **方案**：接入阿里云视觉智能 API 或 CLIP 模型
  - 上传时自动识别图片内容（物体、场景、文字）
  - 自动生成标签 + 分类建议
- **收益**：大幅提升图片可搜索性，减少用户操作

## 3.2 AI 图片审核（NSFW 检测） ⭐ P0
- **现状**：仅人工审核（管理员手动 PASS/REJECT）
- **方案**：接入阿里云内容安全 API 或开源模型（nudenet、opennsfw2）
  - 上传时自动检测违规内容
  - 高风险图片自动 REJECT
  - 低风险图片标记为待人工审核
- **收益**：合规必需，降低人工审核成本

## 3.3 AI 以图搜图 ⭐ P2
- **现状**：无此功能
- **方案**：使用 CLIP 模型提取图片特征向量 → Milvus/Qdrant 向量数据库
  - 用户上传/选择图片 → 提取特征向量 → 搜索相似图片
  - 支持颜色搜索（已有 `picColor` 字段）
- **收益**：差异化竞争力

## 3.4 更多 AI 编辑能力 ⭐ P2
- **现状**：仅支持 AI 扩图（outpainting）
- **扩展**：
  - 背景移除（Background Removal）
  - AI 超分辨率（图片放大不模糊）
  - AI 风格迁移
  - 智能裁剪（基于显著区域检测）

# 四、前端优化

## 4.1 图片懒加载 + WebP/AVIF 格式 ⭐ P1
- **现状**：使用 `<a-image>` 但未系统优化
- **方案**：
  - Intersection Observer 懒加载（仅加载可视区域图片）
  - 图片 URL 自动拼接 `?format=webp`（万象CI 支持）
  - 缩略图使用 BlurHash 占位（更优雅的加载体验）
- **收益**：首屏加载时间减少 60%+

## 4.2 渐进式 Web 应用（PWA） ⭐ P2
- **现状**：纯在线应用
- **方案**：Vite PWA 插件 → Service Worker + manifest.json
  - 离线可浏览已缓存的图片列表
  - 添加到主屏幕
  - 推送通知（审核结果、协作消息）
- **收益**：类原生体验

## 4.3 国际化（i18n） ⭐ P2
- **现状**：仅中文
- **方案**：Vue I18n
- **收益**：可拓展海外用户

## 4.4 移动端适配 / Tauri 桌面应用 ⭐ P3
- **现状**：仅 PC 浏览器
- **方案**：
  - 移动端：响应式布局优化 + Capacitor 打包
  - 桌面端：Tauri（比 Electron 更轻量，Rust 后端）

# 五、安全加固

## 5.1 OAuth2 / 第三方登录 ⭐ P1
- **现状**：仅用户名+密码
- **方案**：Sa-Token OAuth2 模块 + JustAuth
  - 微信扫码登录（小程序场景）
  - GitHub/Google 登录
- **收益**：降低注册门槛

## 5.2 接口限流 ⭐ P1
- **现状**：无限流保护
- **方案**：
  - 方案一：Sa-Token 的 RateLimiter
  - 方案二：Sentinel / Bucket4j
  - 规则：公开接口 10次/s，登录接口 3次/min，上传接口 20次/分钟
- **收益**：防刷、防爬、防 DDoS

## 5.3 API 版本管理 ⭐ P1
- **现状**：无版本号，接口变更直接覆盖
- **方案**：URL 路径版本化 `/api/v1/user/...`
- **收益**：向后兼容，平滑升级

## 5.4 安全审计 ⭐ P2
- SQL 注入扫描（MyBatis-Plus 动态 SQL 是否有注入风险）
- XSS 防护（用户输入的 name/introduction 等）
- CSP（Content Security Policy）头配置
- HTTPS 强制跳转

# 六、可观测性

## 6.1 应用监控 ⭐ P1
- **现状**：仅 `log` 输出，无指标收集
- **方案**：Spring Boot Actuator + Micrometer + Prometheus + Grafana
  - JVM 指标（内存、GC、线程）
  - HTTP 请求 QPS / 延迟分布 / 错误率
  - 数据库连接池状态
  - 缓存命中率
- **收益**：从"凭感觉优化"到"数据驱动优化"

## 6.2 分布式链路追踪 ⭐ P2
- **现状**：无法追踪请求全链路
- **方案**：Micrometer Tracing + Zipkin/Jaeger
- **收益**：快速定位慢请求瓶颈

## 6.3 健康检查 + 告警 ⭐ P1
- **现状**：仅 `/api/health` 返回 ok
- **方案**：Spring Boot Actuator Health Endpoint
  - 检查 MySQL 连接、Redis 连接、COS 连通性
  - 告警：Prometheus AlertManager → 企业微信/钉钉通知

# 七、测试体系

> 本章对项目的测试体系做完整展开。按照**测试金字塔**（Test Pyramid）模型组织：
> 底座是大量单元测试（反馈快、成本低），中层是集成/API 测试，顶端是少量 E2E 和压力测试。

## 7.2 单元测试 ⭐ P0

### 7.2.1 后端单元测试分层

项目的 DDD 分层架构天然适合分层测试。每一层隔离依赖，互不干扰：

```
┌─────────────────────────────────────────────┐
│           测试层次            │   工具       │
├──────────────────────────────┼──────────────┤
│  1. 实体 / 值对象              │ JUnit 5 纯单测 │
│  2. DomainService 领域服务     │ Mockito Mock  Repository │
│  3. Repository 实现            │ Testcontainers + 真实 DB │
│  4. Controller / Application   │ @WebMvcTest + Mock Service │
│  5. 工具类 / 配置类            │ 纯 JUnit 5                │
└──────────────────────────────┴──────────────┘
```

#### 第一层：实体 & 值对象测试（纯单测，零依赖）

这是最基础也最廉价的测试。DDD 实体中包含业务不变量，例如：
- `User.password` 加密后必须能被验证
- `SpaceLevelEnum` 各等级有正确的容量限制
- `PictureReviewStatusEnum` 状态转换合法
- `Picture` 实体计算图片的宽高比、文件大小格式化

**POM 依赖：**

```xml
<!-- JUnit 5 已由 spring-boot-starter-test 引入 -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-test</artifactId>
    <scope>test</scope>
</dependency>
<!-- AssertJ 流式断言（比 Hamcrest 可读性更好）-->
<dependency>
    <groupId>org.assertj</groupId>
    <artifactId>assertj-core</artifactId>
    <version>3.26.3</version>
    <scope>test</scope>
</dependency>
```

**示例代码：**

```java
package com.vio.aitukuviobe.domain.picture.entity;

import static org.assertj.core.api.Assertions.*;

import com.vio.aitukuviobe.domain.picture.valueobject.PictureReviewStatusEnum;
import org.junit.jupiter.api.DisplayName;
import org.junit.jupiter.api.Nested;
import org.junit.jupiter.api.Test;

@DisplayName("Picture 实体单元测试")
class PictureTest {

    @Nested
    @DisplayName("审核状态流转")
    class ReviewStatusTransition {

        @Test
        @DisplayName("待审核 → 通过")
        void shouldTransitionFromPendingToPass() {
            Picture picture = Picture.builder()
                .reviewStatus(PictureReviewStatusEnum.PENDING.getValue())
                .reviewMessage("")
                .build();

            picture.setReviewStatus(PictureReviewStatusEnum.PASS.getValue());
            picture.setReviewMessage("审核通过");

            assertThat(picture.getReviewStatus())
                .isEqualTo(PictureReviewStatusEnum.PASS.getValue());
            assertThat(picture.getReviewMessage()).isEqualTo("审核通过");
        }

        @Test
        @DisplayName("已通过的图片不能再次审核")
        void passReviewedPictureShouldNotBeReReviewed() {
            Picture picture = Picture.builder()
                .reviewStatus(PictureReviewStatusEnum.PASS.getValue())
                .build();

            assertThatThrownBy(() -> picture.review(PictureReviewStatusEnum.REJECT, "违规"))
                .isInstanceOf(BusinessException.class)
                .hasMessageContaining("已审核");
        }
    }

    @Nested
    @DisplayName("图片 URL 格式化")
    class UrlFormatting {

        @Test
        @DisplayName("应生成正确的缩略图 URL")
        void shouldGenerateThumbnailUrl() {
            Picture picture = Picture.builder()
                .url("https://cos.example.com/bucket/pic.jpg")
                .build();

            String thumbnail = picture.getThumbnailUrl(200, 200);

            assertThat(thumbnail)
                .contains(pattern:"imageMogr2/thumbnail/200x200");
        }
    }
}
```

#### 第二层：DomainService 领域服务测试（Mock Repository）

领域服务是 DDD 的核心业务逻辑层。使用 Mockito 隔离 Repository 依赖。

```java
package com.vio.aitukuviobe.domain.user.service.impl;

import static org.assertj.core.api.Assertions.*;
import static org.mockito.ArgumentMatchers.*;
import static org.mockito.BDDMockito.*;

import com.vio.aitukuviobe.domain.user.entity.User;
import com.vio.aitukuviobe.domain.user.repository.UserRepository;
import com.vio.aitukuviobe.domain.user.valueobject.UserRoleEnum;
import com.vio.aitukuviobe.infrastructure.exception.BusinessException;
import org.junit.jupiter.api.BeforeEach;
import org.junit.jupiter.api.DisplayName;
import org.junit.jupiter.api.Nested;
import org.junit.jupiter.api.Test;
import org.junit.jupiter.api.extension.ExtendWith;
import org.mockito.InjectMocks;
import org.mockito.Mock;
import org.mockito.junit.jupiter.MockitoExtension;

@ExtendWith(MockitoExtension.class)
@DisplayName("UserDomainService 单元测试")
class UserDomainServiceImplTest {

    @Mock
    private UserRepository userRepository;

    @InjectMocks
    private UserDomainServiceImpl userDomainService;

    @Nested
    @DisplayName("用户注册")
    class UserRegister {

        @Test
        @DisplayName("正常注册 → 密码加密存储")
        void shouldRegisterUserWithEncryptedPassword() {
            // given
            String rawPassword = "12345678";
            given(userRepository.existsByUserAccount("testUser")).willReturn(false);
            given(userRepository.save(any(User.class))).willAnswer(inv -> {
                User u = inv.getArgument(0);
                u.setId(1L);
                return u;
            });

            // when
            User saved = userDomainService.register("testUser", rawPassword, "test");

            // then
            assertThat(saved.getId()).isEqualTo(1L);
            assertThat(saved.getUserPassword()).isNotEqualTo(rawPassword);
            assertThat(saved.getUserPassword()).startsWith("$2a$"); // BCrypt
            assertThat(saved.getUserRole()).isEqualTo(UserRoleEnum.USER.getValue());
            verify(userRepository).existsByUserAccount("testUser");
            verify(userRepository).save(any(User.class));
        }

        @Test
        @DisplayName("重复账号注册 → 抛异常")
        void shouldThrowWhenAccountExists() {
            given(userRepository.existsByUserAccount("existing")).willReturn(true);

            assertThatThrownBy(() ->
                userDomainService.register("existing", "12345678", "test"))
                .isInstanceOf(BusinessException.class)
                .hasMessageContaining("账号已存在");

            verify(userRepository, never()).save(any());
        }

        @Test
        @DisplayName("密码不足8位 → 抛异常")
        void shouldThrowWhenPasswordTooShort() {
            assertThatThrownBy(() ->
                userDomainService.register("user", "123", "test"))
                .isInstanceOf(BusinessException.class)
                .hasMessageContaining("密码长度");
        }
    }
}
```

#### 第三层：Repository 实现测试（Testcontainers + 真实数据库）

不使用 H2 内存库！H2 与 MySQL 在 SQL 语法、索引行为上差异太大，用 H2 测完换 MySQL 运行时仍有 BUG。Testcontainers 通过 Docker 拉起真实 MySQL/Redis，和线上环境一致。

**POM 依赖：**

```xml

<dependency>

    <artifactId>testcontainers</artifactId>
    <version>1.20.6</version>
    <scope>test</scope>
</dependency>
<dependency>
    <groupId>org.testcontainers</groupId>
    <artifactId>mysql</artifactId>
    <version>1.20.6</version>
    <scope>test</scope>
</dependency>
<dependency>
    <groupId>org.testcontainers</groupId>
    <artifactId>junit-jupiter</artifactId>
    <version>1.20.6</version>

<!-- 用于测试中方便构造数据 -->
```

**示例代码（共享容器，加速测试）：**

```java
package com.vio.aitukuviobe.infrastructure.repository;

import static org.assertj.core.api.Assertions.*;

import com.vio.aitukuviobe.domain.picture.entity.Picture;
import com.vio.aitukuviobe.domain.picture.valueobject.PictureReviewStatusEnum;
import org.junit.jupiter.api.*;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.boot.test.context.SpringBootTest;
import org.springframework.test.context.DynamicPropertyRegistry;
import org.springframework.test.context.DynamicPropertySource;
import org.testcontainers.containers.MySQLContainer;
import org.testcontainers.junit.jupiter.Container;
import org.testcontainers.junit.jupiter.Testcontainers;

@SpringBootTest
@Testcontainers
@DisplayName("PictureRepositoryImpl 集成测试")
class PictureRepositoryImplTest {

    // 使用 static 确保整个测试类共享一个容器（大幅提速）
    @Container
    static MySQLContainer<?> mysql = new MySQLContainer<>("mysql:8.0")
        .withDatabaseName("aituku_test")
        .withUsername("test")
        .withPassword("test");

    @DynamicPropertySource
    static void configureProperties(DynamicPropertyRegistry registry) {
        registry.add("spring.datasource.url", mysql::getJdbcUrl);
        registry.add("spring.datasource.username", mysql::getUsername);
        registry.add("spring.datasource.password", mysql::getPassword);
    }

    @Autowired
    private PictureRepositoryImpl pictureRepository;

    @Autowired
    private PictureMapper pictureMapper;

    @BeforeEach
    void setUp() {
        // 每个测试前清表，保证隔离
        pictureMapper.delete(new LambdaQueryWrapper<>());
    }

    @Test
    @DisplayName("按 spaceId + reviewStatus 分页查询图片")
    void shouldFindBySpaceIdAndReviewStatus() {
        // given
        Picture pic = Picture.builder()
            .spaceId(1L).userId(1L)
            .url("https://example.com/test.jpg")
            .reviewStatus(PictureReviewStatusEnum.PASS.getValue())
            .name("测试图片")
            .build();
        pictureRepository.save(pic);

        // when
        Page<Picture> page = new Page<>(1, 10);
        LambdaQueryWrapper<Picture> wrapper = new LambdaQueryWrapper<Picture>()
            .eq(Picture::getSpaceId, 1L)
            .eq(Picture::getReviewStatus, PictureReviewStatusEnum.PASS.getValue());
        Page<Picture> result = pictureRepository.page(page, wrapper);

        // then
        assertThat(result.getTotal()).isEqualTo(1);
        assertThat(result.getRecords().get(0).getName()).isEqualTo("测试图片");
    }
}
```

> **性能 Tips**：
> 1. 使用 `static MySQLContainer` 在整个测试类中共享容器，而非每方法重建。
> 2. 引入 [Testcontainers Lifecycle](https://java.testcontainers.org/test_framework_integration/manual_lifecycle_control/) 的 "reusable" 模式，在开发机保持容器常驻，一次启动多次复用。
> 3. 将 `@Container` MySQ、Redis 提取到抽象基类 `AbstractIntegrationTest`，所有集成测试继承之。

#### 第四层：Controller 层测试（@WebMvcTest）

不启动完整 Spring 容器，仅加载 Web 层，Mock 掉 Service 依赖。

```java
package com.vio.aitukuviobe.interfaces.controller;

import static org.mockito.BDDMockito.*;
import static org.springframework.test.web.servlet.request.MockMvcRequestBuilders.*;
import static org.springframework.test.web.servlet.result.MockMvcResultMatchers.*;
```

```javascript
const errorRate = new Rate('errors');

  stages: [
    { duration: '1m',  target: 50  },   // 1 分钟爬坡到 50 并发

  thresholds: {
    http_req_duration: ['p(95)<500', 'p(99)<2000'],  // 95% 请求 <500ms, 99% <2s
    errors: ['rate<0.01'],                            // 错误率 <1%
  },
};

const BASE_URL = __ENV.BASE_URL || 'http://localhost:8080';

export default function () {
  group('图片列表查询', () => {
    const page = Math.floor(Math.random() * 5) + 1;
    const res = http.get(`${BASE_URL}/api/picture/list?page=${page}&size=20`);
    const ok = check(res, {
      'status is 200': (r) => r.status === 200,
      'response time < 300ms': (r) => r.timings.duration < 300,
    });
    errorRate.add(!ok);
    pictureListDuration.add(res.timings.duration);

  });
}
```

```java
// 2. 测试方法命名: should{预期行为}_When{条件} 或 {
//    @DisplayName("中文 — 应有什么行为") 注解
// 3. 测试结构: Given-When-Then (AAA)

@Test
@DisplayName("正常注册 — 密码应 BCrypt 加密存储")
void shouldEncryptPasswordOnRegister() {
    // Given / Arrange
    given(userRepository.existsByUserAccount("test")).willReturn(false);
    given(userRepository.save(any())).willAnswer(inv -> inv.getArgument(0));

    // When / Act
    User result = userDomainService.register("test", "12345678", "test");

    // Then / Assert
    assertThat(result.getUserPassword())
        .startsWith("$2a$")
        .isNotEqualTo("12345678");
}

// 4. 使用 @Nested 分组相关测试
// 5. 禁止: 测试间共享可变状态（每个测试独立，不依赖执行顺序）

@SpringBootTest

        pictureService.processAsync(1L);

            .until(() -> pictureRepository.getById(1L).getProcessed());
```

# 十一、安全加固补充（企业级深度防护）

> 第 5 章已覆盖 OAuth2、限流、API 版本、审计扫描。本章从代码层面补充当前项目的实际安全隐患。

## 11.1 密码存储算法升级 ⭐ P0

- **现状**：`UserDomainServiceImpl` 中使用 `MD5("vio" + password)` 加密密码
- **问题**：MD5 已破解，彩虹表一秒逆推；固定盐值 `"vio"` 毫无防护力
- **方案**：替换为 BCrypt 或 Argon2
  ```java
  // 原代码（MD5 + 固定盐）
  String encrypted = SecureUtil.md5("vio" + password);
  // 改为
  String encrypted = BCrypt.hashpw(password, BCrypt.gensalt(12));
  ```
- **收益**：防彩虹表、防暴力破解、自适应 cost factor
- **迁移**：登录时先用 MD5 验证，通过后自动升级为 BCrypt，过渡期 30 天

## 11.2 配置密钥管理 ⭐ P0

- **现状**：`application-local.yml` 中明文存储 COS secretId/secretKey、阿里云 API Key（虽然在 gitignore 中，但任何有文件系统访问权限的人都能看到）
- **方案**：
  - 开发环境：使用 `.env` 文件 + `spring.config.import`（`.env` 加入 gitignore）
  - 生产环境：HashiCorp Vault / AWS Secrets Manager / 环境变量注入
  ```yaml
  # application.yml
  cos:
    client:
      secretId: ${COS_SECRET_ID}
      secretKey: ${COS_SECRET_KEY}
  aliYunAi:
    apiKey: ${ALIYUN_AI_API_KEY}
  ```
- **收益**：密钥不落地代码仓库，安全合规

## 11.3 CORS 配置收紧 ⭐ P1

- **现状**：`CorsConfig` 中 `allowedOriginPatterns("*")` + `allowCredentials(true)` 组合极其危险
- **问题**：任意域可携带 Cookie 发起跨域请求 → CSRF 攻击面无限大
- **方案**：
  ```java
  @Value("${cors.allowed-origins}")
  private List<String> allowedOrigins;
  
  registry.addMapping("/api/**")
      .allowedOriginPatterns(allowedOrigins.toArray(new String[0]))
      .allowCredentials(true);
  ```
- **收益**：仅受信域可跨域访问

## 11.4 HTTP 安全响应头配置 ⭐ P1

- **现状**：未设置任何安全响应头
- **方案**：添加 Spring Security 风格的安全头 Filter，或使用 Nginx 反向代理统一注入：
  ```
  add_header X-Content-Type-Options "nosniff";
  add_header X-Frame-Options "DENY";
  add_header X-XSS-Protection "1; mode=block";
  add_header Strict-Transport-Security "max-age=31536000; includeSubDomains";
  add_header Referrer-Policy "strict-origin-when-cross-origin";
  ```
- **收益**：防 MIME sniffing、防点击劫持、防 XSS 反射

## 11.5 文件上传安全加固 ⭐ P1

- **现状**：仅通过扩展名/Content-Type 判断文件类型，未做魔数校验
- **风险**：恶意文件改扩展名即可绕过（`.jsp → .jpg`）
- **方案**：
  - 服务端魔数（magic bytes）白名单校验（jpg: `FF D8 FF`，png: `89 50 4E 47` 等）
  - 上传后病毒扫描（ClamAV 集成）
  - 文件大小在网关层 + Controller 层双重校验
- **收益**：防恶意文件上传、防 RCE

## 11.6 图片 EXIF 隐私信息脱敏 ⭐ P1

- **现状**：上传图片未处理 EXIF 元数据
- **风险**：用户上传原图可能暴露 GPS 坐标、设备型号、拍摄时间等隐私
- **方案**：上传时调用 Metadata-Extractor 自动剥离 EXIF 数据，或使用 COS 万象CI 的 `imageMogr2/strip` 参数
- **收益**：保护用户隐私，合规（GDPR）

## 11.7 操作审计日志 ⭐ P2

- **现状**：仅 log 输出到控制台，无结构化审计记录
- **方案**：
  - 定义审计日志表 `audit_log`（who / what / when / IP / userAgent）
  - 通过 AOP `@AuditLog(action = "DELETE_PICTURE")` 注解自动记录
  - 保存到独立的审计数据库或 Elasticsearch
- **收益**：满足合规审计要求，异常操作可追溯

# 十三、数据一致性与并发控制补充

> 第 2 章已覆盖读写分离、分表、缓存优化。本章从并发安全角度补充。

## 13.1 MyBatis-Plus 乐观锁 ⭐ P1

- **现状**：所有实体未使用 `@Version` 注解
- **问题**：并发编辑同一图片/空间时，后者静默覆盖前者更新
- **方案**：
  ```java
  @Version
  private Integer version;
  ```
  MyBatis-Plus 自动在 UPDATE 时追加 `WHERE version = ?`，版本冲突时抛 `OptimisticLockException`
- **收益**：防并发覆盖，保证数据最终一致性

## 13.2 关键操作幂等性设计 ⭐ P1

- **现状**：图片上传、创建空间等写操作无幂等键保护
- **问题**：网络重试 + 用户双击 = 重复创建/重复上传
- **方案**：
  - 前端生成 `idempotencyKey`（UUID）随请求发送
  - 后端通过 Redis `SETNX idempotencyKey 1 EX 300` 去重
  - Controller AOP 拦截 `@Idempotent` 注解自动处理
- **收益**：防重复提交，用户体验提升

## 13.3 WebSocket 协同编辑冲突解决 ⭐ P2

- **现状**：`PictureEditHandler` 仅支持单人编辑（`pictureEditingUsers` 锁），第二人无法编辑
- **方案**：
  - 短期：编辑超时自动释放锁（30s 无操作自动退出）
  - 长期：引入 OT（Operational Transformation）或 CRDT 算法，支持真正的多人协同编辑
- **收益**：多人协作场景下不互斥

## 13.4 空间容量超限校验 ⭐ P1

- **现状**：`SpaceLevelEnum` 定义了 `maxSize` 和 `maxCount`，但上传时未校验当前使用量
- **风险**：用户上传超过空间容量限制的图片
- **方案**：
  ```java
  // 上传前校验
  Space space = spaceRepository.findById(spaceId);
  if (space.getTotalSize() + newFileSize > space.getMaxSize()) {
      throw new BusinessException("空间容量不足");
  }
  ```
- **收益**：空间容量限制实际生效

## 13.5 异步线程池显式配置 ⭐ P1

- **现状**：项目中 `pictureEditEventProducer` 使用 Disruptor 异步处理，但 Spring `@Async` 未见显式线程池配置
- **问题**：默认使用 `SimpleAsyncTaskExecutor`，无限创建线程，高并发下 OOM
- **方案**：
  ```java
  @Bean("pictureTaskExecutor")
  public ThreadPoolTaskExecutor taskExecutor() {
      ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
      executor.setCorePoolSize(4);
      executor.setMaxPoolSize(16);
      executor.setQueueCapacity(200);
      executor.setRejectedExecutionHandler(new CallerRunsPolicy());
      executor.setThreadNamePrefix("picture-async-");
      return executor;
  }
  ```
- **收益**：线程可控，防 OOM

# 十四、前端工程化补充

> 第 4 章已覆盖懒加载、PWA、国际化、移动端。本章从工程化角度补充。

## 14.1 前端 Build 产物优化 ⭐ P1

- **现状**：`vite.config.ts` 中无手动 chunk 分割策略
- **方案**：
  ```typescript
  build: {
    rollupOptions: {
      output: {
        manualChunks: {
          'vendor-vue': ['vue', 'vue-router', 'pinia'],
          'vendor-ui': ['ant-design-vue', '@ant-design/icons-vue'],
          'vendor-echarts': ['echarts', 'vue-echarts'],
        },
      },
    },
  },
  ```
- **收益**：减少单个 chunk 体积，首屏并行加载，合理利用浏览器缓存

## 14.2 前后端类型安全（OpenAPI Generator）⭐ P2

- **现状**：前端 `api/typings.d.ts` 手动维护类型，后端改了 VO 字段后前端可能遗漏
- **方案**：
  - 后端 Knife4j 已生成 OpenAPI JSON
  - 前端用 `@umijs/openapi`（已安装）自动生成 TypeScript 类型和请求函数
  ```bash
  npm run openapi  # 已有此 script!
  ```
- **收益**：前后端接口契约自动同步，编译期发现不匹配

## 14.3 前端请求容错增强 ⭐ P1

- **现状**：axios 拦截器仅处理 40100（未登录），无重试、无离线处理
- **方案**：
  - 网络错误自动重试 3 次（指数退避）
  - 请求合并（短时间内相同请求共享结果）
  - Token 过期自动刷新（Refresh Token 机制）
  ```typescript
  // axios-retry 或手写拦截器
  myAxios.interceptors.response.use(undefined, async (error) => {
    if (error.response?.status >= 500) {
      // 服务端错误：重试 1 次
      return myAxios(error.config);
    }
  });
  ```
- **收益**：弱网环境体验提升

## 14.4 前端状态管理规范 ⭐ P2

- **现状**：仅有 `useLoginUserStore.ts` 一个 Pinia store，缺乏使用规范
- **方案**：
  - 制定 Store 分类约定：`stores/user/`（用户状态）、`stores/space/`（空间状态）、`stores/picture/`（图片状态）
  - 禁止跨 Store 循环引用
  - 持久化策略：关键状态（token）使用 `pinia-plugin-persistedstate`
- **收益**：状态管理可维护、可预测

## 14.5 前端环境变量管理 ⭐ P1

- **现状**：未发现 `.env` 文件，API 地址硬编码在 `request.ts` 中
- **方案**：
  ```
  .env                # 默认值
  .env.development    # VITE_API_BASE_URL=http://localhost:8123
  .env.production     # VITE_API_BASE_URL=https://api.aituku.com
  ```
- **收益**：环境切换无需改代码

# 十五、日志与可观测性补充

> 第 6 章已覆盖 Prometheus + Grafana 监控、链路追踪、健康检查。本章补充日志层面的企业级实践。

## 15.1 结构化日志 ⭐ P1

- **现状**：`log.info()` 输出非结构化文本，难以检索和分析
- **方案**：
  - 使用 Logback 的 JSON encoder（`logstash-logback-encoder`）
  - 每条日志包含 traceId、userId、requestUri、duration 等标准字段
  ```xml
  <encoder class="net.logstash.logback.encoder.LogstashEncoder" />
  ```
- **收益**：ELK/Grafana Loki 可直接索引和分析日志

## 15.2 敏感数据日志脱敏 ⭐ P1

- **现状**：密码、API Key 等可能在日志中明文输出
- **方案**：
  - Logback `MessageConverter` 自动识别并脱敏（手机号 → `138****1234`）
  - 自定义 `@Sensitive` 注解 + Jackson 序列化脱敏
- **收益**：日志不泄露用户隐私

## 15.3 集中式日志收集 ⭐ P2

- **现状**：日志只输出到控制台/本地文件
- **方案**：Filebeat → Elasticsearch / Grafana Loki + Grafana 仪表板
- **收益**：多实例日志统一检索、可视化、告警

# 十八、WebSocket 生产级加固

## 18.1 连接数限制与心跳保活 ⭐ P1

- **现状**：`PictureEditHandler` 无连接数上限，无心跳检测
- **方案**：
  - `ConcurrentHashMap` 中 session 数超过阈值时拒绝新连接
  - 服务端定期发送 Ping 帧，超时未响应自动断开
- **收益**：防止连接泄露，资源可控

## 18.2 消息大小限制 ⭐ P1

- **现状**：WebSocket 未设消息大小上限
- **方案**：
  ```java
  // WebSocketConfig
  servletServerContainerFactory.setMaxTextMessageBufferSize(64 * 1024); // 64KB
  ```
- **收益**：防止大消息耗尽服务器内存

## 18.3 WebSocket 断线重连与消息有序 ⭐ P2

- **现状**：客户端 `PictureEditWebSocket.ts` 未见自动重连 + 消息确认机制
- **方案**：
  - 指数退避重连（1s → 2s → 4s → 最大 30s）
  - 消息序号 + ACK 机制保证有序和可靠投递
- **收益**：网络抖动不影响协作体验

# 二十、AI 框架工程化：手写 HTTP → LangChain4j ⭐ P1

> 第 3 章已覆盖 AI 功能层面的增强（自动标签、NSFW 检测、以图搜图等）。
> 本章从**代码工程化**角度，解决当前 AI 调用的底层实现问题。

---

## 20.1 当前 AI 实现分析

**现状：裸 HTTP 调用，零框架抽象**

```
AliYunAiApi.java (Hutool HttpRequest)
    ├── createOutPaintingTask()  → POST dashscope.aliyuncs.com/...
    ├── getOutPaintingTask()     → GET  dashscope.aliyuncs.com/...
    └── 手动 JSON 序列化/反序列化 + 手动错误处理
```

- 所有请求/响应 DTO 手写（`CreateOutPaintingTaskRequest`、`GetOutPaintingTaskResponse` 等）
- 仅对接阿里云 DashScope 的**图像扩图** API（同步创建任务 + 异步轮询结果）
- 无重试、无 fallback、无模型切换能力
- **如要新增 AI 功能（自动标签、NSFW 检测等），每加一个都要手写全套 HTTP 调用**

**结论：项目既没有用 Spring AI，也没有用 LangChain4j，而是原始 HTTP 调用。**

---

## 20.2 为什么选择 LangChain4j（而非 Spring AI）

| 维度 | LangChain4j | Spring AI | 本项目适用性 |
|------|-------------|-----------|-------------|
| **AI Services 注解式编程** | ✅ `@AiService` 接口自动代理 | ❌ 模板式编程 | 🏆 LangChain4j 胜出 |
| **Structured Outputs** | ✅ 自动反序列化到 POJO | ✅ BeanOutputConverter | 平手 |
| **Tools / Function Calling** | ✅ `@Tool` 注解 | ✅ `@Tool` 注解 | 平手 |
| **Chat Memory** | ✅ 多种实现 + 持久化 | ✅ 多种实现 | 平手 |
| **RAG 支持** | ✅ 成熟链路 | ✅ ETL + VectorStore | 平手 |
| **Spring Boot 集成** | ✅ Starter 自动配置 | ✅ 原生 AutoConfiguration | 平手 |
| **阿里云 DashScope** | ✅ `langchain4j-dashscope` | ✅ `spring-ai-alibaba` | 平手 |
| **多模态（图片模型）** | ✅ ImageModel 抽象 | ✅ ImageModel 抽象 | 平手 |
| **代码侵入性** | 接口式，低侵入 | 模板式，中侵入 | 🏆 LangChain4j 胜出 |

**核心优势——@AiService 注解式编程示例：**

```java
// 定义 AI 服务接口（放在 domain 层作为特殊的 Domain Service）
@AiService
public interface PictureAiService {

    // 自动标签提取 —— 框架自动处理 HTTP 调用、JSON 解析、重试
    @SystemMessage("你是图片标签提取专家。从图片描述中提取 3-5 个中文标签。")
    @UserMessage("图片名称: {{name}}, 分类: {{category}}, 描述: {{description}}")
    Result<List<String>> extractTags(
        @V("name") String name,
        @V("category") String category,
        @V("description") String description
    );

    // NSFW 内容审核
    @SystemMessage("你是图片内容安全审核专家。判断图片描述是否包含违规内容。")
    @UserMessage("请审核: {{description}}")
    Result<ModerationResult> moderateContent(
        @V("description") String description
    );

    // 图片搜索意图理解
    @SystemMessage("理解用户的图片搜索意图，提取关键词和分类。")
    @UserMessage("{{query}}")
    Result<SearchIntent> parseSearchIntent(
        @V("query") String query
    );
}
```

对比当前实现，需要手写的内容：
- ❌ HTTP 请求构造 → ✅ 框架自动
- ❌ JSON 序列化/反序列化 → ✅ 框架自动（Structured Outputs）
- ❌ 错误处理和重试 → ✅ 框架内置 RetryPolicy
- ❌ API Key 管理 → ✅ 框架统一的 ApiKey 注入
- ❌ 模型切换 → ✅ 改一行配置

### 阶段四：迁移现有扩图功能

**迁移前（当前代码）：**

```java
// PictureDomainServiceImpl.java:345-357
@Override
public CreateOutPaintingTaskResponse createPictureOutPaintingTask(...) {
    // 1. 手动校验权限
    Picture picture = pictureRepository.getById(pictureId);
    checkPictureAuth(loginUser, picture);
    // 2. 手动构造请求 DTO
    CreateOutPaintingTaskRequest taskRequest = new CreateOutPaintingTaskRequest();
    taskRequest.setInput(new CreateOutPaintingTaskRequest.Input());
    taskRequest.getInput().setImageUrl(picture.getUrl());
    taskRequest.setParameters(createPictureOutPaintingTaskRequest.getParameters());
    // 3. 裸 HTTP 调用
    return aliYunAiApi.createOutPaintingTask(taskRequest);
}
```

**迁移后：**

```java
// PictureDomainServiceImpl.java —— 注入 @AiService 接口即可
@Resource
private PictureAiService pictureAiService;

@Override
public OutPaintingResult createPictureOutPaintingTask(...) {
    Picture picture = pictureRepository.getById(pictureId);
    checkPictureAuth(loginUser, picture);

    // 一行调用，框架处理 HTTP + JSON + 重试 + 错误
    var params = createPictureOutPaintingTaskRequest.getParameters();
    Result<OutPaintingResult> result = pictureAiService.outPainting(
        picture.getUrl(),
        params.getXScale(),
        params.getYScale()
    );
    return result.content();
}
```

## 20.4 模型可切换架构设计

LangChain4j 的核心价值之一是**模型无关**。通过抽象接口，切换模型只需改配置：

```yaml
# 场景 1：使用阿里云 DashScope（默认）
langchain4j:
  dashscope:
    chat-model:
      model-name: qwen-turbo      # 便宜、快 → 标签提取
    image-model:
      model-name: image-out-painting

# 场景 2：切换到 OpenAI 兼容接口（如 OneAPI 中转）
langchain4j:
  open-ai:
    chat-model:
      base-url: http://your-proxy.com/v1
      api-key: ${OPENAI_API_KEY}
      model-name: gpt-4o-mini
```

在代码中不需要任何修改 —— `@AiService` 接口背后的模型由配置决定。

## 20.5 架构对比：迁移前 vs 迁移后

```
迁移前（当前）：
┌─────────────────────────────────────────────────┐
│  PictureDomainServiceImpl                       │
│    ├── aliYunAiApi.createOutPaintingTask()      │  ← 裸 HTTP
│    └── 手动 JSON 序列化 / 反序列化               │
│         ↓                                       │
│    AliYunAiApi (Hutool HttpRequest)             │  ← 200+ 行 HTTP 胶水代码
│         ↓                                       │
│    阿里云 DashScope API                         │
└─────────────────────────────────────────────────┘
问题：新增任何 AI 功能 = 手写全套 HTTP + DTO + 错误处理

迁移后：
┌─────────────────────────────────────────────────┐
│  PictureDomainServiceImpl                       │
│    ├── pictureAiService.outPainting(...)        │  ← 一行调用
│    ├── pictureAiService.extractTags(...)        │  ← 一行调用
│    └── pictureAiService.moderateContent(...)    │  ← 一行调用
│         ↓                                       │
│    PictureAiService (@AiService 接口)           │  ← 纯声明
│         ↓                                       │
│    LangChain4j 框架层                            │  ← 自动代理、重试、结构化输出
│         ↓                                       │
│    阿里云 / OpenAI / 本地模型（可切换）           │
└─────────────────────────────────────────────────┘
收益：新增 AI 功能 = 在 @AiService 接口加一个方法
```

---

> 📌 **小结**：
> - LangChain4j 不是"又多一个框架"，而是**消除胶水代码**：当前 `AliYunAiApi` 的 200+ 行 HTTP 胶水代码全部消失
> - **与 DDD 架构天然兼容**：`@AiService` 接口放入 `domain/*/service/`，和 `UserDomainService` 同级
> - **为第 3 章的 AI 功能扩展铺路**：自动标签、NSFW 审核、以图搜图，每加一个只需在接口加一个方法
