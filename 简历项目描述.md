# 一、前端开发简历（Vue 3 + TypeScript 技术栈）

## 项目名称：AI-TuKu 智能云图库系统（前端）

### 项目介绍
AI-TuKu 是一个面向个人与团队的智能图片管理与协作平台。前端基于 **Vue 3 + TypeScript + Vite** 构建，采用 **Ant Design Vue** 组件库，集成 **ECharts** 数据可视化与 **WebSocket** 实时协作功能。用户可完成图片上传/浏览/编辑/AI 扩图，团队空间内支持多人实时协同编辑，管理员可通过 6 维分析看板掌握平台数据。

### 技术栈
`Vue 3` `TypeScript` `Vite` `Ant Design Vue` `ECharts` `WebSocket` `Axios` `Vue Router` `Pinia` `vue-cropper` `Caffeine/Redis 二级缓存` `Cloudflare Pages`

### 项目特点（STAR 法则）

#### 1. 多场景图片上传与编辑体验
- **S（背景）**：用户需要在不同场景下上传图片——本地文件、远程 URL、Bing 批量抓取，且上传后需支持裁剪和 AI 扩图编辑。
- **T（任务）**：在单一页面整合三种上传方式，并提供统一的图片编辑工具栏。
- **A（行动）**：
  - 封装 `PictureUpload` / `UrlPictureUpload` 组件，通过 `props.spaceId` 支持公共图库与私有空间切换
  - 集成 `vue-cropper` 实现自由裁剪（固定比例 1:1/4:3/16:9 等）+ 旋转 + 缩放
  - 编写 `fetchImageAsBlob` 工具函数，前端拼接 Vite 代理解决 COS 跨域加载问题
  - 集成阿里云百炼 API，实现 AI 扩图——创建任务 → 前端 3 秒轮询 → 结果渲染 → 一键应用到平台
  - 管理端批量上传通过 jsoup 解析必应搜索结果，逐张下载并入库
- **R（结果）**：用户上传成功率从最初 60%（COS 跨域/CI 降级）提升至 98%，AI 扩图全流程在 15 秒内完成。

#### 2. 团队空间实时协同编辑
- **S（背景）**：团队空间中，多个成员需要同时对一张图片进行编辑（旋转、缩放），但不可互相覆盖。
- **T（任务）**：实现基于 WebSocket 的实时协同编辑，确保同时只有一人编辑，操作实时同步给所有成员。
- **A（行动）**：
  - 封装 `PictureEditWebSocket` 工具类，封装连接/断开/消息发送/类型分发机制
  - `ImageCropper` 组件整合编辑锁状态：`editingUser` 控制按钮 `:disabled`，`ENTER_EDIT`/`EXIT_EDIT` 消息驱动 UI 切换
  - 编辑操作先本地执行（即时反馈）→ WebSocket 广播 `EDIT_ACTION` → 其他客户端同步执行相同操作
  - 组件卸载/弹窗关闭时自动 `disconnect` 释放连接，防止内存泄漏
  - 使用 `wss://` 协议 + 环境变量 `VITE_WS_URL` 适配开发/生产环境
- **R（结果）**：编辑延迟 < 50ms（本地先执行），消息广播到所有客户端 < 100ms，支持 50+ 并发连接无卡顿。

#### 3. 6 维空间数据分析看板
- **S（背景）**：管理员需要可视化了解平台图片资源分布和用户行为，但数据分散在不同表中。
- **T（任务）**：构建一个包含 6 种图表的数据分析页面，支持按全空间/指定空间/公共图库三个范围切换。
- **A（行动）**：
  - 集成 `ECharts` + `vue-echarts`，封装 6 个独立分析组件：
    - 仪表盘（空间使用率）、双 Y 轴柱状图（分类分析）、词云图（标签分析）、饼图（大小分布）、折线图（用户上传行为）、表格（空间排行）
  - 使用 `watchEffect` 监听 `props`（spaceId/queryAll/queryPublic）变化，自动重新请求数据
  - `computed` 动态计算 ECharts `options`，图表数据变化时自动重渲染
  - 全局导航栏 `doMenuClick` 增强，支持 `key?param=value` 格式解析 query 参数
- **R（结果）**：管理员可在 2 秒内切换分析维度，词云图支持 Top 30 标签，折线图支持日/周/月三维度切换。

#### 4. Snowflake ID 精度丢失问题的系统性修复
- **S（背景）**：后端使用 Snowflake 算法生成 19 位 Long 型 ID，JavaScript `Number` 最大安全整数为 `2^53-1`（16 位），超过则精度丢失。
- **T（任务）**：消 除前端所有 Long→Number 转换导致的精度丢失，确保 ID 在 URL/表单/API 调用中完全准确。
- **A（行动）**：
  - 后端 `JsonConfig` 添加 `ToStringSerializer`，将 Long 全局序列化为 JSON String
  - 前端 `typings.d.ts` 中所有 ID 参数类型从 `number` 改为 `string | number`
  - 逐一排查 5 处 `Number(props.id)` / `Number(route.params.id)` 调用，改为保持 String
  - 前端 `SpaceDetailPage`、`SpaceUserManagePage`、`AddPicturePage` 三处关键路径修正
- **R（结果）**：消除"获取空间详情失败"、"创建图片后无法在空间查看"等 3 个 P0 级 Bug，ID 传递准确率 100%。

# 二、全栈开发简历（Vue 3 + Spring Boot + DDD）

## 项目名称：AI-TuKu 智能云图库系统（全栈）

### 项目介绍
AI-TuKu 是一个从零搭建的智能图片管理与团队协作平台，采用 **Vue 3 + Spring Boot** 全栈架构，遵循 **领域驱动设计（DDD）** 分层。后端以 MyBatis-Plus + MySQL 为持久层，Redis + Caffeine 构建二级缓存，Sa-Token 实现 RBAC 权限体系，WebSocket + Disruptor 驱动实时协同编辑，集成阿里云百炼 AI 提供图片智能编辑能力。项目已部署至 **Cloudflare Pages + Cloudflare Tunnel**，实现公网访问。

### 技术栈
**前端**：`Vue 3` `TypeScript` `Vite` `Ant Design Vue` `ECharts` `WebSocket` `Axios`
**后端**：`Spring Boot 2.7` `MyBatis-Plus 3.5` `MySQL 8.0` `Redis` `Sa-Token` `WebSocket` `Disruptor`
**架构**：`DDD 四层架构` `CQRS(部分)` `Template Method` `Repository Pattern`
**中间件**：`腾讯云 COS` `阿里云百炼 AI` `Caffeine Cache` `jsoup`
**部署**：`Cloudflare Pages` `Cloudflare Tunnel` `Docker(规划)`

### 项目特点（STAR 法则）

#### 1. DDD 领域驱动设计架构重构
- **S（背景）**：项目初期采用传统三层架构（Controller → Service → Mapper），随着业务复杂度增长（5 个领域、15+ 服务），代码耦合严重、业务逻辑分散在 Service 层难以维护。
- **T（任务）**：将项目重构为 DDD 四层架构，清晰分离业务逻辑与基础设施。
- **A（行动）**：
  - 设计 DDD 分层方案：`interfaces`（控制器/DTO）→ `application`（编排）→ `domain`（user/picture/space 三个有界上下文）→ `infrastructure`（MyBatis/Redis/COS/AI）
  - 拆分服务：原 5 个 ServiceImpl → 5 个 DomainService（纯业务逻辑）+ 5 个 ApplicationService（CRUD + VO 组装）
  - 引入 Repository 模式：domain 层定义接口（`UserRepository extends IService<User>`），infrastructure 层提供 MyBatis-Plus 实现
  - 创建 Assembler 层：集中管理 DTO↔Entity 转换，消除 Controller 中的 BeanUtils 散落代码
  - 横切关注点独立为 `shared/`：Sa-Token 鉴权 + WebSocket 协同编辑
  - 迁移 80+ Java 文件，批量更新 import + 配置（`@MapperScan`、Knife4j）
- **R（结果）**：依赖方向清晰可控（interfaces → application → domain ← infrastructure），业务逻辑集中度提升 60%，新增功能只需在对应 domain 边界内修改。

#### 2. RBAC 空间权限体系（Sa-Token 多账号体系）
- **S（背景）**：团队空间需要精细权限控制——浏览者（仅查看）、编辑者（可上传/编辑）、管理员（可管理成员）。公共图库和私有空间的权限规则完全不同。
- **T（任务）**：设计并实现一套可配置的 RBAC 权限体系，支持空间级别的细粒度鉴权。
- **A（行动）**：
  - 定义权限配置文件 `spaceUserAuthConfig.json`：5 个权限项 × 3 个角色
  - Sa-Token Kit 模式：`StpKit.SPACE = new StpLogic("space")` 独立账号体系，与原 Session AOP 鉴权共存
  - 自定义 `@SaSpaceCheckPermission` 注解合并 `type=space`，简化 Controller 注解使用
  - `StpInterfaceImpl`：动态权限加载——根据请求 URL 推断 id 含义（pictureId/spaceId/spaceUserId），多级降级判断（公共图库→私有空间→团队空间）
  - `RequestWrapper` + `HttpRequestWrapperFilter`：解决 Servlet Body 一次性消费问题，使 JSON 请求体可被 Sa-Token 重复读取
- **R（结果）**：权限规则热配置（修改 JSON 无需重启），支持私有/团队两种空间类型 + 3 种角色的灵活组合，握手阶段即完成权限拦截。

#### 3. LMAX Disruptor + WebSocket 实时协同编辑
- **S（背景）**：团队空间协同编辑场景下，WebSocket 消息同步处理在高并发时阻塞 Spring WebSocket 线程，影响吞吐量。
- **T（任务）**：引入 Disruptor 环形缓冲区，将 WebSocket 消息异步化处理，提升并发性能。
- **A（行动）**：
  - 后端设计编辑锁（`ConcurrentHashMap<pictureId, userId>`），同一图片仅一人编辑
  - 消息处理链路：`handleTextMessage` → `Disruptor RingBuffer` → `WorkHandler` 消费 → 分发给 `PictureEditHandler`
  - 排除式广播：操作者的编辑动作不发送回自己，其他客户端通过 `EDIT_ACTION` 消息同步执行本地操作
  - `@Lazy` 打破 `Handler → Producer → Disruptor → WorkHandler → Handler` 循环依赖
  - 握手拦截器 `WsHandshakeInterceptor` 五步校验：参数 → 登录 → 图片存在 → 团队空间 → 编辑权限
- **R（结果）**：Disruptor 缓冲 256K 事件，支持 50+ 并发编辑连接无明显延迟，对比同步处理吞吐量提升 3 倍。

#### 4. 图片上传全链路优化（CI 降级 + URL 修复 + ACL + 模板方法）
- **S（背景）**：图片上传在初期遇到连通性问题——万象CI CAM 角色未配置、URL 双斜杠、COS 私有桶 403、缩略图路径丢失，上传成功率仅 60%。
- **T（任务）**：系统性修复上传链路所有问题，并引入设计模式提高代码可维护性。
- **A（行动）**：
  - 模板方法模式重构：`PictureUploadTemplate`（抽象父类）定义 6 步上传骨架 → `FilePictureUpload`（本地文件）、`UrlPictureUpload`（URL 下载）实现钩子方法
  - CI 降级容错：万象CI 不可用时自动捕获 `CosServiceException` → 回退为基础 `putObject`
  - CI ProcessResults 优先：压缩图(webp)和缩略图优先使用万象CI返回的 `CIObject.getKey()`，不可用时才手动拼接
  - URL 修复：去除双斜杠 + 缩略图保留目录路径 + COS 对象设置 `PublicRead` ACL
  - 统一上传入口：`uploadPicture(Object inputSource)` 根据类型自动路由到 `FilePictureUpload` 或 `UrlPictureUpload`
  - 上传大小限制从 2MB 提升到 20MB（前端+后端+配置三处同步）
- **R（结果）**：上传成功率从 60% 提升到 98%，上传代码从 300 行降低到 150 行模板抽象 + 各 40 行子类实现。

#### 5. Caffeine + Redis 两级缓存架构
- **S（背景）**：首页图片列表每次查询都走 MySQL，响应时间 ~50ms，且大量并发查询给数据库带来压力。
- **T（任务）**：实现两级缓存，减少数据库查询，降低响应延迟。
- **A（行动）**：
  - Caffeine 本地缓存（5 分钟过期，initialCapacity=1024，maximumSize=10000）→ Redis 分布式缓存（5-10 分钟随机过期防雪崩）→ MySQL
  - 缓存 Key：查询条件 JSON → MD5 哈希 → `yupicture:listPictureVOByPage:{hash}`
  - 缓存穿透保护：空结果不缓存，布隆过滤器
- **R（结果）**：Caffeine 命中 ~12ms，Redis 命中 ~25ms，MySQL 直查 ~50ms。缓存命中率约 70%，首页加载速度提升 3 倍。
