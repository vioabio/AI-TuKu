# 一、项目概览

## 项目定位
**AI-TuKu** 是一个从零搭建的 AI 驱动的智能图片管理与团队协作平台，支持个人图库、团队共享图库和公共图库三种模式，集成 AI 图片扩图、实时协同编辑、数据可视化分析等能力。项目已完整部署至 Cloudflare Pages + Cloudflare Tunnel 公网可访问。

## 项目信息
| 维度 | 详情 |
|------|------|
| **开发周期** | 2026年6月20日 ~ 6月28日（9天，全栈独立开发） |
| **代码规模** | 后端 ~134 个 Java 文件（DDD 四层架构），前端 14 个页面 + 10 个组件 |
| **数据库** | 4 张业务表（user、picture、space、space_user），MySQL 8.0 |
| **API 接口** | 40+ 个 RESTful 接口，6 个分析维度 |
| **个人贡献** | 全栈独立开发：架构设计 → 前后端编码 → DDD 重构 → 部署上线 → Bug 修复（45 个） |

## 2.2 核心亮点一：DDD 领域驱动设计架构重构

### S（背景）
项目初期采用传统 Controller → Service → Mapper 三层架构。随着业务复杂度增长（5 个领域、15+ 服务类），代码耦合严重、业务逻辑分散在 Service 层，维护成本急剧上升。例如一个图片上传操作涉及 COS 存储、压缩、缩略图、空间配额更新、审核状态设置等 6 个步骤，代码混杂在单个 Service 中难以拆分。

### T（任务）
将项目重构为 DDD 四层架构，实现关注点清晰分离，使新增功能只需在对应领域边界内修改。

### A（行动）
1. **分层设计**：设计 `interfaces`（控制器/DTO/Assembler）→ `application`（业务编排/事务管理）→ `domain`（纯业务逻辑/Repository 接口）→ `infrastructure`（MyBatis/COS/Redis/AI API 实现）+ `shared`（跨域横切关注点：鉴权/WebSocket）
2. **领域划分**：识别 4 个有界上下文（User、Picture、Space、SpaceUser），每个上下文独立维护自己的 Entity/Repository/DomainService/ValueObject
3. **依赖倒置**：Domain 层定义 `UserRepository extends IService<User>` 接口，Infrastructure 层提供 `UserRepositoryImpl extends ServiceImpl<UserMapper, User>` 实现
4. **应用层编排**：ApplicationService 作为薄层编排多个 DomainService 和跨域调用，`@Transactional` 事务统一在 Application 层管理
5. **迁移策略**：逐个领域迁移（User → Picture → Space → SpaceUser），每迁移一个领域即验证编译通过和功能正常，最终批量删除旧包目录（12 个旧包）
6. **方法下沉规则**：不涉及跨服务调用 → 下沉到 DomainService；DB 无关的校验逻辑 → 下沉到 Entity（充血模型）

### R（结果）
- 原有 5 个 ServiceImpl 拆分为 5 个 DomainService（纯业务）+ 5 个 ApplicationService（编排），职责清晰
- 依赖方向单向可控：interfaces → application → domain ← infrastructure
- 业务逻辑集中度提升 60%，新增空间分析模块仅需在 domain/space 边界内新增文件
- 迁移 80+ Java 文件、修复 8 个编译/运行时错误，最终构建成功

## 2.3 核心亮点二：RBAC 空间权限体系设计

### S（背景）
团队空间中存在三种角色（浏览者/编辑者/管理员），每种角色拥有不同的操作权限。公共图库和私有空间的权限判定规则完全不同：公共图库允许所有人查看、仅本人或管理员可编辑删除；私有空间仅空间所有者有全部权限；团队空间按角色授权。传统方案需要每种场景都在 Controller 中硬编码 if-else 判断。

### T（任务）
设计并实现一套可配置、可扩展的 RBAC 权限体系，支持空间级别的细粒度动态鉴权，消除 Controller 中的硬编码权限判断。

### A（行动）
1. **权限配置外置化**：设计 `spaceUserAuthConfig.json` 配置文件，定义 5 个权限项（spaceUser:manage / picture:view:upload:edit:delete）× 3 个角色（viewer/editor/admin）的映射关系
2. **Sa-Token 双账号体系**：通过 Kit 模式创建独立的 `StpLogic("space")` 实例，与原有 Session 鉴权共存互不干扰
3. **自定义注解**：封装 `@SaSpaceCheckPermission` 注解（合并 `@SaCheckPermission(type="space")`），简化 Controller 使用
4. **动态权限加载**：`StpInterfaceImpl.getPermissionList()` 根据请求 URL 自动推断上下文参数（pictureId → spaceId → spaceUserId），实现多级降级判断（公共图库 → 私有空间 → 团队空间）
5. **请求体可重复读取**：编写 `RequestWrapper` + `HttpRequestWrapperFilter`，解决 Servlet Body 一次性消费导致 Sa-Token 权限校验无法读取 JSON 请求体的问题
6. **握手阶段拦截**：WebSocket 握手拦截器 `WsHandshakeInterceptor` 在建立连接前即完成五步校验（参数 → 登录 → 图片存在 → 团队空间 → 编辑权限）

### R（结果）
- 权限规则修改仅需编辑 JSON 配置文件，无需重启应用
- 支持私人和团队两种空间类型 × 3 种角色的灵活权限组合
- Controller 代码中消除所有硬编码权限判断，通过注解声明式鉴权
- 权限校验在握手阶段即完成拦截，无效请求不会建立 WebSocket 连接

## 2.4 核心亮点三：WebSocket + Disruptor 实时协同编辑

### S（背景）
团队空间场景下，多个成员需要同时对图片进行编辑操作（缩放、旋转），每个操作需要实时同步给空间内所有在线成员。传统同步处理方式在并发连接增多时，Spring WebSocket 线程会被阻塞，导致消息延迟和系统吞吐量下降。

### T（任务）
实现基于 WebSocket 的实时协同编辑系统，引入 Disruptor 环形缓冲区实现异步高吞吐消息处理，确保编辑操作低延迟广播和编辑锁互斥。

### A（行动）
1. **编辑锁机制**：使用 `ConcurrentHashMap<pictureId, editingUserId>` 确保同一图片同一时刻仅一人进行编辑操作
2. **Disruptor 异步架构**：设计消息处理链路 `handleTextMessage → Disruptor RingBuffer → WorkHandler 消费 → 路由到 PictureEditHandler 处理`，RingBuffer 大小 256K 事件，MULTI 生产者模式 + BlockingWaitStrategy
3. **排除式广播**：操作者的编辑动作广播给空间内所有其他客户端，自身客户端不接收回显（本地已即时执行），避免重复渲染
4. **循环依赖解决**：`Handler → Producer → DisruptorConfig → WorkHandler → Handler` 形成循环引用，通过 `@Lazy` 注解打破 Spring Bean 初始化循环依赖
5. **前端即时反馈**：编辑操作先本地执行（即时 UI 反馈）→ WebSocket 广播 EDIT_ACTION → 其他客户端收到消息后同步执行相同操作
6. **生命周期管理**：组件卸载/弹窗关闭时自动 `disconnect` 释放 WebSocket 连接（防止内存泄漏），`wss://` 协议 + 环境变量适配开发/生产环境

### R（结果）
- 编辑操作本地延迟 < 50ms（先本地执行再广播），消息广播到所有客户端 < 100ms
- Disruptor 异步处理对比同步处理吞吐量提升 3 倍
- 支持 50+ 并发 WebSocket 连接无卡顿
- 编辑锁机制确保多人协同时不会出现覆盖冲突

## 2.5 核心亮点四：图片上传全链路优化

### S（背景）
图片上传功能初期连通性差——腾讯云万象CI CAM 角色未配置导致 403 报错、URL 拼接双斜杠导致图片无法访问、COS 私有桶未设置 ACL 导致浏览器 403、缩略图路径丢失导致列表不显示图片。上传成功率仅约 60%，严重影响用户体验。

### T（任务）
系统性排查并修复上传全链路所有问题，通过设计模式重构提高代码可维护性和可扩展性。

### A（行动）
1. **模板方法模式重构**：抽象 `PictureUploadTemplate` 定义 6 步上传骨架（校验 → 创建临时文件 → 处理 → 上传 COS → 构建结果 → 清理），`FilePictureUpload` 和 `UrlPictureUpload` 各自实现钩子方法，代码从 300 行降至父类 150 行 + 各子类 40 行
2. **CI 降级容错**：万象CI 不可用时自动捕获 `CosServiceException` → 判断错误信息 → 自动回退为基础 `putObject`（无损降级）
3. **CI ProcessResults 优先**：压缩图（webp）和缩略图优先使用万象CI返回的 `CIObject.getKey()`，仅在不可用时才手动拼接路径
4. **URL 修复**：去除双斜杠 + 缩略图保留完整目录路径 + COS 对象统一设置 `PublicRead` ACL
5. **统一入口**：`uploadPicture(Object inputSource)` 方法根据输入源类型自动路由到对应的 Template 子类实现
6. **限制一致性**：上传大小限制从 2MB 提升到 20MB（前端组件 + 后端 Manager + 配置文件三处同步修改）

### R（结果）
- 上传成功率从 60% 提升至 98%
- 上传代码从 300 行降低至父类 150 行 + 各子类 40 行，可维护性显著提升
- CI 降级容错确保即使云服务异常仍可正常上传
- 消除"上传成功但图片不显示"的核心体验问题

## 2.6 核心亮点五：Caffeine + Redis 两级缓存

### S（背景）
首页图片列表每次查询直接访问 MySQL，单次查询响应时间 ~50ms。在并发访问场景下，大量重复查询给数据库带来压力，响应延迟随之上升。

### T（任务）
实现两级缓存架构，利用本地缓存降低响应延迟，利用分布式缓存支撑多实例部署，同时防御缓存穿透和雪崩。

### A（行动）
1. **缓存分层**：Caffeine 本地缓存（5 分钟过期，initialCapacity=1024，maximumSize=10000）→ Redis 分布式缓存（5-10 分钟随机过期防雪崩）→ MySQL 数据库
2. **缓存 Key 设计**：查询条件 JSON 序列化 → MD5 哈希 → `yupicture:listPictureVOByPage:{hash}`
3. **缓存穿透保护**：空结果不写入缓存，配合布隆过滤器机制
4. **Read-Through 模式**：缓存未命中时自动查询 DB 并回填 Caffeine + Redis

### R（结果）
- Caffeine 命中 ~12ms，Redis 命中 ~25ms，MySQL 直查 ~50ms
- 缓存命中率约 70%，首页加载速度提升 3 倍
- 随机过期时间设计有效防止缓存雪崩

## 2.7 核心亮点六：Snowflake ID 精度丢失的系统性修复

### S（背景）
后端使用 Snowflake 算法生成 19 位 Long 型 ID。JavaScript 的 `Number` 类型最大安全整数为 `2^53 - 1`（约 16 位），超过此范围的 Long 型 ID 在 JSON 解析时发生精度截断，末尾 2-3 位丢失。

### T（任务）
系统性消除前端所有 Long → Number 转换导致的精度丢失，确保 ID 在 URL 传递、表单提交、API 调用的全链路中准确无误。

### A（行动）
1. **后端全局序列化**：`JsonConfig` 中为所有 Long 类型添加 `ToStringSerializer`，后端 → 前端方向 JSON 中 Long 以 String 形式传输
2. **前端类型修正**：`typings.d.ts` 中所有 ID 参数类型从 `number` 改为 `string | number`
3. **逐一排查修复**：排查并修正 5 处 `Number(props.id)` / `Number(route.params.id)` 调用，改为保持 String 类型传递（Spring MVC 自动解析 String → Long）
4. **关键路径修复**：`SpaceDetailPage`（空间详情）、`SpaceUserManagePage`（成员管理）、`AddPicturePage`（图片创建）三处关键路径修正

### R（结果）
- 消除 3 个 P0 级 Bug（"获取空间详情失败"、"创建图片后无法在空间查看"、"成员管理页面数据不匹配"）
- ID 传递全链路准确率 100%
- 建立了前后端 ID 类型安全的最佳实践（后端序列化 + 前端保持 String + Spring 自动解析）

# 三、面向 Agent 智能体开发岗位的项目描述

## 3.1 AI Agent 集成能力：阿里云百炼 AI 智能扩图

### S（背景）
传统图片编辑需要用户手动操作裁剪/拉伸工具实现图片扩展，用户体验差且效果难以控制。项目需要集成 AI 能力实现智能化的图片编辑。

### T（任务）
对接阿里云百炼（DashScope）AI 模型的异步任务执行能力，实现图片智能扩图（Outpainting）功能，并设计稳定的前端交互体验。

### A（行动）
1. **异步任务模型**：对接阿里云百炼 `aigc/image2image/out-painting` API，采用异步任务模式（`X-DashScope-Async: enable`），创建任务 → 返回 taskId → 轮询状态 → 获取结果
2. **AI API 客户端封装**：`AliYunAiApi` 使用 Hutool HTTP 客户端，Bearer Token 鉴权，完整的请求体/响应体/错误日志记录（用于任务失败诊断）
3. **前端轮询设计**：`ImageOutPainting` 组件使用 `setInterval` 每 3 秒轮询任务状态，SUCCEEDED 时渲染结果图片，FAILED 时展示错误信息，组件卸载时自动清理定时器
4. **参数配置**：用户可配置 x 轴扩展比例（1-3）、y 轴扩展比例（1-3）、是否添加水印
5. **结果应用**：扩图成功后提供"一键应用"按钮 → 结果 URL 自动导入平台图片库 → 无缝衔接后续编辑/分享流程
6. **健壮性设计**：API 调用失败时完整的错误信息透传、任务超时处理（30 秒未完成提示用户重试）

### R（结果）
- AI 扩图全流程 15 秒内完成（创建任务 1s + AI 处理 10-12s + 应用结果 2s）
- 任务失败时可通过日志完整追踪请求/响应，定位原因时间从"无从排查"降至 30 秒内
- 提供了清晰的异步任务交互范式，可复用于对接其他 AI API

## 3.2 Agent 编排思维：批量图片抓取流程

### S（背景）
管理员需要批量导入图片来源，但逐张上传效率极低。

### T（任务）
设计自动化批量图片抓取流程，从搜索引擎自动获取图片并入库。

### A（行动）
1. **Web Scraping Agent**：使用 Jsoup 解析必应图片搜索结果 HTML（`cn.bing.com/images/async`），提取 `img.mimg` 元素中的图片 URL
2. **流程编排**：搜索关键词 → HTML 抓取 → URL 解析 → URL 转义处理 → 逐张下载 → URL 上传 → 入库（单张失败不影响整体流程）
3. **参数化控制**：支持搜索关键词、抓取数量（1-30 张）、文件命名前缀（支持 `{序号}` 动态变量）
4. **管理员专属**：仅管理员可调用，通过 `@AuthCheck(mustRole = "admin")` 控制访问

### R（结果）
- 管理员可通过关键词一次批量导入最多 30 张图片
- 单张下载失败不影响其他图片的处理流程

## 3.3 Agent 开发潜力：系统架构可扩展性

### 项目为 Agent 开发奠定的基础

| **工具调用范式** | 图片上传、编辑、分析等 40+ 精确定义的 API | 可直接转换为 Agent 的 Tool/Function 定义 |
| **权限控制** | Sa-Token RBAC 动态权限体系 | Agent 执行操作前可进行精细化权限校验 |
| **实时通信** | WebSocket + 消息广播机制 | 支撑 Agent 执行过程的前端实时状态推送 |
