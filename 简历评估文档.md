# 二、面向全栈开发岗位的评估

## 2.1 面试官视角亮点

### 🥇 第一梯队亮点（面试必问，重点准备）

**1. DDD 架构重构的决策逻辑**
- 面试官会问：为什么要从三层架构重构到 DDD？带来了什么实际收益？
- 建议回答思路：
  - 三层架构在业务复杂时 Service 类膨胀（500+ 行代码难以维护）
  - DDD 通过有界上下文将业务逻辑内聚到领域模型，依赖方向可控
  - 实际收益：新增功能（如空间分析）仅需在对应 domain 内修改，不再跨层散落
  - 要诚实说明：DDD 增加了代码量和学习成本，适合业务复杂度高的项目

**2. Disruptor 选型理由**
- 面试官会问：为什么选 Disruptor 而不是直接用线程池或 MQ？
- 建议回答思路：
  - Disruptor 是高性能无锁环形缓冲区，适合低延迟（微秒级）、高吞吐场景
  - 对比：线程池有上下文切换开销，MQ 有网络开销和序列化损失
  - 当前场景（单机、协同编辑）Disruptor 是最轻量高效的方案
  - 要诚实说明：如果是分布式场景（多实例），需要替换为 Redis Pub/Sub 或 MQ

**3. Snowflake ID 精度丢失**
- 面试官会问：这个问题的本质是什么？你的解决方案有哪些 trade-off？
- 建议回答思路：
  - 本质：JS Number 是 IEEE 754 双精度浮点，安全整数范围是 ±2^53-1，Snowflake 的 64 位 Long 超出范围
  - 方案对比：后端序列化为 String（本项目采用）vs 前端用 BigInt vs 换用短 ID 生成器
  - String 方案的代价：需要前后端统一约定，路由参数需注意保持 String 类型

**4. 两级缓存架构的设计考量**
- 面试官会问：Caffeine + Redis 两级缓存的一致性如何保证？
- 建议回答思路：
  - 当前是本地缓存 + 分布式缓存，存在不一致窗口期（Caffeine 和 Redis 的过期时间不同）
  - 解决方案（未实现但应了解）：Redis Pub/Sub 发布缓存失效消息 → Caffeine 监听并清除本地缓存
  - 缓存 Key 使用 MD5 哈希的意义：避免长 JSON 作为 Key 占用 Redis 内存

### 🥈 第二梯队亮点（加分项）

**5. Sa-Token Kit 双账号体系**
- 面试官会问：为什么需要两套鉴权体系？为什么不统一用 Sa-Token？
- 建议回答：Session 鉴权是项目初始化时采用的方案，后续引入 Sa-Token 时为了不破坏已有用户模块，采用 Kit 模式独立管理空间级权限，做到渐进式演进

**6. CI 降级容错**
- 体现了防御性编程思维：不假设外部服务永远可用，准备好降级方案
- 面试时可以作为"线上故障处理"的案例分享

# 三、面向 Agent 智能体开发岗位的评估

## 3.1 当前项目的 Agent 相关度

| Agent 核心概念 | 项目中的对应 | 匹配度 | 差距 |
|--------------|-------------|:---:|------|
| **AI 模型调用** | 阿里云百炼 DashScope API | ⭐⭐⭐⭐ | 仅单个模型调用，无多模型编排 |
| **Function Calling** | 无 | ⭐ | 缺少工具定义和调用框架 |
| **多步推理 (ReAct)** | 无 | ⭐ | 无 Agent 推理循环 |
| **记忆管理** | Session + Sa-Token 登录态 | ⭐⭐ | 无对话历史/上下文管理 |
| **工具编排** | Template Method 上传流程 | ⭐⭐ | 硬编码流程，非 Agent 动态决策 |
| **实时通信** | WebSocket + Disruptor | ⭐⭐⭐⭐ | 可直接复用于 Agent 状态推送 |
| **权限安全** | RBAC + 注解式鉴权 | ⭐⭐⭐⭐ | Agent 执行操作前的权限校验 |
| **异步任务** | AI 扩图异步 + 轮询 | ⭐⭐⭐ | 可扩展为 Agent 长时间任务管理 |

## 3.2 提升 Agent 方向竞争力的建议

### 短期（面试前可以做的）
1. **术语对齐**：将「AI 扩图」重新表述为「AI Agent 驱动的图片编辑工具」——你调用了一个接收自然语言参数、自主执行图像生成任务的 AI 服务，这就是 Agent 的基本范式
2. **工具调用映射**：将项目中 40+ 个 API 接口整理为「如果封装为 Agent Tool，每个 Tool 的输入/输出/权限定义是什么」——这直接对应 Agent 开发中的 Function Calling 设计
3. **异步任务管理**：将 AI 扩图的异步任务模式（创建 → 轮询 → 获取结果）重新表述为「Agent 长任务生命周期管理」——这是 Agent 编排中的核心挑战

### 中期（可以做的项目扩展）
1. **图片管理 Agent**：基于现有 API，开发一个对话式 Agent，用户用自然语言指令操作图库（如"帮我找所有包含猫的图片"→ 调用搜索 API）
2. **多模态 Agent**：利用阿里云视觉智能 API，实现图片内容自动识别 → 自动标签 → 自动分类的 Agent Pipeline
3. **审核 Agent**：接入 AI 内容审核模型，实现上传图片的自动 NSFW 检测 → 自动通过/拒绝/标记人工审核

## 3.3 面试表述策略

### 如果你面试 Agent 岗位，建议这样说

**"我独立开发了一个全栈智能云图库平台，其中重点实践了 AI 能力的系统集成——"**

1. **AI 服务集成**：「我封装了阿里云百炼 AI 的异步推理 API，设计并实现了完整的 Agent 任务生命周期管理——任务创建、状态轮询、结果处理、错误恢复」
2. **工具调用思维**：「项目的后端提供 40+ 个精确定义的 RESTful API，每个 API 都有清晰的输入输出契约——这种设计天然适配 Agent Function Calling 的工具定义范式」
3. **实时状态推送**：「我基于 Disruptor 环形缓冲区实现了高吞吐的实时消息系统，这个架构可以直接支撑 Agent 执行过程中的前端状态流式推送」
4. **安全边界**：「我设计了基于 RBAC 的权限体系，确保 Agent 在执行用户指令时只能操作其权限范围内的资源」

**但不要伪装成 Agent 专家**——诚实地说明你的项目是 AI 功能集成而非完整的 Agent 框架，展示你对 Agent 概念的学习热情和理解深度。

# 五、面试常见问题预测与应对

## 5.1 全栈方向高频问题

| 问题 | 难度 | 应对要点 |
|------|:---:|---------|
| DDD 和三层架构的区别是什么？| ⭐⭐⭐ | 关注点分离、依赖方向、有界上下文、充血 vs 贫血模型 |
| 为什么选 MyBatis-Plus 而不是 JPA？| ⭐⭐ | MyBatis 对复杂 SQL 更灵活、国内生态更好、分页插件成熟 |
| Caffeine 和 Redis 的数据一致性？| ⭐⭐⭐⭐ | 承认存在不一致窗口期、说明 Pub/Sub 失效广播方案 |

| Sa-Token 和 Spring Security 的对比？| ⭐⭐⭐ | Sa-Token 更轻量、API 更简洁、国内社区活跃 |
| 如何保证上传的图片不重复存储？| ⭐⭐ | 当前方案没有去重，可以通过文件 MD5 哈希实现 |
| Disruptor 的核心原理？| ⭐⭐⭐⭐ | 环形缓冲区、无锁 CAS、Sequence 屏障、伪共享填充 |

# 七、总结

## 对全栈岗位：⚡ 强推荐主攻

你的项目对于全栈开发岗位**非常有竞争力**。一个独立完成从数据库设计到上线部署的全栈项目，加上 DDD 架构、高并发处理、设计模式等深度实践，在初中级开发者中属于上游水准。建议在简历中将该项目作为**第一项目经历**展示。

**核心竞争力一句话**：
> 独立完成 DDD 架构的全栈云图库平台，涵盖 RBAC 权限、WebSocket 协同编辑、Disruptor 高并发处理、二级缓存架构和 AI 集成。

## 对 Agent 岗位：🔄 可探索，但需补充

你的项目展示了 AI API 集成和系统工程能力，但距离真正的 Agent 开发岗位还有距离。如果将 Agent 作为第二求职方向，建议：

1. 在项目中增加至少一个对话式 Agent 功能（即使是最简单的自然语言搜索）
2. 深入学习 LangChain/LangGraph 等 Agent 框架，并在项目中落地
3. 阅读 Agent 相关论文（ReAct、AutoGPT、MetaGPT 等），了解领域术语

**可用的过渡话术**：
> 具备全栈工程能力和 AI API 集成经验，对 Agent 技术栈（Function Calling、多步推理、工具编排）有系统学习，正在向 AI Agent 开发方向转型。
